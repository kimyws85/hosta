UX/UI 커리큘럼
Deliverable

UX 설계 요약 문서

감각이나 취향이 아니라, 목표·사용자·행동·검증 기준을 문장으로 고정해 이후 Figma/개발 단계에서 흔들리지 않게 만드는 기준 문서입니다.

프로젝트: 포트폴리오 웹(예시) 산출물: 기준 문서 1종 용도: Figma 설계 / AI 검증 입력 / 개발 Handoff

1) 사이트 목표 (Site Goal)

이 사이트는 첫 방문 사용자가 작업자의 역할과 문제 해결 역량을 빠르게 이해하고, 프로젝트 상세를 확인한 뒤 협업 또는 연락 여부를 판단할 수 있도록 돕습니다.

성공 조건
사용자가 "무엇을 하는 사람인지"를 5초 내 추정하고, 프로젝트 상세 확인 후 연락 행동(또는 연락 페이지 진입)을 수행
실패 조건
첫 화면을 서비스 소개/블로그로 오해하거나, 프로젝트 상세 도달 전에 이탈 / CTA 도달 후에도 확신 부족으로 미행동

2) 대표 사용자 (Primary User)

유형
외주·협업 파트너 / 채용 담당자
맥락
여러 후보를 빠르게 비교해야 하는 상황, 긴 글보다 구조·근거 중심으로 판단
정보 상태
작업 방식/역할/결과 수준을 아직 모름. "이 사람이 내 문제를 해결할 수 있나"를 가장 먼저 확인
핵심 욕구
시간을 덜 쓰고, 리스크 없이, 신뢰 판단을 끝내고 싶음

3) 핵심 행동 (Core Action)

  • 프로젝트 목록에서 관심 항목 선택
  • 프로젝트 상세에서 역할/성과/과정 확인
  • CTA(연락/협업 제안/이력서 요청) 중 하나 수행
행동 단일화 원칙
한 화면의 1순위 CTA는 1개만 고정하고, 나머지는 보조(링크/서브 CTA)로 격하

4) 사용자 시나리오 (User Scenario)

사용자는 검색 또는 공유 링크로 사이트에 진입합니다. 첫 화면에서 이 사이트의 목적과 작업자의 정체성을 빠르게 파악한 뒤, 스크롤 또는 네비게이션으로 프로젝트 목록을 훑어봅니다. 관심 항목을 클릭해 상세로 이동하고, 문제 해결 방식·역할·성과가 충분히 확인되면 연락 또는 협업 제안 행동을 수행하거나, 판단이 부족하면 보류하고 이탈합니다.

5) UX 요구사항 (UX Requirements)

  • 첫 화면에서 사이트 목적이 5초 내 인지되어야 합니다.
  • 핵심 행동은 화면 내에서 가장 먼저 탐지되는 요소여야 합니다.
  • 정보는 비교 → 판단 → 행동 순서로 노출되어야 합니다.
  • 신뢰 요소(성과/역할/결과/근거)는 행동 이전에 제공되어야 합니다.
  • 중복 설명/불필요한 선택지는 제거를 우선 검토해야 합니다.

6) 화면 역할 (Screen Roles)

Main
목적 인지 + 탐색 기준 제공("무엇을 하는 곳인지 / 어디부터 보면 되는지")
Projects
비교 가능성 확보(동일한 카드 구조로 빠르게 판단)
Detail
의사결정 지원(역할·과정·성과를 구조로 제시)
Contact
행동 완료(불안 제거, 최소 입력, 결과 예측 가능)

7) 변경 금지 규칙 (Design Constraints)

  • 첫 화면은 "목적 인지"가 실패하면 아래 구조가 좋아도 무의미합니다.
  • CTA를 앞당길 경우, 선행 정보(근거/사례/역할)가 부족해지는지 먼저 검증합니다.
  • 개선은 추가보다 제거가 우선입니다. 중복·장식·선택지부터 줄입니다.
  • 이 문서 없이 구조를 변경할 경우, AI 검증 프레임으로 재검증 후 반영합니다.
문서 성격: 설계 기준 고정용 / 전달용(개발·AI 협업) / 업데이트 시 변경 근거를 함께 기록
Session 01 · Step 2

UX 요구사항 도출

이 단계는 "무엇을 만들까"가 아니라, 잘못 만들어질 수 있는 가능성을 조건으로 차단하는 작업입니다. 목표·시나리오(1회차 Step 1)가 고정된 상태에서, 그 목표가 실제로 달성되려면 화면이 반드시 만족해야 하는 행동 기반 조건(Requirement)을 문장으로 정리합니다.

입력: 사이트 목표 / 대표 사용자 / 사용자 시나리오 / 핵심 행동 출력: UX 요구사항 목록(분류 + 우선순위 + 검증 질문)

1) UX 요구사항의 의미를 고정하기

UX 요구사항은 기능 요구사항이나 디자인 가이드가 아닙니다. 사용자 행동이 원하는 방향으로 흘러가기 위해, 화면이 반드시 보장해야 하는 인지/흐름/행동 조건입니다.

기능 요구사항과 차이
"무엇을 할 수 있어야 하는가"가 아니라, "사용자가 헷갈리지 않고 행동을 완료하려면 무엇이 보장되어야 하는가"
작성 방식
관찰 가능, 이진 판단 가능(성공/실패), 화면 단위로 연결 가능

2) 분류 체계(4축)

정보 인지(Information Clarity) 흐름 제어(Flow Control) 행동 유도(Action Guidance) 안정·신뢰(Stability & Trust)

3) 요구사항 예시

P0
첫 화면에서 사이트 목적이 5초 내 인지
검증 질문: 처음 보는 사용자가 1문장으로 사이트 목적을 말할 수 있는가?
P0
정보는 "인지 → 근거 → 선택" 순서로 노출
검증 질문: CTA에 도달했을 때 사용자가 '행동해도 될 만큼' 확신을 얻었는가?
P0
핵심 CTA는 화면에서 가장 먼저 탐지되는 행동 요소
검증 질문: 사용자가 스크롤을 하지 않아도 다음 행동 후보를 떠올릴 수 있는가?
Session 01 · Step 3

정보 구조 설계(IA)

이 단계는 레이아웃이 아니라 의미 구조를 설계하는 작업입니다. "무엇을 위에 두고, 무엇을 묶고, 무엇을 반복하며, 사용자가 어떤 순서로 정보를 만나는가"를 결정합니다.

입력: 목표·시나리오 + UX 요구사항 핵심 산출: 페이지 구조 + 정보 묶음 + 네비게이션 + 위계 기준

1) IA 설계의 핵심 흐름

A
필수 정보 추출
"이 정보를 못 보면 어떤 오해가 발생하는가"를 기준으로 필수 정보를 뽑습니다.
B
콘텐츠 블록 분해
소개/프로젝트/성과/프로세스/연락처럼 큰 덩어리로 시작합니다.
C
묶음 기준 설정
같은 행동을 유도하는 정보는 함께, 같은 인지 단계에서 필요한 정보는 함께 묶습니다.
D
페이지 단위 배치
각 페이지가 맡는 1차 역할에 맞게 정보 묶음을 배치합니다.

2) 페이지 구조 예시

Home - Hero(정체성 1문장 + 핵심 가치) - Featured Projects(대표 3개) - Proof(성과/수치/링크) - CTA(연락 방식 + 결과 예고) Projects - Filter/Sort(보조) - Project List(카드 구조 동일) Project Detail - Summary(문제/목표/역할/기간) - Process(핵심 단계 3~5) - Output(결과물/화면/링크) - CTA(연락)
Session 01 · Step 4

화면 역할 정의와 UI 요소 기능 분류

Step 4는 "각 화면이 사용자 여정에서 맡는 기능적 위치"를 고정하는 단계입니다. 역할이 고정되지 않으면, UI 요소는 서로 경쟁하면서 목적을 침범합니다.

핵심 작업: 페이지별 1차 역할 1개 고정 핵심 산출: 역할 정의 문장 + UI 요소 기능 분류

화면 역할(Screen Role) 유형

인식 화면(Orientation)

첫 진입에서 "여기가 어떤 곳인지"를 빠르게 규정

대표 위치: Home 상단

탐색 화면(Exploration)

사용자가 비교·탐색을 통해 선택지를 좁히는 화면

대표 위치: Projects 목록

판단 지원(Decision Support)

행동 직전에 "해도 되는가"를 결정하도록 근거 제공

대표 위치: 프로젝트 상세

행동 화면(Action)

사용자가 실제로 완료해야 하는 화면

대표 위치: Contact/신청 폼

페이지별 1차 역할 문장

Home: Orientation — 첫 5초 안에 정체성과 가치가 규정되고, 탐색을 시작하게 한다. Projects: Exploration — 비교 가능한 구조로 역량을 빠르게 검증하게 한다. Detail: Decision Support — "해도 된다"는 근거를 제공해 행동 결정을 돕는다. Contact: Action — 최소 마찰로 연락/요청을 완료하게 한다.
Session 01 · Step 5

설계 산출물 문서화

Step 5는 "정리"가 아니라 설계를 흔들리지 않게 고정하는 단계입니다. 이후 Figma/개발/AI 협업에서 설계가 '의견'으로 약화되지 않도록 기준점을 만듭니다.

출력: "설계 기준 문서" 1종(사람·AI 공용) 원칙: 짧고 명확 · 관찰 가능 · 화면 단위 연결

문서 구성 5요소(필수)

필수 1
사이트 목표 1문장
사용자 변화 중심 문장으로 고정. "보여준다"가 아니라 "사용자가 무엇을 판단/선택/완료한다"
필수 2
대표 사용자 정의(3줄)
"누구나"를 금지하고, 대표 사용자 1명을 고정
필수 3
사용자 시나리오
즉시 인식(3~5초) → 탐색(스크롤/비교) → 판단(행동/이탈) 흐름으로 서술
필수 4
UX 요구사항 요약
정보 인지/흐름 제어/행동 유도/신뢰 조건을 "~여야 한다"로 압축
필수 5
페이지 역할 정의
Orientation/Exploration/Decision/Action 중 하나로 고정

다음 단계에서 어떻게 쓰는가

Figma(2회차)
Frame/Section 설계가 "감각"으로 흐르려 할 때, 목표·요구사항·역할 문장으로 되돌아가 판정
AI 협업(3회차)
AI에게 개선을 맡기지 않고, 검증 질문을 이 문서에서 직접 뽑아 사용
개발 전달(4회차)
컴포넌트/상태/라우팅 요구사항의 '근거'로 활용
Session 02 · Step 1

Figma 기본 세팅과 작업 단위 정의

2회차 Step 1의 목적은 "Figma 사용법"이 아니라, 1회차에서 고정한 설계 기준을 시각 작업 환경에 그대로 이식하는 것입니다.

핵심단어: 작업 단위 / Page·Frame·Section / 기준점 고정 / 네이밍 출력: Figma 파일 "뼈대" 세팅 완료

1) 작업 단위를 먼저 고정해야 하는 이유

Figma에서 작업 단위를 고정하지 않으면, 이후 모든 산출물이 섞입니다. 와이어프레임과 컴포넌트가 한 페이지에 섞이고, 화면 상태와 최종안이 같은 프레임에 공존합니다.

핵심
Page / Frame / Section을 서로 다른 책임 단위로 분리
Page는 설계 묶음 단위, Frame은 화면 단위, Section은 의미 단위입니다.

2) Page 구조 세팅

01_Design_Basis - 1회차 설계 기준 텍스트 고정 02_Wireframe - Home / Projects / Detail / Contact 와이어프레임 03_Components - Button / Card / SectionHeader 등 04_Flow_Notes (선택) - 사용자 흐름 메모, 검증 결과 기록

3) 네이밍 규칙

Frame 네이밍
Home__Orientation / Projects__Exploration / Detail__Decision / Contact__Action
Section 네이밍
Purpose_Message / Comparison_List / Proof_Block / Primary_CTA
금지
Hero, Section1, Content2 같은 의미 없는 이름

4) Step 1 산출물

Figma Page 구조
완료 01_Basis / 02_Wireframe / 03_Components 분리
Frame 기준점
완료 Desktop Frame 1개 고정
Session 02 · Step 2

Auto Layout으로 정렬·간격 시스템 설계

이 단계의 핵심은 Auto Layout "기능을 배우는 것"이 아니라, 정렬과 간격을 감각에서 규칙으로 전환해 틀릴 수 없게 만드는 것입니다.

핵심단어: Alignment / Spacing / 기준선 / 중첩 Auto Layout 출력: 간격·정렬 규칙이 고정된 Section 템플릿

1) 간격(Spacing)은 위계 표현

관계 유형의미간격 규칙
동일 묶음 내부한 메시지 안의 요소8 / 12
서브 묶음 분리같은 섹션 안에서 의미가 갈리는 블록16
섹션 간 분리역할이 바뀌는 구간40 / 56

2) Auto Layout 적용 순서

1) Section Wrapper(세로 흐름, padding 고정) 2) Section Header(제목/설명 블록) 3) Content Group(카드 그리드/리스트) 4) Item 내부(카드 내부 텍스트) 5) Action Group(버튼/링크)

3) Step 2 산출물

Section 템플릿 2종
완료 Informational / Exploration 템플릿
간격 규칙표
완료 내부(8/12), 서브(16), 섹션(40/56)
중첩 구조
완료 Section → Header → ContentGroup → Item
Session 02 · Step 3

와이어프레임을 통한 정보 위계 시각화

이 단계의 목적은 "레이아웃을 그리는 것"이 아니라, 정보가 어떤 순서로 인식되어야 하는지를 시각적으로 고정하는 것입니다.

핵심단어: 정보 위계 / 시선 흐름 / 우선순위 고정 출력: 페이지별 와이어프레임(검증용)

1) 정보 위계란 무엇인가

정보 위계는 "무엇이 먼저 보이고, 무엇이 나중에 해석되는가"에 대한 설계입니다. 사용자는 화면을 위에서 아래로 읽지 않습니다. 크기, 위치, 묶음, 간격을 통해 우선순위를 추론합니다.

위계 판단 4요소
① 위치(상단/중앙) · ② 크기(덩어리 크기) · ③ 묶음(같은 박스) · ④ 간격(분리 정도)

2) 와이어프레임 제작 순서

1) Home(Orientation) — 정체성·가치 인식 2) List/Projects(Exploration) — 비교·탐색 3) Detail(Decision) — 판단 근거 4) Contact(Action) — 완료

3) 검증 질문

첫 화면(Above the Fold)
이 화면을 3초만 봤을 때 "이 사이트는 무엇을 제공하는가"를 말할 수 있는가
중간 흐름

컴포넌트 기초: 반복 요소를 시스템화

와이어프레임이 "구조가 맞는가"를 검증하는 단계였다면, Step 4는 "이 구조를 앞으로도 유지·확장할 수 있는가"를 고정하는 단계입니다.

핵심단어: 컴포넌트 / 책임 범위 / 재사용 규칙 / Variant 출력: 컴포넌트 목록 + 규칙 문서

1) 컴포넌트는 "역할"로 정의합니다

컴포넌트 후보 판별 질문
① 다른 페이지에서도 같은 역할인가
② 내용만 바뀌고 구조는 동일한가
③ UX 검증 기준이 동일하게 적용되는가

2) 예시 컴포넌트 목록

- SectionHeader: 섹션 제목/설명/보조 문구 - ProjectCard: 프로젝트 요약 비교 구조 - PrimaryActionButton: 핵심 행동 유도 - TagPill: 분류/맥락 제공 - KeyValueRow: 이력/성과/기간 정보 - EmptyStateBlock: 결과 없음/다음 행동 유도

3) Step 4 산출물

컴포넌트 목록

Session 02 · Step 5

반응형 사고를 구조에 반영

반응형은 "데스크톱을 줄이는 작업"이 아니라, 화면이 바뀌어도 UX 목표가 유지되도록 정보 우선순위를 다시 고정하는 작업입니다.

핵심단어: 우선순위 / 브레이크포인트 / 표현 방식 전환 / 터치 UX 출력: 반응형 규칙 문서 + 섹션별 전환 기준

1) 반응형 핵심 질문

3가지 질문
① 화면이 작아져도 반드시 남아야 하는 정보는 무엇인가
② 화면이 작아지면 가장 먼저 뒤로 밀려도 되는 정보는 무엇인가
③ 같은 정보라도 표현 방식이 바뀌어야 하는가

2) 반응형 설계 순서

1) 정보 우선순위 재확인 2) 남길 정보/미룰 정보/삭제할 정보 결정 3) 표현 방식 전환(그리드→리스트) 4) 터치 UX 조건 반영

3) Step 5 산출물

우선순위 규칙
남김 / 미룸 / 삭제 기준 + 이유
섹션별 전환표
Desktop 표현 ↔ Mobile 표현 전환 규칙
터치 UX 조건
클릭 영역/간격/행동 위치 규칙
Session 03 · Step 2

AI 입력 설계: 컨텍스트를 정확히 주는 법

Step 2의 목적은 "프롬프트 잘 쓰기"가 아닙니다. AI가 마음대로 개선안을 만들지 못하도록, 설계 목표·사용자·핵심 행동·검증 관점을 먼저 고정하는 단계입니다.

핵심단어: 컨텍스트 / 제약 / 검증 질문 / 범위 고정 / 출력 포맷 출력: AI 검증 결과(지적/근거/리스크/우선순위)

1) 컨텍스트 입력 순서

1) 페이지 목적(역할 1문장) 2) 대표 사용자(왜 들어오는지 / 뭘 빨리 찾는지) 3) 핵심 행동(반드시 완료해야 하는 1개) 4) 성공 조건 / 실패 조건 5) 화면 구성 요약(섹션 목록 + 역할) 6) 검증 프레임(목적 인지/정보 위계/행동 흐름/인지 부하)

2) 출력 포맷 고정

- Problem: (문제 지점 1문장) - Evidence: (왜 문제인지, 화면/정보 구조 근거) - Behavior Impact: (사용자가 어떤 행동을 하게 되는지) - Severity: (High / Medium / Low) - Fix Direction (optional): (최소 변경 관점에서 1~2개 방향만)

3) 범위 고정 문장(예시)

- 당신의 역할은 '검증자'이며, 새로운 UI/기능 제안은 금지한다. - 구조 재작성 제안은 금지하고, 문제 지점 '지적'과 '근거'만 제공한다. - 개선이 필요해 보이더라도, 해결책은 1~2개 선택지로만 제시한다. - 결과는 반드시: Problem / Evidence / Behavior Impact / Severity 로 출력한다.
Session 03 · Step 3

사용자 행동 시뮬레이션 기반 리뷰

이 단계에서는 화면을 "평가"하지 않습니다. 사용자를 가정해 실제 행동을 끝까지 실행시키고, 설계자가 의도하지 않은 선택과 이탈이 발생하는 지점을 드러냅니다.

1) 행동 시뮬레이션의 최소 단위

1) 해석: 이 화면을 무엇이라고 이해하는가 2) 선택: 다음에 무엇을 하겠다고 판단하는가 3) 행동: 실제로 어떤 UI를 클릭/탐색하는가

2) CTA 도달 시뮬레이션

마찰 유형 분류 - 인지 마찰: 이해는 했지만 확신이 없음 - 구조 마찰: 어디서 행동해야 할지 모름 - 심리 마찰: 행동 결과가 불안함

Session 03 · Step 4

AI 피드백 선별 및 개선안 설계

이 단계에서 AI 피드백은 '정답'이 아닙니다. 피드백을 선별하고 최소 변경 원칙으로 개선안을 설계합니다.

1) 수용/보류/폐기 분류

수용(Accept)
  • 목적 오해로 초기 이탈 가능성
  • CTA 도달 전에 핵심 정보가 늦게 등장
  • 행동 결과가 불안해 완료율이 떨어지는 지점
보류(Hold)
  • 시각적 완성도 관련 제안
  • 현재 단계에서 검증이 어려운 신규 요소
폐기(Discard)
  • 핵심 행동을 분산시키는 CTA 추가
  • 정보량을 늘려 인지 부하를 키우는 제안

2) 최소 변경 원칙

최소 수정 단위 점검 질문 - 구조를 바꾸지 않고 해결 가능한가? - 순서/강조/삭제로 해결 가능한가? - 추가가 아니라 제거로 해결 가능한가?

3) Step 4 산출물

피드백 분류표
수용 / 보류 / 폐기
개선안 2안
보수적(안전) / 공격적(직선화)
채택안 1개
선택 근거(목표/프레임 기준)
Session 03 · Step 5

개선 전·후 비교 문서화

이 단계는 "개선 결과를 예쁘게 정리"하는 단계가 아니라, 설계 판단 근거를 고정하는 단계입니다.

1) 비교 문서의 기본 구조(4파트)

1) 변경 요약 (What) 2) 문제 지점 정리 (Why) 3) 개선 근거: UX 프레임 연결 (Reason) 4) 유지해야 할 규칙 (Rules)

2) 프레임 연결 규칙

목적 인지 첫 화면 오해 가능성 감소 정보 위계 핵심 근거의 조기 노출 행동 흐름 CTA 도달 경로 단축 인지 부하 중복 정보 제거
