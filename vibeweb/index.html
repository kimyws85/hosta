기존 코딩 학원과 무엇이 다른가요?

단순히 코드를 외우는 수업이 아닙니다. AI 시대에 맞는 방식으로 배웁니다.

AI 기반 웹 제작 환경 구축

Claude + 웹디자인 기초 + 프롬프트 웹 제작

코딩을 전혀 몰라도 괜찮습니다. 이 달의 목표는 Claude라는 AI 도구를 완벽히 세팅하고, 프롬프트(말로 하는 명령)만으로 실제 웹사이트를 만들어보는 경험을 하는 것입니다. 디자인 감각의 기초도 함께 잡아갑니다.

웹디자인 기초

  • 웹의 구조 이해 — 모든 웹사이트는 헤더(상단), 본문(내용), 푸터(하단)로 이루어집니다. 좋은 웹사이트의 레이아웃이 어떻게 구성되는지 실제 사례로 분석합니다.
  • 색상 이론 기초 — 주색(brand color), 보조색(accent), 배경색, 텍스트 색의 역할을 이해합니다. 색상 대비(contrast)가 가독성에 미치는 영향, 어두운 배경 vs 밝은 배경의 특성을 배웁니다.
  • 폰트와 타이포그래피 — 제목/소제목/본문/캡션의 크기 위계(hierarchy)를 잡는 법을 익힙니다. 가독성 좋은 폰트 선택 기준, 줄 간격(line-height)이 읽힘에 미치는 영향을 이해합니다.
  • 여백(Spacing)의 중요성 — 요소들 사이의 간격이 왜 디자인 품질을 결정하는지 이해합니다. 여백이 없는 사이트 vs 여백이 적절한 사이트를 직접 비교 분석합니다.
  • 좋은 웹 vs 나쁜 웹 분석 — 실제 국내외 사이트들을 보며 "왜 이게 좋은가", "왜 이게 별로인가"를 색상·폰트·여백·구조 관점에서 설명할 수 있는 눈을 기릅니다.

Claude + VSCode 세팅

  • Claude Pro 가입 — Claude.ai 유료 플랜(Pro) 가입 방법, 플랜별 차이점, 무료 vs 유료를 왜 유료로 써야 하는지 이유를 설명합니다.
  • VSCode 설치 및 세팅 — 개발자들이 가장 많이 쓰는 에디터 VSCode를 설치하고, 기본 테마, 폰트 크기, 자동 저장 설정 등 편한 환경을 만듭니다.
  • Claude Code (CLI) 설치 — 터미널/명령줄에서 Claude를 사용할 수 있는 Claude Code를 설치합니다. 왜 웹 인터페이스보다 CLI(Command Line Interface)가 개발에 더 강력한지 이해합니다.
  • VSCode + Claude Code 연동 — VSCode 안에서 바로 Claude와 대화하면서 코드를 만들고 수정하는 실제 개발 환경을 완성합니다. 파일을 열고 바로 작업하는 워크플로우를 익힙니다.
  • 기본 사용법 실습 — 파일 열기, 폴더 구조 이해, Claude에게 질문하는 법, Claude가 만든 파일을 브라우저에서 확인하는 법까지 실습합니다.

프롬프트로 웹 만들기

  • UI 컴포넌트 단위로 만들기 — 버튼(크기·색상·hover 효과), 카드(이미지+텍스트 조합), 슬라이더/캐러셀, 아이콘 배치, 네비게이션 바 등 웹에서 자주 쓰이는 요소들을 하나씩 Claude에게 만들어달라고 요청하며 경험을 쌓습니다.
  • 프롬프트 수정 요청하기 — "버튼 색상을 파란색으로 바꿔줘", "글자 크기 키워줘", "카드 간격 더 넓게 해줘" 등 원하는 결과를 말로 요청하고 수정하는 반복 과정을 익힙니다.
  • 자기소개 페이지 완성 — 이름, 소개글, 프로필 이미지, SNS 링크, 기술 스택 등이 들어간 실제 포트폴리오용 자기소개 페이지를 Claude와 함께 완성합니다.
  • 랜딩 페이지 완성 — 히어로 섹션(메인 문구+이미지), 특징 소개(카드 3개), 후기/수치 섹션, CTA 버튼(행동 유도)이 포함된 실제 서비스 랜딩 페이지를 만듭니다.
1개월 후 결과물 자기소개 페이지 1개 + 랜딩 페이지 1개 완성. 코드를 한 줄도 직접 안 써도 됩니다.

웹 표준 기술 이해

HTML / CSS / JS + Claude

1개월차에서 프롬프트로만 만들었다면, 이번 달은 Claude가 만든 코드를 직접 읽고 이해합니다. HTML, CSS, JavaScript의 기본 구조를 파악하면 "왜 이렇게 보이는지"를 알게 되고, 원하는 부분을 직접 수정할 수 있는 능력이 생깁니다.

HTML — 웹의 뼈대

  • 태그 구조 이해 — <div>, <section>, <p>, <h1~h6>, <img>, <a>, <ul><li>, <span> 등 자주 쓰이는 태그들의 역할과 사용법을 이해합니다.
  • 시맨틱 태그 — <header>, <nav>, <main>, <section>, <article>, <aside>, <footer> 등 의미있는 태그를 쓰는 이유(검색엔진 최적화, 접근성)를 이해하고 적용합니다.
  • HTML 파일 구조 — <!DOCTYPE html>, <head>, <body>의 역할, CSS/JS 파일을 연결하는 방법, meta 태그의 역할을 이해합니다.

CSS — 웹의 스타일

  • 선택자·클래스·id 개념 — CSS가 어떻게 특정 요소를 찾아서 스타일을 적용하는지 이해합니다. 클래스(.class)와 id(#id)의 차이와 언제 무엇을 쓰는지 배웁니다.
  • CSS 변수(Custom Properties) — :root에 색상, 폰트, 간격을 변수로 정의하고 전체 스타일을 한 곳에서 관리하는 실무 방식을 익힙니다.
  • Flexbox 레이아웃 — 요소를 가로/세로로 정렬하는 Flexbox의 핵심 속성들 (display:flex, justify-content, align-items, gap 등)을 실습합니다.
  • Grid 레이아웃 — 카드 목록, 2단 레이아웃 등을 Grid로 구현하는 법을 배웁니다. Flexbox와 Grid를 언제 각각 쓰는지 판단하는 기준도 익힙니다.
  • 반응형 디자인 (미디어 쿼리) — 화면 크기에 따라 레이아웃이 바뀌는 반응형 웹을 만드는 법을 배웁니다. 모바일(480px), 태블릿(768px), 데스크톱 기준으로 스타일을 분기하는 법을 익힙니다.
  • transition · hover 효과 — 버튼을 올렸을 때 색이 바뀌고, 카드를 올렸을 때 살짝 올라오는 등의 부드러운 인터랙션 효과를 CSS만으로 구현하는 법을 익힙니다.

JavaScript — 웹의 동작

  • 변수·함수 기초 — let, const, 함수 선언, 화살표 함수 등 JS의 기본 문법을 이해합니다. 코드를 외우는 것이 목표가 아니라, Claude가 만든 코드를 읽고 이해하는 것이 목표입니다.
  • DOM 조작 기초 — document.querySelector로 요소를 찾고, classList.add/remove로 클래스를 추가/제거하는 법을 이해합니다. 이것만 알면 대부분의 인터랙션을 구현할 수 있습니다.
  • 이벤트 리스너 — 클릭, 스크롤, 마우스 이동 등의 이벤트에 반응하는 addEventListener의 기본 패턴을 익힙니다.
  • 실습: 모달 팝업 — 버튼 클릭 시 팝업창이 뜨고, X 클릭 시 닫히는 모달을 직접 구현합니다.
  • 실습: 모바일 햄버거 메뉴 — 모바일에서 햄버거 메뉴 클릭 시 메뉴가 펼쳐지는 토글 네비게이션을 구현합니다.
  • 실습: 스크롤 애니메이션 — 스크롤 시 요소가 부드럽게 나타나는 IntersectionObserver 패턴을 익힙니다.

Claude와 함께하는 방식

  • Claude 코드 읽고 이해하기 — Claude가 만든 코드를 보면서 "이 부분이 왜 이렇게 된 건지"를 Claude에게 물어보고 설명을 들으며 코드 이해력을 키웁니다.
  • 직접 수정 + Claude 요청 병행 — 간단한 수정(색상 변경, 텍스트 수정)은 직접 하고, 복잡한 기능은 Claude에게 요청하는 방식으로 실력을 쌓습니다.
  • 1개월차 결과물 코드로 재구현 — 1개월차에 프롬프트만으로 만든 자기소개 페이지를 이번에는 HTML/CSS 구조를 직접 이해하며 다시 만들어봅니다.
2개월 후 결과물 반응형 웹사이트 1개 완성. HTML/CSS/JS 구조를 읽고 이해할 수 있는 수준. 간단한 부분은 직접 수정 가능.

컴포넌트 기반 개발

React + Claude

React는 현재 프론트엔드 취업 시장의 표준입니다. 복잡한 웹 앱을 "컴포넌트"라는 단위로 나눠서 만드는 방식을 배웁니다. Claude와 함께라면 React의 진입장벽을 낮추고 실무 수준의 앱을 빠르게 완성할 수 있습니다.

React 핵심 개념

  • 왜 React인가 — HTML로만 만들면 같은 코드를 여러 곳에 반복 작성해야 합니다. React는 버튼, 카드, 네비게이션 같은 요소를 "컴포넌트"로 한 번만 만들고 어디서든 재사용할 수 있게 해줍니다. 현재 국내외 대부분의 웹 서비스가 React로 만들어집니다.
  • 컴포넌트 개념 이해 — Button.jsx, Card.jsx, Navbar.jsx처럼 UI를 독립적인 조각으로 나누는 법, 컴포넌트를 조합해서 페이지를 만드는 방식을 이해합니다.
  • JSX 문법 — HTML처럼 생겼지만 JavaScript인 JSX 문법을 이해합니다. className, onClick, {} 안에 JS 표현식 넣기 등 HTML과 다른 점을 익힙니다.
  • Props — 컴포넌트에 데이터 전달 — 같은 Card 컴포넌트에 제목, 내용, 이미지를 다르게 넣어서 여러 카드를 만드는 props 개념을 이해하고 실습합니다.
  • State — 변하는 데이터 관리 — 버튼 클릭 횟수, 모달 열림/닫힘, 탭 선택 상태 등 화면에서 변하는 값을 useState로 관리하는 법을 익힙니다.
  • 파일 구조 이해 — src/, components/, pages/, public/ 폴더의 역할을 이해하고 실무에서 파일을 어떻게 구조화하는지 패턴을 익힙니다.

Claude와 React 개발

  • 컴포넌트 생성 요청 — "ProductCard 컴포넌트를 만들어줘. props로 image, title, price, description을 받아야 해" 처럼 구체적으로 요청해서 바로 쓸 수 있는 컴포넌트를 만드는 법을 익힙니다.
  • 기능 구현 요청 — "탭을 클릭하면 해당 탭의 내용만 보여주는 기능 만들어줘", "검색어를 입력하면 목록이 필터링되는 기능 만들어줘" 등의 실무 기능을 Claude와 함께 구현합니다.
  • 에러 해결 — React 개발 중 만나는 에러 메시지를 Claude에게 붙여넣고 원인과 해결법을 이해하는 방식으로 디버깅 능력을 키웁니다.

최종 프로젝트: 실무형 앱 완성

  • 프로젝트 선택 — 쇼핑몰 상품 목록 페이지 / 포트폴리오 사이트 / 레스토랑 메뉴 앱 중 하나를 선택합니다.
  • 컴포넌트 설계 — 만들 앱을 어떤 컴포넌트로 나눌지 설계하고, Claude와 함께 하나씩 만들어갑니다.
  • 기능 구현 — 필터링, 검색, 탭 전환, 모달 등 실제 서비스에서 볼 수 있는 기능들을 구현합니다.
  • 배포 — Vercel을 통해 실제 인터넷에서 접속 가능한 URL로 배포하는 방법을 배웁니다. 이제 포트폴리오에 링크를 넣을 수 있습니다.

서비스 설계 역량 강화

UI/UX + 기획 + Figma

개발만 할 줄 아는 사람과 기획부터 설계까지 할 수 있는 사람의 차이는 큽니다. 4개월차에는 Figma로 와이어프레임·프로토타입을 만들고, 서비스 기획서 작성법을 익히며 "AI 활용 기획자" 방향으로 커리어를 확장합니다.

UX 리서치 & 분석

  • 사용자 인터뷰 방법 — 사용자가 서비스를 어떻게 쓰는지 파악하기 위한 인터뷰 질문을 설계하고 진행하는 방법을 익힙니다. Claude로 인터뷰 스크립트를 작성하는 법도 배웁니다.
  • 페르소나 작성 — 서비스의 주요 사용자를 구체적으로 묘사한 페르소나 카드를 작성합니다. "28세 직장인 김민지, 퇴근 후 자기계발에 관심 있음" 같은 구체적 인물을 설정하고 그 사람의 니즈에 맞게 서비스를 설계하는 방법을 배웁니다.

클라이언트 & 서버
모든 웹은 요청과 응답으로 작동한다
클라이언트 (Client)
브라우저요청사용자
요청을 보내는 쪽. 사용자의 브라우저가 가장 대표적인 클라이언트입니다. URL을 입력하거나 버튼을 클릭하는 순간 서버에 요청을 보냅니다. 브라우저뿐 아니라 모바일 앱, 데스크톱 앱, 심지어 다른 서버도 클라이언트가 될 수 있습니다. 클라이언트의 역할은 크게 세 가지입니다. 첫째, 서버에 필요한 것을 요청합니다. 둘째, 서버에서 받은 데이터를 화면에 표시합니다. 셋째, 사용자의 입력을 받아 서버로 전달합니다. 프론트엔드 개발자가 만드는 모든 코드(HTML, CSS, JS)는 클라이언트에서 실행됩니다. 클라이언트 코드는 사용자가 F12를 눌러 볼 수 있으므로 비밀 정보(API 키, 비밀번호)를 절대 클라이언트 코드에 넣으면 안 됩니다.
브라우저 -- 가장 흔한 클라이언트 모바일 앱 -- iOS/Android 앱도 클라이언트 데스크톱 앱 -- 일렉트론 앱 (VS Code, Discord) CLI 도구 -- curl, fetch 명령으로 API 호출 다른 서버 -- 서버-서버 간 API 호출 (마이크로서비스) // 브라우저 점유율 (2025년) Chrome: 65% -- 개발 표준 기준 Safari: 19% -- iOS/macOS 기본, 별도 테스트 필요 Edge: 4% -- 기업 환경에서 많이 씀 Firefox: 3% -- 개발자 사용 많음
실무 포인트 -- Chrome DevTools에서 Network 탭을 열면 클라이언트가 어떤 요청을 보내는지 실시간으로 볼 수 있습니다. API 연동 문제가 생기면 여기서 원인의 80%를 찾을 수 있습니다.
서버 (Server)
응답24/7데이터
요청을 받아 응답을 돌려주는 컴퓨터. 항상 켜져 있으며, 클라이언트가 요청한 HTML, 이미지, 데이터 등을 보내줍니다. "서버"는 소프트웨어(프로그램)를 가리킬 수도 있고, 하드웨어(컴퓨터 자체)를 가리킬 수도 있습니다. 서버가 하는 일은 다양합니다. 웹 서버는 HTML 파일을 전달하고, API 서버는 JSON 데이터를 전달하고, 데이터베이스 서버는 데이터를 저장하고 조회합니다. 실제 서비스는 이런 서버들이 여러 개 조합되어 작동합니다. 개발 중에는 내 컴퓨터에서 서버를 실행하고(localhost), 배포 후에는 클라우드 서버가 이 역할을 24시간 365일 대신합니다. 서버가 다운되면 서비스 전체가 멈추므로, 고가용성(HA)과 장애 대응이 중요합니다.
웹 서버: HTML, CSS, JS, 이미지 파일 전달 (Nginx, Apache) API 서버: JSON 데이터 전달 (REST API) (Node.js/Express, Python/FastAPI) DB 서버: 데이터 저장 및 조회 (MySQL, PostgreSQL, MongoDB) 개발 환경 (내 컴퓨터): localhost:3000 -- Vite React 개발 서버 localhost:8080 -- 백엔드 API 서버 배포 환경 (클라우드): https://myapp.vercel.app -- 전 세계 접속 가능 서버 사양: CPU, RAM, 스토리지에 따라 비용 결정
핵심 -- 서버리스(Serverless) 아키텍처를 쓰면 서버를 직접 관리하지 않아도 됩니다. Vercel Functions, Supabase 등이 이 방식입니다. 프론트엔드 개발자도 간단한 백엔드 기능을 구현할 수 있게 됩니다.
요청 & 응답 흐름
HTTPDNS흐름
상세 흐름: 1. 주소창에 URL 입력: naver.com 2. 브라우저 캐시 확인 (DNS 캐시가 있으면 바로 사용) 3. DNS 서버에 naver.com의 IP 주소 질의 4. DNS 서버 응답: 223.130.200.104 5. 해당 IP의 443번 포트로 HTTPS 연결 요청 6. SSL/TLS 핸드셰이크 (암호화 채널 수립) 7. HTTP GET 요청 전송 (헤더 포함) 8. 서버가 HTML 파일로 응답 (200 OK) 9. 브라우저가 HTML 파싱 시작 10. HTML 안의 CSS, JS, 이미지 추가 요청 11. 모든 리소스 로딩 완료 -> 화면 완성 실제 소요 시간: DNS 조회: 1~50ms TCP 연결: 20~100ms 서버 응답: 10ms~수초 (서버 성능에 따라) 전체 로딩: 0.5~3초 (최적화 여부에 따라)
웹 페이지 하나를 여는 데도 이 10단계 이상이 실행됩니다. 로그인, 댓글 작성, 데이터 저장도 같은 기본 구조를 따릅니다. 이 흐름을 이해하면 "페이지가 느린 이유", "API 오류의 원인", "CORS 에러가 발생하는 위치"를 빠르게 파악할 수 있습니다. Chrome DevTools의 Network 탭에서 각 단계의 소요 시간을 시각적으로 확인할 수 있습니다. 실무에서 "서버 응답 시간이 길다"와 "네트워크 지연이 크다"는 해결 방법이 다르므로 구분할 줄 알아야 합니다. 프론트엔드 개발자는 서버 응답 시간 외의 나머지 단계(렌더링 최적화, 캐시 활용, 이미지 최적화)를 책임집니다.
정적 vs 동적 사이트
StaticDynamicSPA
정적(Static) 사이트: 서버 -> 미리 만들어진 HTML 파일 그대로 전달 장점: 빠름, 보안 강함, CDN 캐싱 쉬움 단점: 내용이 자주 바뀌면 관리 어려움 예: 포트폴리오, 랜딩페이지, 문서 사이트 배포: GitHub Pages, Vercel, Netlify (무료) 동적(Dynamic) 사이트: 서버 -> 사용자/시간/상태에 따라 HTML 즉석 생성 장점: 사용자별 맞춤 내용, 실시간 데이터 단점: 서버 비용, 속도가 상대적으로 느림 예: SNS, 쇼핑몰, 은행, 이메일 배포: AWS, GCP, Azure (유료) SPA (Single Page Application): React/Vue로 만든 앱. 처음에 빈 HTML 한 장 받음 JS가 실행되면서 동적으로 화면을 채움 URL이 바뀌어도 서버 요청 없이 화면만 교체 예: Gmail, Notion, Figma SSR (Server-Side Rendering): Next.js가 대표적. 정적+동적의 장점을 합침 SEO + 빠른 초기 로딩 + 동적 내용 모두 가능
이 수업에서는 정적 사이트(1~2개월차)에서 시작해 SPA React 앱(3개월차)으로 나아갑니다. 포트폴리오 사이트는 정적으로 만들어 Vercel에 무료 배포합니다. 동적 사이트는 사용자 로그인 정보, 상품 재고, 게시판 댓글처럼 실시간으로 바뀌는 데이터가 필요할 때 씁니다. React 앱은 기술적으로는 정적 파일이지만 동작은 동적입니다. 검색 엔진 최적화(SEO)가 중요한 서비스는 SSR(Next.js)을 함께 씁니다. 취업 포트폴리오 수준에서는 SPA로 충분합니다.
핵심 -- 이 수업의 포트폴리오 사이트는 정적 방식으로 Vercel에 무료 배포합니다.
포트 (Port)
localhost포트개발환경
포트 번호 = 같은 IP에서 서비스를 구분하는 번호 자주 쓰는 포트: :80 -- HTTP 기본 포트 (http://naver.com -> :80) :443 -- HTTPS 기본 포트 (자물쇠 아이콘) :22 -- SSH (서버 원격 접속) :3306 -- MySQL 데이터베이스 :5432 -- PostgreSQL 데이터베이스 :6379 -- Redis 캐시 서버 개발 환경 예시: 프론트엔드: localhost:5173 (Vite) 백엔드 API: localhost:8080 (Node/Express) 데이터베이스: localhost:3306 (MySQL) -> 세 개를 동시에 실행해 서로 통신 포트 충돌: "Port 3000 is already in use" 에러가 뜨면 다른 프로세스가 그 포트를 점유 중. lsof -i :3000 명령으로 확인 후 종료.
같은 IP 주소(컴퓨터)에서 여러 서비스를 동시에 실행할 때 포트 번호로 구분합니다. 백엔드 서버와 프론트엔드 개발 서버를 동시에 실행할 때 포트가 다르므로 충돌하지 않습니다. CORS 에러가 발생하는 이유도 여기 있습니다. 같은 컴퓨터라도 포트가 다르면 "다른 출처(origin)"로 취급하기 때문입니다. http://localhost:3000에서 http://localhost:8080의 API를 호출하면 CORS가 발생합니다. 이것은 정상적인 보안 동작이며, 개발 환경에서는 프록시 설정으로 해결합니다. 배포 환경에서는 동일한 도메인을 쓰거나 CORS 헤더를 설정합니다.
실무 포인트 -- 개발할 때 "포트가 이미 사용 중"이라는 에러가 자주 납니다. 그냥 터미널을 껐다 켜거나, VS Code를 재시작하면 대부분 해결됩니다.
클라우드 서버
AWSVercel클라우드
주요 클라우드 서비스: AWS (Amazon Web Services): 전 세계 점유율 1위 (약 33%) EC2(서버), S3(파일저장), RDS(DB) 등 국내 대기업 및 스타트업 대부분 사용 GCP (Google Cloud): Google의 인프라 활용 (BigQuery, AI/ML 강점) Azure (Microsoft): 기업 및 B2B 환경에서 강세 프론트엔드 배포 특화: Vercel -- Next.js 만든 회사. 무료 플랜 관대 Netlify -- 드래그 앤 드롭 배포, 폼 기능 내장 Cloudflare Pages -- CDN 최강, 무료 서버 비용 대략: 작은 서버 (EC2 t3.micro): 월 약 10~15달러 Vercel 무료 플랜: 개인 프로젝트 충분히 커버 Supabase 무료 플랜: DB + 인증 + API 무료
클라우드는 서버 컴퓨터를 직접 구매하고 관리하는 대신 대여해서 쓰는 방식입니다. 사용한 만큼만 비용을 내는 종량제 모델이 대부분입니다. 현업에서 거의 모든 서비스가 클라우드를 씁니다. 프론트엔드 개발자는 Vercel이나 Netlify를 가장 많이 사용하고, 복잡한 서버 설정 없이 GitHub 연동만으로 배포할 수 있습니다. AWS는 EC2, S3, Lambda 등 다양한 서비스를 조합해 사용하며, 취업 후 자연스럽게 접하게 됩니다. Vercel의 무료 플랜으로 이 수업의 포트폴리오를 모두 배포할 수 있습니다. 클라우드 비용은 트래픽이 폭발적으로 늘거나 서비스를 잘못 설정하면 갑자기 커질 수 있으므로, 처음에는 무료 플랜 안에서 시작하는 것이 좋습니다.

보안암호화SSL
프로토콜 비교: http:// -- 평문 전송 (도청 가능) 중간에서 패킷 캡처하면 내용 그대로 보임 https:// -- TLS(SSL) 암호화 전송 키 없이는 내용 해독 불가 TLS 핸드셰이크 과정: 1. 클라이언트 -> 서버: "TLS 연결 원함" 2. 서버 -> 클라이언트: 인증서(공개키 포함) 전달 3. 클라이언트: 인증서 유효성 검증 (CA 확인) 4. 대칭 세션 키 교환 (암호화된 채널 수립) 5. 이후 모든 데이터는 세션 키로 암호화 SSL 인증서 종류: DV (Domain Validation): 도메인 소유만 확인 -> Let's Encrypt 무료 발급 OV (Organization): 조직 확인 (기업) EV (Extended): 최고 수준 (은행, 금융) 브라우저 표시: 자물쇠 아이콘 = HTTPS 적용 중 "안전하지 않음" = HTTP만 사용 중
HTTP는 HyperText Transfer Protocol의 약자로, 웹의 데이터 전송 규칙입니다. 1991년에 만들어진 HTTP/1.0부터 현재의 HTTP/3까지 진화해왔습니다. HTTPS는 HTTP에 TLS(Transport Layer Security) 암호화를 추가한 버전입니다. 로그인 폼, 결제 정보, 개인정보가 있는 모든 페이지는 반드시 HTTPS여야 합니다. HTTP만 쓰는 사이트는 Chrome에서 "안전하지 않음"으로 표시되고, 방문자가 이탈할 가능성이 높습니다. Let's Encrypt를 통해 무료로 SSL 인증서를 발급받을 수 있습니다. Vercel과 Netlify는 배포 시 자동으로 HTTPS를 적용해 줍니다. Google 검색 순위에도 HTTPS 여부가 영향을 줍니다. HTTP/2는 한 연결에서 여러 요청을 동시에 처리해 속도를 높이고, HTTP/3는 UDP 기반으로 더욱 빠릅니다.

URL 전체 구조: https://api.mysite.com:443/users/123/posts?page=2&sort=latest#comments | | | | | | | | | | | | 쿼리스트링 앵커(해시) | | | | 리소스 경로 | | | 동적 ID (사용자 123번) | | 포트 (443은 생략 가능) | 서브도메인.도메인.TLD 프로토콜 쿼리스트링 활용 예: ?page=2&sort=latest -- 페이지네이션, 정렬 ?category=react&lang=ko -- 필터링 ?ref=instagram -- 유입 경로 추적 (UTM) ?q=검색어 -- 검색 React Router에서의 URL: / -- 홈 (index route) /about -- 소개 페이지 /posts -- 글 목록 /posts/42 -- 42번 글 상세 (dynamic route) /posts?tag=react -- 태그 필터 (query string)
URL의 각 부분을 이해하는 것은 API 연동, 라우팅 설계, 링크 공유 기능 구현에 필수입니다. 쿼리 파라미터는 상태를 URL에 저장해 링크로 공유 가능하게 만드는 데 씁니다. 예를 들어 필터된 검색 결과, 페이지 번호, 정렬 방식을 URL에 담으면 사용자가 링크를 복사해 공유할 수 있습니다. 앵커(#)는 긴 페이지에서 특정 섹션으로 바로 이동할 때 씁니다. React Router에서는 URL 경로에 따라 어떤 컴포넌트를 렌더링할지 결정합니다. RESTful API 설계에서 URL은 자원을 표현하고, HTTP 메서드는 행동을 표현합니다. URL에 한글이나 특수문자가 들어가면 인코딩(URL Encoding, %EC%9B%B9)이 필요합니다. JavaScript에서 encodeURIComponent()로 처리할 수 있습니다.

도메인IPDNS
DNS 조회 흐름 (상세): 1. 브라우저 캐시 확인 (빠름) 2. OS 캐시 확인 (/etc/hosts 파일) 3. 로컬 DNS 서버(공유기/ISP) 질의 4. 루트 네임서버 질의 5. TLD 서버(.com, .kr) 질의 6. 권한 네임서버(도메인 등록 업체) 응답 naver.com -> 223.130.200.104 DNS 레코드 종류: A 레코드: 도메인 -> IPv4 주소 mysite.com -> 1.2.3.4 CNAME: 도메인 -> 다른 도메인 별칭 www.mysite.com -> mysite.vercel.app MX 레코드: 이메일 서버 지정 TXT 레코드: 도메인 소유 확인, SPF 설정 TTL (Time To Live): DNS 응답을 캐시하는 시간 낮으면: 변경 즉시 반영, DNS 서버 부하 높으면: 빠른 조회, 변경에 시간 걸림 일반적으로 3600초(1시간) ~ 86400초(1일)
DNS는 인터넷의 전화번호부 역할을 합니다. 전 세계에 수천 개의 DNS 서버가 분산되어 있으며, 계층 구조로 운영됩니다. 도메인을 구매하면 해당 도메인의 DNS 레코드를 관리할 수 있습니다. Vercel이나 Netlify에 커스텀 도메인을 연결할 때 CNAME 레코드를 추가하는 작업을 합니다. 이것이 "도메인을 배포 서버에 연결"하는 것의 실제 과정입니다. DNS 변경은 TTL 시간만큼 전파되는 데 시간이 걸립니다. 최대 48시간까지 걸릴 수 있지만, 보통 1~4시간 내에 적용됩니다. DNS 조회 결과는 캐시되므로, 변경 직후에는 사용자마다 다른 결과를 볼 수 있습니다.
HTTP 메서드
GETPOSTREST
메서드별 특징: GET: - URL에 데이터 포함 (?name=value) - 캐싱 가능 (브라우저가 결과를 저장) - 북마크 가능 - 길이 제한 있음 (약 2000자) - 사용: 페이지 로딩, 검색, 데이터 조회 POST: - 데이터를 요청 본문(body)에 담음 - 캐싱 안 됨 - 길이 제한 없음 (파일 업로드 가능) - 사용: 로그인, 회원가입, 파일 업로드 PUT vs PATCH: PUT -> 전체 교체 (빠진 필드는 null) PATCH -> 일부만 수정 (보낸 필드만 변경) 예: 사용자 프로필 수정 시 PUT: { name, email, phone, address } 전부 필요 PATCH: { name: "새이름" } 이름만 보내면 됨 DELETE: - body 없이 URL만으로 삭제 대상 지정 - 멱등성: 여러 번 호출해도 결과 동일
HTTP 메서드는 서버에게 "무엇을 하고 싶은지"를 전달하는 동사 역할입니다. REST API 설계에서 URL은 자원(명사)을 나타내고, 메서드는 행동(동사)을 나타냅니다. 브라우저 주소창으로는 GET 요청만 가능합니다. fetch()나 axios로 POST, PUT, DELETE 등을 보낼 수 있습니다. 안전한(safe) 메서드는 서버 상태를 변경하지 않습니다(GET, HEAD). 멱등(idempotent) 메서드는 같은 요청을 여러 번 보내도 결과가 같습니다(GET, PUT, DELETE). POST는 멱등하지 않으므로, 더블 클릭 방지 처리가 필요합니다. API 문서를 읽을 때 메서드와 URL을 함께 보는 것이 핵심입니다.
실무 포인트 -- Chrome DevTools Network 탭에서 각 요청의 Method 컬럼을 확인하세요. API 문제가 생기면 메서드가 맞는지 먼저 확인하는 것이 디버깅의 첫 단계입니다.
HTTP 헤더 (Header)
헤더인증메타정보
자주 쓰는 요청 헤더: Content-Type: application/json -> 요청 body가 JSON 형식임을 명시 Content-Type: multipart/form-data -> 파일 업로드 시 사용 Authorization: Bearer eyJhbGciOiJIUzI1NiJ9... -> JWT 인증 토큰 전달 Accept: application/json -> JSON 응답을 원함 Accept-Language: ko-KR,ko;q=0.9 -> 한국어 페이지 요청 자주 쓰는 응답 헤더: Content-Type: application/json; charset=utf-8 -> 응답이 JSON임을 알림 Cache-Control: max-age=3600, public -> 1시간 동안 CDN과 브라우저에서 캐시 가능 Cache-Control: no-cache, no-store -> 캐시 금지 (민감한 데이터) Set-Cookie: sessionId=abc; HttpOnly; Secure; SameSite=Strict -> 쿠키 설정 Access-Control-Allow-Origin: https://mysite.com -> CORS 허용 출처 // JS에서 헤더 설정: fetch('/api/users', { method: 'POST', headers: { 'Content-Type': 'application/json', 'Authorization': `Bearer ${token}` }, body: JSON.stringify({ name: '홍길동' }) });
HTTP 헤더는 요청과 응답에 대한 부가 정보를 담는 메타데이터입니다. 본문(body)에는 실제 데이터를 담고, 헤더에는 그 데이터를 어떻게 처리해야 하는지 설명하는 정보를 담습니다. 인증이 필요한 API는 Authorization 헤더에 토큰을 담아 보냅니다. Content-Type을 잘못 설정하면 서버가 body를 파싱하지 못해 에러가 납니다. 캐시 관련 헤더를 적절히 설정하면 성능을 크게 개선할 수 있습니다. Set-Cookie 헤더로 서버가 브라우저에 쿠키를 심습니다. CORS 에러의 실제 해결책도 Access-Control-Allow-Origin 헤더를 서버에서 추가하는 것입니다. DevTools Network 탭에서 각 요청의 Headers 탭을 클릭하면 요청/응답 헤더를 모두 볼 수 있습니다.
HTTP 상태 코드
200404500
상태 코드 전체 분류: 1xx 정보성 (거의 사용 안 함) 100 Continue -- 요청 계속 보내도 됨 2xx 성공 200 OK -- 일반적인 성공 201 Created -- 리소스 생성 성공 (POST) 204 No Content -- 성공했지만 반환 데이터 없음 (DELETE) 3xx 리다이렉트 301 Moved Permanently -- 영구 이동 (SEO에 중요) 302 Found -- 임시 이동 304 Not Modified -- 캐시 사용 (If-Modified-Since) 4xx 클라이언트 오류 (내 잘못) 400 Bad Request -- 요청 형식이 잘못됨 401 Unauthorized -- 인증 필요 (로그인 안 함) 403 Forbidden -- 권한 없음 (로그인해도 접근 불가) 404 Not Found -- 없는 URL 405 Method Not Allowed -- 잘못된 HTTP 메서드 409 Conflict -- 이미 존재하는 데이터 (중복 이메일) 422 Unprocessable -- 유효성 검사 실패 429 Too Many Requests -- API 요청 한도 초과 5xx 서버 오류 (서버 잘못) 500 Internal Server Error -- 서버 내부 오류 502 Bad Gateway -- 게이트웨이 오류 503 Service Unavailable -- 서버 과부하/점검 504 Gateway Timeout -- 응답 시간 초과
상태 코드는 요청이 어떻게 처리됐는지 알려주는 숫자 코드입니다. 프론트엔드 개발자는 2xx(성공)와 4xx(실패) 코드를 기준으로 UI를 처리합니다. 401은 토큰 만료나 로그인이 필요한 경우로, 로그인 페이지로 리다이렉트하거나 토큰 갱신 로직을 실행합니다. 403은 로그인했어도 해당 리소스에 접근 권한이 없는 경우입니다. 404는 URL 오타나 삭제된 리소스입니다. 5xx 에러는 서버 문제이므로 프론트에서 할 수 있는 것은 에러 메시지를 사용자에게 친절하게 보여주는 것뿐입니다. 429는 API를 너무 빠르게 많이 호출할 때 발생하는 Rate Limiting입니다. 상태 코드를 기반으로 에러 메시지를 분기 처리하는 것이 좋은 UX의 핵심입니다.
실무 포인트 -- API 에러 발생 시 상태 코드를 먼저 확인합니다. 401이면 로그인 만료, 404면 URL 오타, 500이면 백엔드 개발자에게 전달합니다.

브라우저의 역할
ChromeSafari엔진
브라우저 구성 요소: 렌더링 엔진: HTML/CSS 해석 -> 화면 그리기 JS 엔진: JavaScript 실행 (Chrome: V8, Safari: JavaScriptCore) 네트워킹: HTTP/HTTPS 요청 처리 스토리지: 쿠키, localStorage, IndexedDB DevTools: 개발자 도구 브라우저별 렌더링 엔진: Chrome/Edge -- Blink (WebKit에서 분기) Safari -- WebKit (iOS는 다른 앱도 WebKit 강제) Firefox -- Gecko 브라우저 점유율 (2025): Chrome: 65% Safari: 19% -- 한국 모바일: iOS 비율 높음 Edge: 4% Firefox: 3% 브라우저 호환성 확인: caniuse.com -- CSS/JS 기능별 지원 현황
브라우저는 HTML, CSS, JS 파일을 사람이 볼 수 있는 화면으로 변환하는 복잡한 프로그램입니다. 렌더링 엔진이 HTML을 해석해 DOM 트리를 만들고, CSS를 해석해 CSSOM 트리를 만든 뒤, 두 트리를 합쳐 화면을 그립니다. JavaScript 엔진이 코드를 실행하고 DOM을 조작합니다. 각 브라우저마다 엔진이 달라 같은 코드가 다르게 동작할 수 있습니다. 특히 Safari(WebKit)는 iOS에서 Chrome, Firefox 등 다른 앱도 강제로 WebKit을 사용하게 되어 있습니다. 즉, iOS에서는 어떤 브라우저를 설치해도 내부 엔진은 Safari와 동일합니다. 이로 인해 "Chrome에서는 되는데 아이폰에서는 안 된다"는 문제가 생깁니다. 개발 시 Chrome을 기준으로 개발하되, 배포 전 Safari와 모바일에서 반드시 테스트해야 합니다. 브라우저의 최신 기능은 caniuse.com에서 지원 현황을 확인하고 사용해야 합니다.
렌더링 순서
DOMCSSOMPaint
Critical Rendering Path (상세): 1. HTML 파싱 -> DOM 트리 생성 <html> <head> <body> 순서로 읽어 내려감 script 태그 만나면 파싱 중단 -> JS 실행 후 재개 2. CSS 파싱 -> CSSOM 트리 생성 link 태그의 외부 CSS 파일 다운로드 CSS는 파싱을 막지 않지만 렌더링을 막음 (CSSOM 없이는 렌더 트리를 못 만들기 때문) 3. DOM + CSSOM -> 렌더 트리 생성 display:none 요소는 렌더 트리에서 제외 4. Layout (Reflow) 각 요소의 크기와 위치 계산 부모가 바뀌면 자식도 재계산 -> 비용 큼 5. Paint (Rasterize) 픽셀로 그리기 (색상, 배경, 이미지) transform, opacity는 GPU 가속 가능 6. Composite 레이어를 합쳐 최종 화면 출력 JS 로딩 최적화: // 나쁜 방법 (파싱 차단) <script src="app.js"></script> // 좋은 방법 <script src="app.js" defer></script> // HTML 파싱 후 실행 <script src="app.js" async></script> // 다운로드 완료 즉시 실행
렌더링 순서를 이해하면 "왜 JS를 body 맨 끝에 두는가", "왜 CSS를 head에서 로딩하는가", "왜 레이아웃을 자주 바꾸면 느려지는가"의 답을 알 수 있습니다. DOM 조작이 잦으면 Reflow(레이아웃 재계산)가 반복되어 성능이 떨어집니다. React의 Virtual DOM이 이 문제를 해결하기 위해 등장했습니다. CSS에서 transform과 opacity를 이용한 애니메이션이 권장되는 이유는 Layout, Paint 단계를 건너뛰고 GPU에서 처리되기 때문입니다. defer 속성은 HTML 파싱을 막지 않고 백그라운드에서 JS를 로드하다 파싱 완료 후 실행합니다. async는 다운로드 완료 즉시 실행되므로 실행 순서가 보장되지 않습니다. 모듈 기반 JS에서는 type="module"을 쓰면 자동으로 defer처럼 동작합니다. Chrome DevTools Performance 탭에서 각 단계의 소요 시간을 분석할 수 있습니다.
핵심 -- script 태그에 defer를 붙이면 HTML 파싱을 막지 않습니다. 항상 defer를 기본으로 쓰세요.
뷰포트 (Viewport)
반응형모바일meta
뷰포트 메타 태그: <meta name="viewport" content="width=device-width, initial-scale=1.0"> width=device-width: 뷰포트 너비 = 기기 화면 너비 initial-scale=1.0: 초기 줌 배율 1배 (원본 크기) 반응형 CSS와의 연결: /* 뷰포트 너비를 기준으로 스타일 분기 */ @media (max-width: 768px) { /* 768px 이하 */ .container { padding: 16px; } .grid { grid-template-columns: 1fr; } } @media (min-width: 1024px) { /* 1024px 이상 */ .container { max-width: 1200px; margin: 0 auto; } } CSS 단위와 뷰포트: vw -- viewport width의 % (100vw = 화면 전체 너비) vh -- viewport height의 % (100vh = 화면 전체 높이) vmin, vmax -- 더 작은/큰 쪽 기준 주의사항: 100vh on iOS: iOS Safari는 주소창 높이가 포함되어 실제 100vh가 화면보다 크게 나오는 문제 해결: min-height: -webkit-fill-available;
뷰포트는 브라우저가 콘텐츠를 표시하는 실제 화면 영역입니다. viewport meta 태그가 없으면 모바일 브라우저는 데스크톱 너비(980px)로 가정해 전체 페이지를 축소해서 보여줍니다. 그러면 글자가 아주 작게 보이고 확대해야 내용을 읽을 수 있게 됩니다. width=device-width를 설정하면 모바일 화면 너비를 뷰포트로 사용합니다. 반응형 웹에서 미디어 쿼리가 뷰포트 너비를 기준으로 작동합니다. vw, vh 단위는 뷰포트 크기에 비례하는 값으로, 화면 꽉 채우기, 글자 크기 조정 등에 활용됩니다. 모바일에서 가로 스크롤이 생기는 문제의 주요 원인은 overflow:hidden이 빠진 요소나 고정 width를 px로 지정한 경우입니다. iOS Safari는 100vh가 주소창 높이를 포함하는 버그가 있어, 전체 화면 섹션을 만들 때 주의가 필요합니다.
캐시 (Cache)
캐시성능새로고침
브라우저 캐시 유형: 메모리 캐시: RAM에 저장 (탭 닫으면 사라짐, 매우 빠름) 디스크 캐시: 하드디스크에 저장 (지속적, 빠름) Service Worker 캐시: 오프라인에서도 동작 가능 캐시 제어 (HTTP 헤더): Cache-Control: max-age=31536000 -- 1년 캐시 (이미지, 폰트, CSS - 해시값으로 파일명 관리) Cache-Control: no-cache -- 캐시하되 매번 서버에 유효성 확인 Cache-Control: no-store -- 캐시 완전 금지 (개인정보 페이지) 번들러의 캐시 전략: // Vite/Webpack이 생성하는 파일명 app-Dj3k9mN2.js // 해시값이 파일명에 포함 style-BNmx4kL1.css // 파일 내용이 변경되면 해시값이 바뀜 // -> 브라우저가 새 파일로 인식해 다시 받음 // -> 캐시 문제 자동 해결 개발 중 캐시 해제: Ctrl+Shift+R (또는 Cmd+Shift+R) -- 강력 새로고침 DevTools > Network > Disable cache (DevTools 열린 동안) DevTools > Application > Storage > Clear site data
캐시는 한 번 받은 파일을 저장해서 다음에 다시 받지 않아도 되도록 하는 메커니즘입니다. 이미지, CSS, JS, 폰트 파일은 보통 오래 캐시됩니다. 캐시 덕분에 재방문 시 로딩이 훨씬 빠릅니다. 개발 중 "수정한 CSS가 반영이 안 된다"면 99% 캐시 문제입니다. Ctrl+Shift+R(강력 새로고침)이나 DevTools의 Disable cache로 해결할 수 있습니다. 배포 환경에서 사용자가 캐시된 구버전을 보는 문제를 방지하기 위해, Vite/Webpack은 빌드 시 파일명에 해시값을 붙입니다. 파일 내용이 바뀌면 해시값이 바뀌므로 브라우저가 새 파일로 인식합니다. 이로써 "캐시는 길게, 내용 변경 시 즉시 갱신"이 동시에 가능합니다. HTTP Cache-Control 헤더로 캐시 기간을 세밀하게 제어할 수 있습니다. CDN과 캐시 전략을 잘 조합하면 서버 비용을 크게 줄이고 속도도 높일 수 있습니다.

JS가 HTML을 움직이는 원리
DOM이란?
DOM트리객체모델
Document Object Model. 브라우저가 HTML을 읽어서 메모리에 올린 트리 구조의 객체 모델입니다. HTML 파일 자체는 텍스트에 불과하지만, 브라우저가 파싱해서 DOM 트리를 생성하면 JavaScript로 조작 가능한 살아있는 객체가 됩니다. DOM은 HTML 문서의 구조, 스타일, 내용을 표현하는 프로그래밍 인터페이스(API)입니다. JS는 이 DOM API를 통해 HTML 요소를 선택하고, 내용을 바꾸고, 스타일을 수정하고, 새 요소를 추가하거나 삭제합니다. DOM을 수정하면 브라우저가 자동으로 화면을 업데이트합니다. 이것이 "버튼을 클릭하면 팝업이 나타나는" 모든 동적 기능의 작동 원리입니다. React는 DOM을 직접 조작하는 대신 Virtual DOM을 통해 변경된 부분만 효율적으로 업데이트합니다. DOM 조작은 느린 작업이므로, 빈번한 DOM 수정은 성능 저하의 주요 원인이 됩니다.
DOM의 계층 구조: window -- 브라우저 창 전체 +-- document -- HTML 문서 전체 +-- html -- <html> 루트 요소 |-- head | |-- title | +-- meta, link, script +-- body |-- header |-- main | +-- section | +-- article +-- footer DOM 노드 종류: Element 노드: <div>, <p> 등 HTML 태그 Text 노드: 태그 안의 텍스트 내용 Comment 노드: <!-- 주석 --> Document 노드: document 객체 자체

HTML의 모든 요소가 트리의 노드로 연결되어 있습니다. 부모-자식-형제 관계가 CSS 선택자 작성과 JS DOM 탐색의 기반이 됩니다. CSS에서 nav > ul > li라고 쓰는 것이 DOM 트리의 부모-자식 관계를 그대로 표현한 것입니다. JS에서 parentElement, children, nextElementSibling 등의 속성으로 트리를 탐색할 수 있습니다. React 컴포넌트 트리도 결국 DOM 트리로 변환됩니다. 깊게 중첩된 DOM 트리는 렌더링 성능에 영향을 줄 수 있으므로, HTML 구조를 가능한 단순하게 유지하는 것이 좋습니다. DevTools Elements 탭에서 DOM 트리를 시각적으로 탐색할 수 있고, 특정 요소를 우클릭해 "중단점 설정"으로 DOM 변경을 추적할 수도 있습니다.

DOM 조작은 JS가 화면을 동적으로 바꾸는 핵심 수단입니다. querySelector는 CSS 선택자를 그대로 사용할 수 있어 직관적입니다. classList.toggle은 메뉴 열기/닫기, 다크모드 전환, 드롭다운 같은 토글 UI에 빠질 수 없는 패턴입니다. textContent로 내용을 바꾸는 것이 innerHTML보다 안전합니다. innerHTML에 사용자 입력값을 직접 넣으면 XSS(크로스 사이트 스크립팅) 공격에 취약합니다. el.style로 인라인 스타일을 직접 수정하는 방법보다, 클래스를 추가/제거해서 CSS에서 스타일을 제어하는 방법이 유지보수에 유리합니다. DOM 조작은 Reflow와 Repaint를 일으키므로, 변경 사항을 최대한 묶어서 한 번에 처리하는 것이 성능에 좋습니다. DocumentFragment를 사용하거나, 요소를 DOM에서 분리 후 수정하고 재삽입하는 방법을 씁니다.

이벤트는 사용자와 웹 페이지가 상호작용하는 모든 순간을 의미합니다. 클릭, 키보드 입력, 스크롤, 마우스 이동 등 모든 행동이 이벤트입니다. addEventListener는 같은 요소에 여러 개를 등록할 수 있으며, 이벤트 핸들러 함수를 인자로 받습니다. 이벤트 객체(e 또는 event)에는 이벤트에 대한 상세 정보가 담겨 있습니다. e.preventDefault()는 링크 클릭 시 페이지 이동, 폼 제출 시 새로고침 같은 브라우저 기본 동작을 막을 때 씁니다. e.target은 실제로 클릭된 요소, e.currentTarget은 이벤트 리스너가 등록된 요소입니다. 둘이 다를 수 있는 이유는 이벤트 버블링 때문입니다. 불필요한 이벤트 리스너는 메모리 누수의 원인이 되므로 removeEventListener로 정리해야 합니다. React에서는 onClick, onSubmit 같은 JSX 속성으로 이벤트를 등록하며, React가 내부적으로 addEventListener를 관리합니다.

// 이벤트 전파 방향 캡처링 (top -> bottom): document -> html -> body -> div -> button 버블링 (bottom -> top): button -> div -> body -> html -> document // 실제 예시 <div class="modal-overlay"> <!-- 3: 실행됨 --> <div class="modal"> <!-- 2: 실행됨 --> <button class="close-btn">닫기</button> <!-- 1: 실행됨 --> </div> </div> // 모달 바깥 클릭 시 닫기 구현 overlay.addEventListener('click', (e) => { if (e.target === overlay) { // 오버레이 자체 클릭 시만 closeModal(); } }); // 또는 모달 안 클릭이 버블링으로 전파 못하게 modal.addEventListener('click', (e) => { e.stopPropagation(); // 버블링 중단 }); // 이벤트 위임 (Event Delegation) 패턴 // 100개의 li에 각각 등록 대신, ul 하나에만 등록 ul.addEventListener('click', (e) => { if (e.target.tagName === 'LI') { e.target.classList.toggle('selected'); } });
이벤트 버블링은 하위 요소에서 발생한 이벤트가 상위 요소로 순서대로 전파되는 현상입니다. 이를 이해하면 복잡한 UI 인터랙션을 간단하게 구현할 수 있습니다. 대표적인 활용이 이벤트 위임(Delegation) 패턴입니다. 100개의 리스트 아이템에 각각 이벤트를 등록하는 대신, 부모 요소 하나에 이벤트를 등록하고 어떤 자식이 클릭됐는지 e.target으로 확인합니다. 메모리 효율적이고, 동적으로 추가된 요소에도 자동으로 이벤트가 적용됩니다. 모달 바깥 클릭 시 닫기도 버블링을 활용합니다. 오버레이(배경)에 클릭 이벤트를 등록하되, 모달 안쪽 클릭은 stopPropagation으로 전파를 막습니다. 캡처링(top-down)은 addEventListener의 세 번째 인자를 true로 설정할 때 사용하며, 특수한 경우에만 씁니다.

Virtual DOM은 실제 DOM의 메모리 상 복사본입니다. React는 상태가 변경될 때마다 새로운 Virtual DOM 트리를 만들고, 이전 Virtual DOM과 비교(Diffing)해서 어떤 노드가 바뀌었는지 최소한으로 계산합니다. 그리고 실제로 변경된 부분만 실제 DOM에 반영(Reconciliation)합니다. 이것이 React가 효율적인 이유입니다. React에서 state를 직접 수정하면 안 되는 이유가 여기 있습니다. 직접 수정하면 이전 Virtual DOM과 비교할 기준이 없어져 React가 변경을 감지하지 못합니다. 반드시 setState나 useState setter를 사용해야 React가 변경을 감지하고 재렌더링합니다. list를 렌더링할 때 key prop이 필요한 이유도 Diffing 알고리즘과 관련 있습니다. key로 각 아이템을 식별해야 어떤 아이템이 추가/삭제/이동됐는지 효율적으로 판단할 수 있습니다. React 18의 Concurrent Mode는 렌더링을 중단하고 더 급한 업데이트(사용자 입력)를 먼저 처리하는 기능입니다.
핵심 -- React를 배울 때 "왜 state를 직접 수정하면 안 되나?"의 답이 Virtual DOM Diffing 메커니즘에 있습니다. setState를 통해서만 React가 변경을 감지할 수 있습니다.

서버와 데이터를 주고받는 방법
API란?
API인터페이스약속
Application Programming Interface. 서버가 데이터를 주고받기 위해 열어둔 창구입니다. "이 주소로 이렇게 요청하면 이 데이터를 줄게"라는 약속입니다. 날씨 앱, 지도 서비스, 소셜 로그인 모두 외부 API를 활용합니다. 프론트엔드 개발자는 API를 직접 만들지 않고 사용하는 방법을 익힙니다. API는 웹 API만 있는 것이 아닙니다. 브라우저 API(DOM, fetch, localStorage), 운영체제 API, 라이브러리 API 등 소프트웨어 간 통신 규약을 모두 API라고 부릅니다. 웹 개발에서 "API"라고 하면 보통 REST API(HTTP 기반 서버 API)를 의미합니다. API 문서를 읽고 이해하는 능력이 프론트엔드 개발의 핵심 역량 중 하나입니다. Swagger나 Postman 같은 도구로 API를 테스트할 수 있습니다.
날씨 앱 --> 기상청 API --> 날씨 데이터 지도 앱 --> 카카오맵 API --> 지도 타일 로그인 --> 카카오 로그인 API --> 사용자 정보 API 종류: 브라우저 API: DOM, fetch, localStorage, Geolocation 서버 API: REST API, GraphQL API 외부 API: 카카오, 네이버, Google 등 제공 라이브러리 API: React, axios 등의 함수/메서드 API 테스트 도구: Postman -- GUI로 API 요청 테스트 Thunder Client -- VS Code 확장 curl -- 터미널에서 API 호출

현업에서 가장 많이 쓰는 API 설계 방식입니다. URL은 자원(명사), HTTP 메서드는 행동(동사)으로 표현합니다. REST API 문서를 읽고 이해하는 능력이 프론트엔드 개발의 핵심 역량 중 하나입니다. 잘 설계된 REST API는 URL만 봐도 어떤 데이터를 다루는지 직관적으로 알 수 있습니다. 중첩 자원은 /users/5/posts처럼 계층적으로 표현합니다. 페이지네이션, 필터, 정렬은 쿼리 파라미터로 처리합니다. 실무에서는 API 응답에 성공/실패 여부, 데이터, 페이지네이션 정보가 함께 옵니다. GraphQL은 REST의 대안으로, 클라이언트가 원하는 데이터 구조를 직접 지정할 수 있습니다. 하지만 대부분의 프로젝트에서는 REST API로 충분합니다.

CORS (Cross-Origin Resource Sharing)
CORS보안출처
// 브라우저 콘솔에 뜨는 에러 Access to fetch at 'https://api.example.com' from origin 'http://localhost:3000' has been blocked by CORS policy. // 출처(Origin) = 프로토콜 + 도메인 + 포트 http://localhost:3000 -- 출처 A http://localhost:8080 -- 출처 B (포트 다름 = 다른 출처!) https://mysite.com -- 출처 C // 해결 방법 1: 서버에서 헤더 추가 Access-Control-Allow-Origin: http://localhost:3000 // 해결 방법 2: 개발 환경 프록시 (Vite) // vite.config.js server: { proxy: { '/api': 'http://localhost:8080' } } // 해결 방법 3: 배포 시 같은 도메인 사용 // mysite.com (프론트) + mysite.com/api (백엔드)
보안을 위해 브라우저가 다른 출처의 API 요청을 기본으로 차단하는 정책입니다. API 연동 시 100% 마주치는 에러입니다. 출처(Origin)는 프로토콜, 도메인, 포트 세 가지를 합친 것이며, 하나라도 다르면 다른 출처입니다. CORS는 브라우저의 보안 정책이므로 서버-서버 간 요청에서는 발생하지 않습니다. 서버에서 Access-Control-Allow-Origin 헤더를 추가하는 것이 정석 해결법입니다. 개발 환경에서는 Vite의 프록시 설정으로 우회할 수 있습니다. 프록시는 브라우저가 같은 출처(localhost:5173)로 요청하게 만들고, Vite 서버가 대신 백엔드에 전달하는 방식입니다. 배포 환경에서는 동일한 도메인을 사용하거나, 서버에 CORS 설정을 추가합니다.
실무 포인트 -- CORS 에러는 프론트 문제가 아닙니다. 백엔드 개발자에게 "Allow-Origin 헤더 추가 요청"이라고 정확히 전달하세요.

fetch는 브라우저 내장 API 호출 함수입니다. async/await으로 비동기 코드를 순서대로 읽기 쉽게 작성할 수 있습니다. fetch는 네트워크 에러만 reject하고 HTTP 에러(404, 500)는 reject하지 않으므로, res.ok를 반드시 확인해야 합니다. 에러 처리를 함께 넣는 것이 실무 기본입니다. axios 라이브러리를 쓰면 HTTP 에러도 자동으로 catch되고, 요청/응답 인터셉터, 요청 취소 등 편리한 기능을 제공합니다. 인증 & 토큰 (JWT)
JWT인증토큰
로그인 흐름: 1. 클라이언트 --> 서버: ID/PW 전송 2. 서버 --> 클라이언트: JWT 토큰 발급 3. 이후 요청마다 헤더에 토큰 포함 Authorization: Bearer eyJhbGci... 4. 서버가 토큰 검증 후 데이터 응답 JWT 구조: 헤더.페이로드.서명 eyJhbGci.eyJ1c2Vy.SflKxwRJ | | | 알고리즘 사용자정보 위변조검증 토큰 저장 위치: localStorage -- 간편하지만 XSS에 취약 httpOnly 쿠키 -- JS 접근 불가, 더 안전 메모리(변수) -- 새로고침 시 사라짐, 가장 안전 토큰 갱신: Access Token: 만료 짧음 (15분~1시간) Refresh Token: 만료 김 (7일~30일) Access Token 만료 시 Refresh Token으로 재발급
로그인 상태를 유지하는 방법입니다. JWT(JSON Web Token)는 서버가 발급하는 암호화된 인증 증명서입니다. 토큰이 유효하면 로그인된 것으로 인정합니다. JWT는 헤더(알고리즘 정보), 페이로드(사용자 정보), 서명(위변조 방지)으로 구성됩니다. 서버는 비밀키로 서명을 생성하고, 이후 요청에서 서명을 검증해 토큰이 위변조되지 않았는지 확인합니다. 토큰은 localStorage 또는 httpOnly 쿠키에 저장합니다. 보안이 중요한 서비스는 httpOnly 쿠키를 권장합니다. Access Token은 만료가 짧고, Refresh Token으로 재발급합니다. 프론트엔드에서는 401 응답을 받으면 자동으로 토큰 갱신 로직을 실행하는 인터셉터를 구현합니다.

로그인 상태 유지, 장바구니 기억, 광고 추적 등에 사용됩니다. 서버가 Set-Cookie 헤더로 심고 브라우저가 자동으로 관리합니다. HttpOnly 쿠키는 JS로 접근이 불가해 XSS 공격으로부터 안전합니다. 보안이 중요한 인증 토큰은 httpOnly 쿠키에 저장하는 것이 권장됩니다. SameSite 속성은 CSRF(크로스 사이트 요청 위조) 공격을 방지합니다. 쿠키는 도메인별로 분리되어 있어 naver.com의 쿠키를 google.com이 읽을 수 없습니다. 서드파티 쿠키(광고 추적용)는 개인정보 보호 강화로 Chrome에서도 차단되는 추세입니다. DevTools Application 탭에서 쿠키 목록과 속성을 직접 확인하고 수정, 삭제할 수 있습니다.
세션 (Session)
세션서버저장인증
서버 메모리 or DB: sessions = { "abc123": { userId: 42, role: "admin", loginAt: "..." } } 브라우저 쿠키: sessionId=abc123 -- 실제 데이터는 서버에, ID만 클라이언트에 세션 vs JWT 비교: 세션: 서버에 상태 저장 (stateful) 장점: 서버에서 즉시 무효화 가능 단점: 서버 메모리 사용, 서버 확장 어려움 JWT: 토큰에 정보 포함 (stateless) 장점: 서버 부하 적음, 확장 쉬움 단점: 즉시 무효화 어려움 (만료까지 유효) 세션 스토어: Redis -- 인메모리 DB, 빠름 (가장 많이 씀) MySQL -- 영구 저장, 느림 Memory -- 서버 재시작 시 초기화
보안이 중요한 데이터(로그인 정보, 권한)는 서버에 저장하고, 브라우저에는 세션 ID만 쿠키로 저장합니다. 세션 방식은 서버가 상태를 관리하므로 stateful합니다. 관리자가 특정 사용자의 세션을 즉시 무효화(강제 로그아웃)할 수 있다는 장점이 있습니다. 반면 서버가 여러 대일 때 세션을 공유하는 것이 복잡해지는 단점이 있습니다. Redis 같은 외부 세션 스토어를 사용하면 여러 서버가 세션을 공유할 수 있습니다. JWT는 서버에 상태를 저장하지 않아 확장성이 좋지만, 토큰을 즉시 무효화하기 어렵습니다. 현대 웹 서비스 대부분은 JWT 방식을 사용하며, 이 수업에서도 JWT를 다룹니다. 세션과 JWT의 장단점을 이해하면 면접에서 좋은 인상을 줄 수 있습니다.

localStorage브라우저저장영구
// 저장 localStorage.setItem('theme', 'dark'); localStorage.setItem('user', JSON.stringify({ id: 1, name: '홍길동' })); // 읽기 const theme = localStorage.getItem('theme'); const user = JSON.parse(localStorage.getItem('user')); // 삭제 localStorage.removeItem('theme'); localStorage.clear(); // 전체 삭제 // 존재 여부 확인 if (localStorage.getItem('theme')) { applyTheme(localStorage.getItem('theme')); } 특징: 용량: 5~10MB (브라우저마다 다름) 수명: 직접 삭제하기 전까지 영구 보존 범위: 같은 출처(origin) 내에서 공유 전송: 서버로 자동 전송 안 됨 타입: 문자열만 저장 (객체는 JSON.stringify 필요)
다크모드 설정, 최근 검색어, 임시 저장 데이터, 사용자 선호 언어 등 UI 상태 저장에 적합합니다. 서버로 자동 전송되지 않아 쿠키보다 용량이 크고 서버 부하가 없습니다. 민감한 정보(비밀번호, 토큰)는 저장하면 XSS에 취약하므로 주의가 필요합니다. 문자열만 저장 가능하므로 객체는 JSON.stringify로 변환해야 합니다. 같은 출처(프로토콜+도메인+포트)의 모든 탭에서 공유됩니다. 시크릿 모드에서는 브라우저를 닫으면 삭제됩니다. React에서는 커스텀 훅(useLocalStorage)을 만들어 state와 localStorage를 동기화하는 패턴이 흔합니다. DevTools Application 탭에서 값을 직접 확인, 수정, 삭제할 수 있습니다.

sessionStorage임시저장탭
// localStorage와 동일한 API sessionStorage.setItem('step', '2'); sessionStorage.getItem('step'); sessionStorage.removeItem('step'); 차이점: localStorage -- 브라우저 닫아도 유지, 모든 탭 공유 sessionStorage -- 탭 닫으면 사라짐, 탭별 독립 활용 예: 결제 진행 단계 저장 (step 1 -> 2 -> 3) 다단계 폼 중간 저장 일회성 팝업 "오늘 그만 보기" 페이지 이동 전 스크롤 위치 저장
결제 단계별 입력값, 다단계 폼 데이터처럼 탭 세션 내에서만 유지하면 되는 임시 데이터에 사용합니다. 같은 사이트를 새 탭에서 열면 sessionStorage는 공유되지 않고 각각 독립적으로 작동합니다. localStorage와 API가 완전히 동일하므로 교체가 쉽습니다. 새로고침해도 유지되지만 탭을 닫으면 사라지므로, "한 번의 방문 동안만 유지해야 하는 데이터"에 적합합니다. 실무에서는 localStorage를 더 자주 사용하지만, 보안이 필요한 임시 데이터는 sessionStorage가 적합합니다. 결제 페이지에서 이전 단계 입력값을 sessionStorage에 저장하면 뒤로가기해도 데이터가 유지됩니다. 탭을 닫으면 자동으로 삭제되므로 정리할 필요가 없다는 장점도 있습니다.

호스팅이란?
호스팅서버배포
내가 만든 웹 파일(HTML/CSS/JS)을 항상 켜져 있는 서버에 올려서 누구나 접속할 수 있게 하는 것입니다. 서버 컴퓨터를 빌리는 행위입니다. 로컬에서만 보이던 사이트가 전 세계에서 접속 가능해지는 과정입니다. 포트폴리오를 면접관에게 링크로 공유할 수 있게 됩니다. 호스팅 서비스는 크게 공유 호스팅, VPS, 클라우드, 정적 호스팅으로 나뉩니다. 프론트엔드 개발자는 정적 호스팅(Vercel, Netlify)을 가장 많이 사용합니다. 배포(Deploy)는 개발 완료된 코드를 호스팅 서버에 올리는 행위를 말합니다. 현대 프론트엔드 배포는 Git push 한 번으로 자동화되어 있습니다.
내 컴퓨터: localhost:3000 -- 나만 접속 가능 호스팅 서버: https://mysite.com -- 전 세계 접속 가능 호스팅 유형: 공유 호스팅: 여러 사이트가 한 서버 공유 (저렴) VPS: 가상 서버 1개 독점 (중간) 클라우드: AWS EC2 등 (유연, 종량제) 정적 호스팅: HTML/CSS/JS만 서비스 (무료 가능) 프론트엔드 개발자 추천: 정적 사이트 -> Vercel / Netlify (무료) 풀스택 앱 -> Vercel (서버리스 함수 포함) 백엔드 필요 -> Railway / Render / AWS

VercelNetlifyGitHub Pages
GitHub Pages -- github.io 도메인 무료 제공 git push 하면 자동 배포 제한: 정적 파일만, 빌드 과정 제한적 Vercel -- vercel.app 도메인 무료 React/Next.js 배포 특화 GitHub 연동 -> push하면 자동 반영 Preview URL: PR마다 별도 미리보기 URL 생성 서버리스 함수 지원 (간단한 API 가능) Netlify -- netlify.app 도메인 무료 드래그 앤 드롭으로도 배포 가능 폼 기능 내장 (백엔드 없이 폼 제출 처리) Split Testing (A/B 테스트) 지원 배포 과정 (Vercel 기준): 1. GitHub에 코드 push 2. Vercel이 자동으로 감지 3. npm run build 실행 4. dist 폴더를 CDN에 배포 5. URL 생성 완료 (약 30초~1분)
HTML/CSS/JS만 있는 정적 사이트는 무료로 배포 가능합니다. 이 수업의 포트폴리오는 여기서 배포합니다. Vercel은 GitHub 레포와 연결하면 코드를 push할 때마다 자동으로 최신 버전이 배포됩니다. Pull Request를 열면 Preview URL이 자동 생성되어 배포 전 미리 확인할 수 있습니다. Netlify는 드래그 앤 드롭으로도 배포 가능해 초보자에게 친숙합니다. GitHub Pages는 GitHub 저장소에서 바로 배포할 수 있어 간단하지만, 빌드 과정이 제한적입니다. Vercel의 무료 플랜은 개인 프로젝트에 충분하며, 월 100GB 대역폭, 서버리스 함수 호출 10만 회까지 지원합니다. 포트폴리오는 반드시 온라인에 배포하세요.
핵심 -- 포트폴리오는 반드시 온라인에 배포하세요. "링크 있어요"와 "없어요"의 차이는 취업에서 큽니다.
도메인 연결
도메인DNSHTTPS
1. 도메인 구매: Gabia, 후이즈, Namecheap .com 연 약 15,000원 .kr 연 약 18,000원 .dev 연 약 15,000원 (HTTPS 강제) 2. DNS 설정: 구매한 도메인 -> 내 서버 IP 연결 A 레코드: 도메인 -> IP 주소 CNAME: 도메인 -> 다른 도메인으로 연결 (Vercel: CNAME cname.vercel-dns.com) 3. HTTPS 설정: Vercel/Netlify: 자동 적용 (설정 불필요) 직접 서버: Let's Encrypt (무료 SSL 인증서) 갱신: certbot으로 90일마다 자동 갱신 4. 연결 확인: DNS 전파: 최대 48시간 (보통 1~4시간) whatsmydns.net에서 전파 상태 확인
Vercel/Netlify는 구매한 도메인을 몇 번의 클릭으로 연결할 수 있습니다. 포트폴리오에 자기 도메인을 달면 훨씬 전문적으로 보입니다. 도메인 구매는 연 1~2만원 수준으로 부담이 적습니다. .dev 도메인은 Google이 관리하며 HTTPS가 강제되어 개발자 포트폴리오에 인기가 높습니다. DNS 설정은 도메인 등록 업체의 관리 페이지에서 CNAME 레코드를 추가하는 것이 전부입니다. Vercel은 도메인을 추가하면 자동으로 SSL 인증서를 발급하고 HTTPS를 적용합니다. DNS 변경 후 전파에 시간이 걸리므로 즉시 반영되지 않을 수 있습니다. whatsmydns.net에서 전 세계 DNS 서버의 전파 상태를 실시간으로 확인할 수 있습니다.
CDN (Content Delivery Network)
CDN성능Cloudflare
CDN 없음: 한국 사용자 -> 미국 서버 -> 느림 (200ms+) CDN 있음: 한국 사용자 -> 한국 CDN 서버 -> 빠름 (20ms) CDN 작동 원리: 1. 사용자가 mysite.com 접속 2. DNS가 가장 가까운 CDN 서버로 연결 3. CDN 서버에 캐시된 파일이 있으면 즉시 응답 4. 캐시가 없으면 원본 서버에서 가져와 캐시 후 응답 주요 CDN 서비스: Cloudflare -- 무료 CDN + 보안 + DNS + DDoS 방어 AWS CloudFront -- AWS 생태계 연동 Vercel/Netlify -- CDN이 기본 내장 캐시 무효화: 파일명 해시로 자동 처리 (app-Dk3m.js) 수동: Purge Cache (CDN 관리자 페이지)
전 세계 각지에 캐시 서버를 두고, 사용자에게 가장 가까운 서버에서 파일을 전달하는 기술입니다. 이미지, JS 파일 등 정적 파일 로딩 속도가 크게 빨라집니다. Vercel과 Netlify는 CDN이 기본 내장되어 있어 별도 설정 없이 혜택을 받을 수 있습니다. Cloudflare는 무료 플랜에서도 CDN, DNS, DDoS 방어를 제공합니다. CDN은 정적 파일에 효과적이고, API 응답 같은 동적 데이터에는 적용이 제한적입니다. 번들러가 파일명에 해시를 붙이면 CDN 캐시가 자동으로 갱신됩니다. 한국에 CDN 엣지 서버가 있는 서비스를 선택하면 국내 사용자 경험이 좋아집니다. Vercel은 서울 리전(ICN)을 지원하므로 한국 사용자에게 빠른 응답 속도를 제공합니다.
CI/CD (자동 배포)
CICD자동배포
CI (Continuous Integration): 코드 변경 시 자동 테스트 CD (Continuous Deployment): 테스트 통과 시 자동 배포 Vercel 기준: GitHub main 브랜치에 push -> Vercel이 자동으로 빌드 -> 새 버전 배포 완료 -> 이전 버전은 자동 보관 (롤백 가능) GitHub Actions 예시: name: Deploy on: push: branches: [main] jobs: build: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - run: npm install - run: npm run build - run: npm run test - uses: deploy-action PR 기반 워크플로우: 1. feature 브랜치에서 개발 2. PR 생성 -> Preview URL 자동 생성 3. 코드 리뷰 + 테스트 통과 확인 4. main에 머지 -> 자동 배포
코드를 올리면 자동으로 빌드하고 배포되는 파이프라인입니다. Vercel/Netlify는 GitHub 연동만 하면 기본으로 제공합니다. 배포 실패 시 이전 버전으로 자동 롤백되어 서비스 중단 없이 운영할 수 있습니다. GitHub Actions는 GitHub에서 제공하는 CI/CD 도구로, 테스트 자동화, 린트 검사, 빌드, 배포를 워크플로우로 정의할 수 있습니다. PR 기반 워크플로우에서는 feature 브랜치에서 개발하고, PR을 생성하면 Preview URL이 자동으로 만들어져 배포 전 미리 확인할 수 있습니다. 코드 리뷰와 테스트가 통과하면 main에 머지하고 자동 배포됩니다. 이 과정이 자동화되면 개발자는 코드 작성에만 집중할 수 있습니다. 포트폴리오 프로젝트에서도 이 워크플로우를 적용하면 실무 경험을 어필할 수 있습니다.

번들러
파일들을 묶어 빠르게 만드는 도구
번들러가 왜 필요한가?
번들링최적화요청
파일 100개를 브라우저가 따로 요청: HTML, CSS, JS, 이미지... 요청 100번 -> 느림 번들러 사용 후: JS 파일 수십 개 -> app.js 하나로 합침 CSS 파일 여러 개 -> style.css 하나로 합침 -> 요청 횟수 감소, 로딩 빠름 번들러가 하는 일: 1. 모듈 해석: import/require 의존성 추적 2. 번들링: 여러 파일을 하나로 합침 3. 변환: JSX -> JS, TS -> JS, Sass -> CSS 4. 최적화: 코드 압축, 불필요 코드 제거 5. 개발 서버: HMR, 자동 새로고침
React 프로젝트는 파일이 수십에서 수백 개입니다. 번들러가 이를 최적화해서 브라우저가 빠르게 로딩할 수 있는 파일로 만들어 줍니다. npm run build 한 번으로 배포용 파일이 자동으로 생성됩니다. 번들러 없이는 import/export 문법을 브라우저에서 직접 사용할 수 없습니다(ESM 모듈 제외). JSX(React 문법), TypeScript도 번들러를 거쳐야 브라우저가 이해할 수 있는 JS로 변환됩니다. 개발 중에는 HMR(Hot Module Replacement)로 코드 수정 시 페이지 새로고침 없이 변경 사항이 즉시 반영됩니다. 이것이 개발 속도를 크게 높여주는 기능입니다. 번들러는 프론트엔드 개발 인프라의 핵심이므로 기본 원리를 이해하는 것이 중요합니다.

번들러가 하는 일들
tree-shakingBabel최적화
모듈 번들링: import/require를 하나로 합침 트랜스파일: 최신 JS -> 구 브라우저 호환 JS (Babel) JSX -> createElement() 함수 호출 TypeScript -> JavaScript CSS 처리: Sass/Less -> CSS 변환 PostCSS: Autoprefixer 등 플러그인 처리 CSS Modules: 클래스명 충돌 방지 이미지 최적화: 작은 이미지는 Base64로 인라인 큰 이미지는 별도 파일로 복사 Tree shaking: 사용하지 않는 코드(export) 제거 코드 분할: 필요한 페이지의 JS만 로딩 (lazy) React.lazy() + Suspense Minification: 공백, 주석 제거, 변수명 축약 Source Map: 압축된 코드 -> 원본 코드 매핑 (디버깅용)
번들러는 단순히 합치는 것을 넘어 코드를 최적화합니다. npm run build 한 번으로 이 모든 작업이 자동으로 실행됩니다. Tree-shaking으로 사용하지 않는 코드를 제거해 번들 크기를 줄입니다. 코드 분할(Code Splitting)은 초기 로딩 속도를 높이는 핵심 기법입니다. 모든 페이지의 JS를 한 번에 받는 대신, 현재 페이지에 필요한 JS만 먼저 받고 나머지는 필요할 때 로딩합니다. React에서는 React.lazy()와 Suspense로 컴포넌트 단위의 코드 분할이 가능합니다. Source Map은 빌드된 압축 코드에서 에러가 발생했을 때 원본 소스 코드의 위치를 알려주므로 디버깅에 필수적입니다. 개발 환경에서는 활성화하고 프로덕션에서는 보안을 위해 비활성화하는 것이 일반적입니다.
SPA vs MPA
SPAMPAReact
MPA (전통적): / (홈) -> 서버: home.html 전송 /about -> 서버: about.html 전송 (새로고침) /products -> 서버: products.html 전송 (새로고침) SPA (React): 처음에 index.html 하나만 받음 이후 JS Router가 URL에 따라 화면을 교체 서버 요청 없음 -> 부드러운 전환 SSR (Next.js): 서버에서 첫 HTML을 완성해서 전달 (SEO 유리) 이후는 SPA처럼 클라이언트에서 라우팅 = "초기 로딩은 서버, 이후는 클라이언트" SSG (Static Site Generation): 빌드 시점에 모든 페이지를 HTML로 생성 블로그, 문서 사이트에 적합 (Gatsby, Astro)
SPA(Single Page Application)는 HTML을 처음 한 번만 받고, 이후 JS가 페이지를 동적으로 교체합니다. MPA(Multi Page Application)는 페이지 이동할 때마다 서버에서 새 HTML을 받습니다. React는 SPA 방식이며, 이것이 "새로고침 없이 부드럽게 전환되는" 경험을 만드는 원리입니다. SPA의 단점은 초기 로딩이 느리고 SEO가 불리하다는 것입니다. Next.js의 SSR은 이 단점을 보완합니다. 서버에서 첫 HTML을 완성해서 보내므로 검색 엔진이 내용을 읽을 수 있고, 초기 로딩도 빠릅니다. SSG는 빌드 시점에 모든 페이지를 미리 만들어두는 방식으로, 블로그나 문서 사이트에 적합합니다. 전통적인 웹사이트는 MPA 방식이었으나, 현대 웹앱 대부분은 SPA로 전환되었습니다.
실무 포인트 -- React로 만든 앱이 "새로고침 없이 빠르게 전환"되는 이유가 SPA 구조 때문입니다. SEO가 중요한 쇼핑몰, 블로그는 SSR(Next.js)을 함께 씁니다.

느린 페이지는 사용자 이탈, 매출 감소, 검색 순위 하락으로 직결됩니다. 취업 후 성능 개선은 바로 비즈니스 성과로 연결됩니다. 포트폴리오에서 Lighthouse 점수를 높게 유지하면 기술력을 증명하는 수단이 됩니다. Google은 2021년부터 Core Web Vitals를 검색 순위 요소에 포함시켰습니다. 모바일 사용자는 데스크톱보다 네트워크가 느리고 CPU 성능이 낮으므로 성능 최적화가 더 중요합니다. 성능은 개발 초기부터 고려해야 합니다. 나중에 최적화하려면 비용이 훨씬 많이 듭니다. 성능 측정은 한 번이 아니라 지속적으로 모니터링해야 합니다. 배포 후 실제 사용자 환경에서의 성능(Field Data)이 실험실 환경(Lab Data)보다 중요합니다.
Core Web Vitals
LCPCLSGoogle
LCP (Largest Contentful Paint) -- 가장 큰 콘텐츠가 로딩되는 시간 -- 좋음: 2.5초 이하 / 나쁨: 4초 이상 -- 개선: 이미지 최적화, 서버 응답 단축, CSS 인라인 INP (Interaction to Next Paint) -- FID 대체 -- 사용자 입력에 반응하는 시간 -- 좋음: 200ms 이하 -- 개선: JS 실행 최적화, 메인 스레드 블로킹 방지 CLS (Cumulative Layout Shift) -- 화면이 갑자기 이동하는 정도 -- 좋음: 0.1 이하 -- 원인: 이미지 크기 미지정, 동적 콘텐츠 삽입 -- 개선: width/height 명시, font-display: swap TTFB (Time To First Byte) -- 서버 첫 응답까지 시간 -- 좋음: 800ms 이하 -- 서버 성능, CDN, 캐싱에 영향 받음
Google이 정의한 핵심 성능 지표입니다. 이 지표들이 Google 검색 순위에 직접 영향을 줍니다. LCP는 사용자가 "페이지가 로딩됐다"고 느끼는 시점을 측정합니다. 히어로 이미지나 큰 텍스트 블록이 대상입니다. INP는 2024년에 FID를 대체한 새로운 지표로, 전체 페이지 수명 동안의 모든 상호작용 응답성을 측정합니다. CLS는 이미지에 width/height를 명시하지 않으면 로딩 중 레이아웃이 흔들리는 문제입니다. 광고, 웹폰트, 동적으로 삽입되는 콘텐츠도 CLS의 주요 원인입니다. PageSpeed Insights에서 실제 사용자 데이터(Field Data)와 실험실 데이터(Lab Data)를 모두 확인할 수 있습니다. 포트폴리오에서 "Core Web Vitals 전 항목 녹색" 달성은 좋은 어필 포인트입니다.

접근성이란?
a11yWCAG포용성
시각 장애, 청각 장애, 지체 장애, 인지 장애가 있는 사용자도 웹을 이용할 수 있도록 설계하는 것입니다. 공공기관 웹사이트는 법적 의무 사항입니다. 접근성은 장애인만을 위한 것이 아니라, 어두운 환경에서 스마트폰을 쓰거나 한 손이 바쁜 모든 사람에게 도움이 됩니다. WCAG(Web Content Accessibility Guidelines)는 W3C가 정한 국제 접근성 표준입니다. 한국은 KWCAG(한국형 웹 콘텐츠 접근성 지침) 2.2를 적용합니다. 접근성을 고려한 웹사이트는 SEO에도 유리합니다. 스크린 리더가 읽을 수 있는 구조는 검색 엔진도 잘 이해합니다. 미국에서는 ADA(장애인 차별 금지법) 위반으로 웹사이트 소송이 증가하고 있어, 글로벌 서비스에서 접근성은 필수입니다.
접근성이 필요한 사용자: 시각 장애: 스크린 리더 사용 (NVDA, VoiceOver) 저시력: 화면 확대, 고대비 모드 색각 이상: 색상만으로 정보 전달 불가 청각 장애: 동영상 자막 필요 지체 장애: 키보드만으로 조작 인지 장애: 명확한 언어, 단순한 UI 스크린 리더 -- 화면을 읽어주는 프로그램 키보드 탐색 -- 마우스 없이 Tab키로 이동 고대비 모드 -- 색상 구분이 어려운 사용자 WCAG 등급: A: 최소 기준 (반드시 준수) AA: 권장 기준 (대부분 사이트 목표) AAA: 최고 수준 (특수 목적 사이트)

스크린 리더는 alt 텍스트로 이미지를 읽어줍니다. 시맨틱 태그를 쓰면 자동으로 접근성이 높아집니다. 장식용 이미지는 alt=""로 설정하면 스크린 리더가 건너뜁니다. 차트나 그래프 이미지는 alt에 데이터의 핵심 내용을 텍스트로 설명합니다. button 태그는 키보드 포커스, 클릭 이벤트, 스크린 리더 인식이 기본으로 지원됩니다. div에 onClick을 추가하면 이 모든 것을 수동으로 구현해야 합니다. 시맨틱 태그(header, nav, main, footer)는 스크린 리더가 페이지 구조를 파악하는 데 핵심적인 역할을 합니다. 제목 태그(h1~h6)는 계층 순서를 지켜야 합니다. h1 다음에 h3가 오면 스크린 리더 사용자가 혼란을 느낍니다.

마우스를 쓸 수 없는 사용자는 Tab키로만 탐색합니다. outline을 CSS로 제거하는 것은 접근성을 해치는 나쁜 관행입니다. :focus-visible을 사용하면 키보드 포커스에만 outline을 표시하고 마우스 클릭 시에는 표시하지 않아 디자인과 접근성을 모두 잡을 수 있습니다. 모달 창이 열리면 포커스가 모달 안에 갇혀야(focus trap) 합니다. Tab을 눌러도 모달 밖의 요소로 이동하면 안 됩니다. ESC 키로 모달을 닫을 수 있어야 합니다. 대화형 요소(버튼, 링크, 입력)는 기본적으로 포커스 가능합니다. div, span 같은 비대화형 요소에 클릭 기능을 추가하려면 tabindex="0"과 키보드 이벤트(Enter, Space)를 함께 처리해야 합니다. 하지만 가능하면 button이나 a 태그를 사용하는 것이 훨씬 간단하고 안전합니다.

ARIA는 HTML만으로 전달할 수 없는 동적 상태와 관계를 스크린 리더에 알려줍니다. aria-expanded는 아코디언, 드롭다운 같은 펼침/접힘 UI에 필수입니다. aria-live는 AJAX로 동적으로 변경되는 내용(알림, 검색 결과 수)을 스크린 리더에 알립니다. 중요한 원칙은 "ARIA를 쓰기 전에 HTML 시맨틱 태그로 해결할 수 있는지 먼저 확인"하는 것입니다. button 태그가 있는데 div에 role="button"을 쓰는 것은 불필요합니다.

폼 입력 요소는 반드시 label과 연결해야 스크린 리더가 "이메일 입력란"처럼 읽어줄 수 있습니다. placeholder만으로는 접근성이 보장되지 않습니다. placeholder는 입력하면 사라져서 무엇을 입력해야 하는지 잊게 만들기 때문에 label을 항상 함께 써야 합니다. 에러 메시지도 aria-describedby로 연결하면 스크린 리더가 에러를 읽어줍니다. role="alert"는 에러 메시지가 동적으로 나타날 때 스크린 리더가 즉시 알리도록 합니다. sr-only 클래스는 화면에는 보이지 않지만 스크린 리더가 읽을 수 있게 하는 기법입니다. 검색 입력란처럼 디자인상 label을 보이게 하기 어려운 경우에 사용합니다. display:none이나 visibility:hidden은 스크린 리더도 무시하므로 사용하면 안 됩니다.

React 프로젝트에서는 eslint-plugin-jsx-a11y를 설치하면 코딩 단계에서 접근성 문제를 미리 잡을 수 있습니다. 자동 검사 도구가 100점을 줘도 실제 접근성이 완벽하다는 뜻은 아닙니다. 논리적 순서, 명확한 레이블, 일관된 인터랙션 같은 요소는 수동으로 확인해야 합니다. 취업 포트폴리오에서 "접근성 100점 달성"은 차별화 포인트가 됩니다.
핵심 -- Lighthouse 접근성 점수 100점을 목표로 하세요. 취업 포트폴리오에서 "접근성 100점 달성"은 차별화 포인트가 됩니다.

React 18의 createRoot, Next.js 13의 App Router 같은 대규모 변경은 코드 수정이 필요합니다. 한 번에 여러 패키지를 메이저 업데이트하면 문제 원인을 찾기 어려우므로 하나씩 단계적으로 진행합니다.

01
구조 / Structure

HTML

브라우저에게 페이지의 구조와 의미를 알려주는 마크업 언어. 모든 웹 페이지의 출발점입니다.

기본 구조
태그 구조
<태그>내용</태그>
여는 태그와 닫는 태그가 쌍을 이루며 내용을 감쌉니다. 모든 HTML의 기본 단위입니다.
속성 (Attribute)
<a href="url">링크</a>
태그에 추가 정보를 붙이는 방법. href, src, class, id, style 등이 있습니다.
중첩 구조
<div><p>텍스트</p></div>
태그 안에 태그를 넣을 수 있습니다. 부모-자식 관계가 HTML 구조의 핵심입니다.
문서 기본 구조
<!DOCTYPE> <html> <head> <body>
모든 HTML 파일의 뼈대. head는 설정, body는 실제 화면에 보이는 내용입니다.
시맨틱 태그 — 의미 있는 구조
<header> / <footer>
페이지 상단(로고, 내비게이션)과 하단(저작권, 연락처) 영역을 나타냅니다.
<nav>
내비게이션 링크 묶음. 메뉴바, 사이드바 링크 등에 사용합니다.
<main> / <section>
페이지의 주요 콘텐츠 영역. section은 주제별로 내용을 나눌 때 씁니다.
<article>
독립적으로 완결된 콘텐츠 — 블로그 글, 뉴스 기사, 카드 아이템 등.
자주 쓰는 태그들
텍스트 태그
<h1>~<h6> <p> <span> <strong> <em>
제목(h1이 가장 크고 h6이 가장 작음), 단락, 인라인 텍스트, 굵게, 기울임.
링크 & 미디어
<a> <img> <video> <iframe>
페이지 이동, 이미지 삽입, 영상 재생, 외부 콘텐츠 삽입.
목록
<ul> <ol> <li>
순서 없는 목록(ul), 번호 있는 목록(ol), 각 항목(li). 메뉴, 리스트 UI에 필수.
컨테이너
<div> <span>
div는 블록 컨테이너(줄 바꿈), span은 인라인 컨테이너(줄 유지). 레이아웃의 기본 도구.
폼 요소
<form> <input> <textarea> <button> <select>
사용자 입력을 받는 요소들. 로그인, 검색창, 신청 폼 등에 사용.
class & id
<div class="card" id="hero">
CSS와 JS에서 특정 요소를 선택하기 위한 식별자. class는 여러 요소에, id는 하나에만.
02
스타일 / Style

CSS

HTML 요소에 색상, 크기, 간격, 배치, 애니메이션을 입히는 언어. 디자인 감각과 기술이 만나는 곳입니다.

선택자 — 어떤 요소에 스타일을 줄 것인가
태그 선택자
p { color: gray; }
해당 태그 전체에 적용. p 태그, h1 태그 등 종류별로 스타일 지정.
클래스 선택자
.card { padding: 20px; }
가장 많이 쓰는 방식. 점(.)으로 시작하며 여러 요소에 동시 적용 가능.
가상 클래스
.btn:hover { background: blue; }
마우스를 올렸을 때(hover), 클릭할 때(active), 포커스 받을 때(focus) 스타일 변화.
자식 / 후손 선택자
.card > h3 { font-size: 18px; }
>는 직접 자식만, 공백은 모든 후손에게 적용. 구조 기반으로 정밀하게 선택.
박스 모델 — 모든 요소는 박스다
content
실제 내용이 들어가는 영역. width와 height로 크기를 지정합니다.
padding
padding: 20px 16px;
내용과 테두리 사이의 안쪽 여백. 버튼, 카드의 내부 공간을 만듭니다.
border
border: 1px solid #e5e7eb;
요소의 테두리. 두께, 스타일(solid/dashed), 색상을 지정합니다.
margin
margin: 0 auto;
요소 바깥의 여백. 다른 요소와의 거리를 만들고, auto로 가운데 정렬도 가능.
레이아웃 — 요소를 어떻게 배치할 것인가
Flexbox
display: flex; justify-content: space-between; align-items: center;
1차원(가로 or 세로) 배치의 표준. 내비게이션, 카드 행 배치, 버튼 정렬에 필수.
Grid
display: grid; grid-template-columns: repeat(3, 1fr); gap: 20px;
2차원(가로+세로) 배치의 표준. 카드 그리드, 갤러리, 대시보드 레이아웃에 최적.
Position
position: absolute; top: 0; right: 0;
relative(기준점), absolute(부모 기준 절대 위치), fixed(화면 고정), sticky(스크롤 따라감).
반응형 (Media Query)
@media (max-width: 768px) { .grid { grid-template-columns: 1fr; } }
화면 크기에 따라 스타일을 다르게 적용. 모바일/태블릿/데스크톱 각각 최적화.
시각 표현 — 색상, 폰트, 효과
색상
color: #3b82f6; background: rgba(59,130,246,0.1);
HEX, rgb, rgba(투명도 포함) 형식. 텍스트 색(color)과 배경색(background) 구분.
폰트 & 텍스트
font-size: 16px; font-weight: 700; line-height: 1.6; text-align: center;
크기, 굵기, 줄간격, 정렬. 가독성 좋은 텍스트 디자인의 핵심 속성들.
CSS 변수
:root { --color-primary: #3b82f6; } .btn { color: var(--color-primary); }
색상, 간격, 폰트를 변수로 관리. 한 곳만 바꾸면 사이트 전체 스타일이 바뀝니다.
트랜지션 & 애니메이션
transition: all 0.3s ease; @keyframes fadeIn { from { opacity: 0; } }
트랜지션은 상태 변화 시 부드럽게 전환. keyframes로 복잡한 움직임을 연출합니다.
border-radius & shadow
border-radius: 12px; box-shadow: 0 4px 16px rgba(0,0,0,0.08);
모서리 둥글게, 그림자 효과. 카드, 버튼, 모달의 입체감을 만드는 데 자주 씁니다.
z-index & overflow
z-index: 100; overflow: hidden;
z-index는 요소의 쌓임 순서(모달이 위에 보이게). overflow는 넘친 내용 처리 방식.
03
동작 / Behavior

JavaScript

웹 페이지에 동작과 로직을 추가하는 프로그래밍 언어. 클릭, 입력, 데이터 처리 — 모든 상호작용의 근간입니다.

데이터 다루기 — 값을 저장하고 처리한다
변수
let count = 0; const name = '홍길동';
데이터를 담아두는 상자. let은 값 변경 가능, const는 변경 불가. var는 현대 JS에서 쓰지 않습니다.
자료형
string: '텍스트' number: 42 boolean: true / false array: [1, 2, 3] object: { key: value }
데이터의 종류. 문자열, 숫자, 참/거짓, 배열(목록), 객체(묶음)가 기본 자료형입니다.
배열 (Array)
const items = ['사과', '바나나']; items.push('포도'); items.filter(i => i.length > 2); items.map(i => `<li>${i}</li>`);
목록 데이터. push(추가), filter(걸러내기), map(변환)이 실무에서 가장 많이 쓰입니다.
객체 (Object)
const user = { name: '홍길동', age: 25 }; user.name; // '홍길동'
여러 속성을 하나로 묶는 데이터 구조. API에서 받아오는 데이터 대부분이 객체 형태입니다.
로직 구성 — 조건과 반복
함수 (Function)
function greet(name) { return '안녕, ' + name; } const add = (a, b) => a + b;
반복되는 코드를 묶어 이름 붙인 단위. 화살표 함수(=>)는 현대 JS의 표준 방식입니다.

반복문
items.forEach(item => { console.log(item); }); for (let i = 0; i < 10; i++) { }
같은 작업을 여러 번 실행. forEach는 배열 순회, for는 횟수 기반 반복에 씁니다.
템플릿 리터럴
const msg = `안녕하세요, ${name}님!`; const html = `<li class="item">${text}</li>`;
백틱(`)을 쓰면 문자열 안에 변수를 쉽게 삽입. HTML 동적 생성 시 매우 자주 씁니다.
DOM & 이벤트 — 화면을 직접 조작한다
DOM 선택
const btn = document.querySelector('.btn'); const items = document.querySelectorAll('li');
JS에서 HTML 요소를 잡아오는 방법. querySelector는 CSS 선택자를 그대로 씁니다.
DOM 조작
el.innerHTML = '<b>새 내용</b>'; el.classList.add('active'); el.style.display = 'none';
잡아온 요소의 내용, 클래스, 스타일을 바꿉니다. 이것이 동적 웹 페이지의 핵심 원리입니다.
이벤트 리스너
btn.addEventListener('click', () => { modal.classList.add('show'); });
사용자 행동(클릭, 입력, 스크롤, 마우스오버)에 반응. 모든 상호작용의 시작점입니다.
fetch & async/await
const res = await fetch('https://api.example.com/data'); const data = await res.json();
서버에서 데이터를 가져오는 방법. async/await으로 순서대로 읽기 쉽게 작성합니다.
JSON
{ "name": "홍길동", "age": 25 } JSON.parse(str) / JSON.stringify(obj)
서버와 데이터를 주고받는 표준 포맷. API 응답 대부분이 JSON 형태입니다.
localStorage
localStorage.setItem('token', value); localStorage.getItem('token');
브라우저에 데이터를 저장. 새로고침해도 유지됩니다. 로그인 상태, 설정값 저장에 사용.

일반 직업도 위험하다

"내 직업은 창의적이라서 괜찮다"는 생각은 위험합니다. 단순 반복뿐 아니라 지식 기반 직업도 빠르게 변화하고 있습니다.

  • 회계 담당자의 하루치 엑셀 작업 → AI가 5분 만에 처리
  • 번역가의 하루 번역량 → AI가 1초 만에 완성
  • 디자인 초안 작업 → AI가 10초 만에 시안 10개 생성
  • 고객센터 상담원 → AI 챗봇이 24시간 무인 운영
  • 법률 문서 검토 → AI가 수백 페이지를 몇 분 만에 분석
  • 콘텐츠 작성 · 보고서 초안 → AI가 즉시 초안 생성

개발자도 예외가 아니다

"개발자는 AI로 대체 안 되겠지"라는 생각은 가장 위험한 착각입니다. 코드만 짜는 개발자는 이미 위험합니다.

  • 반복적인 CRUD 구현 → AI가 더 빠르고 정확하게 처리
  • 버그 찾기 · 수정 → AI가 코드 전체를 분석해 즉시 제안
  • 단순 API 연동 작업 → AI가 코드 한 번에 완성
  • AI 활용 개발자 1명 = 일반 개발자 3~5명의 생산성
  • 실제로 IT 기업들이 채용 공고에 "AI 활용 능력" 필수 추가 중
  • 살아남는 개발자 = AI로 더 큰 문제를 해결하는 개발자

AI를 먼저 배운 사람의 기회

두려워하기보다 먼저 배우는 것이 답입니다. AI를 이해하는 사람에게는 엄청난 레버리지가 생깁니다.

  • AI 활용자 1명이 팀 전체의 생산성을 혼자 냄
  • 코드를 전혀 몰라도 AI로 실제 서비스를 만드는 시대
  • 기획자가 직접 프로토타입을 만들어 개발팀과 대화 가능
  • 더 적은 비용으로 더 빠르게 사이드 프로젝트 완성 가능
  • AI를 모르는 경쟁자보다 항상 앞서 나갈 수 있는 구조
  • AI는 위기가 아닌, 먼저 잡으면 초강력 무기
결론: AI를 모르면 어떤 직군이든 도태,
AI를 알면 어떤 직군이든 경쟁력
지금 당장 실무에서 AI를 쓰지 않으면 뒤처지는 시대가 이미 왔습니다. 이 수업은 AI를 두려워하는 사람이 AI를 능숙하게 다루는 사람으로 바뀌는 과정입니다. 4개월 후에는 AI를 무기로 삼을 수 있습니다.

PART 01

LLM이 뭔가요? — 대형 언어 모델의 기본 개념

LLM이란?

Large Language Model, 즉 "대형 언어 모델"의 줄임말입니다. 인터넷, 책, 코드, 논문 등 수천억 개의 텍스트 데이터를 학습한 AI입니다.

  • GPT-4o (OpenAI), Claude (Anthropic), Gemini (Google)이 대표적
  • 우리가 사용하는 Claude도 LLM의 한 종류
  • 단어를 하나씩 예측하며 문장을 완성하는 방식으로 작동
어떻게 학습하나요?

LLM은 3단계 과정을 통해 만들어집니다. 이 과정을 이해하면 왜 AI가 이런 특성을 가지는지 설명할 수 있습니다.

  • 사전학습(Pre-training) — 수천억 텍스트로 언어 자체를 이해
  • 파인튜닝(Fine-tuning) — 대화·코드·지시 따르기 특화 훈련
  • RLHF — 사람이 좋은 답과 나쁜 답을 평가해 AI를 개선
토큰이란?

AI는 텍스트를 "토큰" 단위로 처리합니다. 단어 하나가 1~3개의 토큰이 됩니다. 이것이 AI 사용 비용과 직결됩니다.

  • 영어 1단어 ≈ 1.3 토큰 / 한글 1단어 ≈ 2~4 토큰
  • Claude Pro: 200K 토큰 컨텍스트 (약 15만 단어 분량)
  • 대화가 길어질수록 더 많은 토큰을 소비
컨텍스트 윈도우란?

AI가 한 번에 기억할 수 있는 텍스트의 양입니다. 이 범위를 벗어나면 AI는 앞의 내용을 잊어버립니다.

  • Claude의 강점 중 하나가 매우 긴 컨텍스트 윈도우
  • 코드 전체를 붙여넣고 수정 요청해도 정확하게 처리 가능
  • 긴 대화를 해도 초반 내용을 기억하는 이유

5
반복 생성

한 토큰씩 예측·생성을 반복해서 최종 답변이 만들어집니다. 스트리밍으로 보이는 이유입니다.

핵심은 AI가 "답을 알고 있는 게" 아니라 "가장 그럴듯한 다음 단어를 계속 예측"하는 것입니다. 학습 데이터가 방대하기 때문에 그 예측이 매우 정확하게 느껴질 뿐입니다.
PART 03

왜 프롬프트를 잘 써야 하나요? — 질문의 질이 답의 질을 결정합니다

AI의 성능은 고정되어 있습니다. 같은 AI라도 프롬프트 방식에 따라 결과 품질이 완전히 달라집니다.

나쁜 프롬프트의 패턴
  • "웹사이트 만들어줘" → 어떤 웹사이트인지 AI가 추측해야 함
  • "코드 고쳐줘" → 어디가 문제인지, 어떻게 고쳐야 하는지 불명확
  • "예쁘게 만들어줘" → "예쁜"의 기준이 사람마다 다름
  • "더 좋게 해줘" → 무엇이 좋은 것인지 AI가 판단해야 함
  • 역할 부여 없음 → AI가 적절한 수준을 모름
  • 배경 설명 없음 → 지금까지의 맥락을 AI가 모름
좋은 프롬프트의 요소
  • 역할 부여 — "너는 10년 경력 시니어 프론트엔드 개발자야"
  • 배경 설명 — "지금 카페 소개 랜딩 페이지를 만들고 있어"
  • 구체적 요청 — "헤더 + 카드 3개 + CTA 버튼이 필요해"
  • 스타일 지정 — "색상은 따뜻한 브라운 계열, 폰트는 깔끔하게"
  • 출력 형식 — "HTML/CSS 파일 분리해서 작성해줘"
  • 제약 조건 — "jQuery 쓰지 말고 순수 JS로만 해줘"
프롬프트 실력 수준
초급
막연하게 요청 "로그인 기능 만들어줘" — AI가 기술 스택, 디자인, 요구사항을 모두 추측해서 만듦
중급
조건을 붙여서 요청 "React + Tailwind로 이메일/비밀번호 로그인 폼을 만들어줘. 유효성 검사 포함해줘"
고급
맥락·역할·제약·출력형식 모두 포함 "너는 시니어 풀스택 개발자야. 지금 Next.js 프로젝트의 로그인 페이지야. 이미 만든 Button 컴포넌트를 재사용하고, 에러 상태도 처리해줘. TypeScript로 작성해줘."
PART 04

AI의 한계 — 맹목적으로 믿으면 안 됩니다

AI를 잘 활용하려면 AI가 잘하는 것과 못하는 것을 정확히 알아야 합니다. 수강생에게 반드시 전달해야 할 내용입니다.

주의 1
할루시네이션 (Hallucination)

AI가 틀린 정보를 자신있게 말하는 현상입니다. 존재하지 않는 라이브러리, 없는 함수, 잘못된 역사 사실 등을 마치 사실인 것처럼 얘기합니다.

예: "React에서 useAutoLayout() 훅을 쓰면 돼" → 실제로 없는 훅을 자신있게 알려줌
대처법: 중요한 정보는 반드시 공식 문서로 검증. 코드는 실행해서 확인.
주의 2
학습 데이터 기준 날짜

AI는 특정 시점까지의 데이터로만 학습했습니다. 그 이후에 나온 새로운 라이브러리, 프레임워크 업데이트, 최신 뉴스는 모릅니다.

예: 최근에 출시된 Next.js 새 기능을 물어보면 예전 방식으로 알려줄 수 있음
대처법: "이게 최신 방식이 맞아?"라고 한 번 더 확인. 공식 changelog 참조.
주의 3
수학·계산 오류

언어 모델이기 때문에 복잡한 수학 계산이나 정확한 숫자 연산에서 실수를 할 수 있습니다. 단순 사칙연산은 괜찮지만 복잡한 계산은 주의해야 합니다.

예: 긴 숫자 곱셈이나 통계 계산을 시키면 가끔 틀린 값을 내놓음
대처법: 중요한 계산 결과는 직접 검산하거나 계산기/코드로 검증.
주의 4
맥락 한계 (Context 초과)

대화가 매우 길어지면 AI가 앞의 내용을 잊어버리거나 일관성을 잃을 수 있습니다. 컨텍스트 윈도우를 초과하기 때문입니다.

예: 긴 코딩 세션에서 "아까 내가 말한 디자인 시스템 색상으로 해줘" → AI가 기억 못 할 수 있음
대처법: 중요한 정보는 새 대화 시작 시 다시 붙여넣기. 세션을 짧게 관리.
주의 5
개인정보·저작권

AI에 민감한 개인정보, 회사 기밀 데이터, 저작권 있는 코드를 그대로 붙여넣는 것은 주의해야 합니다.

예: 고객 데이터 파일을 AI에 업로드해서 분석 요청 → 개인정보 유출 위험
대처법: 민감 정보는 익명화 처리 후 사용. 회사 정책 확인 필수.
주의 6
AI 의존 과다

AI가 만들어준 코드를 이해하지 못한 채 그냥 쓰면 나중에 수정이나 디버깅이 불가능해집니다. AI는 도구이지 전부가 아닙니다.

예: AI가 만든 코드가 에러 날 때 "왜 에러가 나는지" 전혀 모르는 상황
대처법: AI 코드를 보면서 반드시 이해하려는 노력. 이해 안 되면 Claude에게 설명 요청.
PART 05

프롬프트를 잘 쓰는 법 — Claude에게 원하는 것을 정확히 전달하는 기술

같은 AI라도 질문 방식에 따라 결과 품질이 10배 이상 차이납니다. 프롬프트 작성법은 이 수업의 핵심 역량입니다. 멘토가 먼저 완벽히 익혀야 합니다.

PART A

좋은 프롬프트의 핵심 4원칙 — 이 4가지만 지켜도 결과가 달라집니다

역할 → 맥락 → 구체적 요청 → 반복 수정. 이 순서대로 작성하는 것이 프롬프트 기본기입니다.

01
역할 부여 (Role)

Claude에게 먼저 역할을 부여합니다. 역할에 맞는 전문 수준의 답변이 나옵니다.

예시 "너는 10년 경력 시니어 프론트엔드 개발자야"
"너는 UX 디자이너야. 사용자 친화적으로 설명해줘"
"너는 웹 개발 선생님이야. 초보자가 이해할 수 있게 설명해줘"
역할이 없으면 AI가 적절한 수준을 모릅니다. 초보자에게 고급 코드를 주거나, 전문가에게 너무 쉬운 설명을 하게 됩니다.
02
맥락 제공 (Context)

지금 무엇을 만들고 있는지, 어디까지 진행됐는지 배경을 충분히 설명합니다.

예시 "지금 카페 소개 랜딩 페이지를 만들고 있어. 헤더는 이미 만들었고 지금은 메뉴 섹션을 만들려고 해"
"이 코드는 React로 작성됐고, Tailwind CSS를 쓰고 있어. 기존 스타일을 유지해줘"
맥락이 없으면 AI가 처음부터 다시 추측해야 합니다. 많을수록 정확도가 올라갑니다.
03
구체적 요청 (Specific)

색상, 크기, 레이아웃, 포함될 요소, 기술 스택, 제약 조건을 명확하게 명시합니다.

예시 "카드 3개를 가로로 배치해줘. 각 카드에는 이미지, 제목, 설명, 버튼이 들어가야 해. 색상은 파란 계열로"
"jQuery 없이 순수 JavaScript로만 작성해줘. 외부 라이브러리 사용 금지"
막연한 요청은 막연한 결과를 낳습니다. "예쁘게 만들어줘"는 기준이 없습니다.
04
반복 수정 (Iterate)

첫 결과가 완벽하길 기대하지 않습니다. 대화하며 계속 수정해나가는 것이 올바른 방식입니다.

예시 "버튼 색을 파란색으로 바꿔줘"
"카드 간격이 너무 좁아. 더 넓게 해줘"
"모바일에서 1열로 쌓이도록 반응형 추가해줘"
대화형 AI의 진짜 강점은 반복 수정입니다. 포기하지 말고 계속 요청하세요.
PART B

나쁜 프롬프트 vs 좋은 프롬프트 — 실제 예시로 배우는 차이

같은 목적이라도 어떻게 요청하느냐에 따라 결과가 완전히 달라집니다. 수강생에게 이 예시들을 직접 보여주세요.

나쁨 "웹사이트 만들어줘" 어떤 웹사이트인지, 어떤 기술을 써야 하는지 AI가 전부 추측해야 함
좋음 "너는 시니어 프론트엔드 개발자야. 카페 소개 랜딩 페이지를 HTML/CSS로 만들어줘. 헤더에는 카페 이름과 네비, 메인에는 배경 이미지와 문구, 아래에는 메뉴 카드 3개가 있어야 해. 색상은 따뜻한 브라운 계열" 원하는 결과에 가까운 코드가 바로 나옴
나쁨 "코드 고쳐줘" 어디가 문제인지, 어떻게 고쳐야 하는지 AI가 모름
좋음 "이 코드에서 버튼 클릭 시 모달이 열려야 하는데 안 열려. 에러 메시지는 'Cannot read properties of null'. 아래 코드에서 문제를 찾아서 수정해줘: [코드 붙여넣기]" 정확한 원인 파악 후 수정 코드 제공
나쁨 "더 예쁘게 만들어줘" "예쁜"의 기준이 없어서 AI가 임의로 판단함
좋음 "카드에 hover 시 살짝 위로 올라오는 효과 추가해줘. 그림자도 더 진하게. 색상은 지금 파란색 계열 유지하면서 좀 더 밝게 해줘" 구체적인 방향이 있어서 의도에 맞는 결과 나옴
나쁨 "React로 쇼핑몰 만들어줘" 범위가 너무 넓어서 AI가 기본 구조만 만들거나 과도하게 복잡하게 만듦
좋음 "React로 상품 목록 페이지만 만들어줘. 상품 카드는 이미지, 이름, 가격, '장바구니 담기' 버튼으로 구성. 카드 클릭 시 상세 페이지로 이동. useState로 장바구니 개수 표시. Tailwind 사용" 명확한 범위와 요구사항으로 바로 사용 가능한 코드 제공

프롬프트 대화 흐름 — 처음 요청부터 완성까지 실제 과정

AI와의 대화는 한 번에 끝나지 않습니다. 이 흐름을 수강생에게 실제로 보여주면 훨씬 이해가 빠릅니다.

4개월 완주 권장

AI 활용 기획자

기획자는 많습니다. 그러나 AI로 직접 프로토타입을 만들 수 있는 기획자는 극소수입니다. 개발자 언어를 이해하고 Figma로 직접 설계하는 기획자 — 회사가 가장 원하는 인재입니다.

"AI 기획자"가 일반 기획자보다 채용 시장에서 유리한 이유:
→ 개발자에게 정확하게 전달 가능 · 빠른 실행 · AI로 업무 속도 3배

실제로 하는 일

  • 새로운 서비스·기능 기획서 작성 및 Claude로 초안 5분 만에 완성
  • Figma로 와이어프레임 → 클릭 가능한 하이파이 프로토타입 제작
  • 사용자 인터뷰 설계·진행·분석 — Claude로 인사이트 정리
  • 개발팀에 기능 요구사항 명세서 전달 및 협의
  • 경쟁사 분석 → 차별화 전략 수립
  • 서비스 지표(DAU, 전환율) 분석 후 개선안 제안

취업 가능한 포지션

서비스 기획자 프로덕트 매니저(PM) UX 기획자 IT 기획자 콘텐츠 기획자 그로스 마케터

3개월 수료 시 도전 가능

AI 활용 프론트엔드 개발자

AI 없이 개발하는 사람과 AI로 개발하는 사람의 속도 차이는 이미 3~5배입니다. 같은 시간에 더 많이 만들고, 혼자서도 완성도 높은 서비스를 만드는 개발자 — 채용 시장에서 가장 주목받는 포지션입니다.

채용 공고에 "AI 활용 능력 우대" 명시가 빠르게 늘고 있습니다.
→ 같은 실력이라면 AI 잘 쓰는 개발자가 무조건 뽑힙니다

실제로 하는 일

  • Figma 디자인 → React 컴포넌트로 구현 (Claude로 속도 3배)
  • 반응형 웹 페이지 제작 — 모바일·태블릿·데스크톱 모두 대응
  • REST API 연동으로 실시간 데이터를 화면에 표시
  • Claude로 코드 초안 생성 → 직접 리뷰·수정·최적화
  • 에러 디버깅 — Claude에게 에러 붙여넣고 원인 파악
  • Vercel로 배포 → 실제 URL로 접속 가능한 서비스 완성

취업 가능한 포지션

프론트엔드 개발자 웹 퍼블리셔 UI 개발자 풀스택 개발자(주니어) AI 활용 웹 개발자

취업 가능한 회사

  • 스타트업 — 빠른 실행이 필요한 곳, AI 개발자 수요 급증
  • 중소 IT 기업 — React 기반 웹 서비스 유지보수 및 신규 개발
  • 웹에이전시 — 월 단위로 다양한 클라이언트 사이트 제작
  • 프리랜서 — 랜딩 페이지·쇼핑몰·포트폴리오 사이트 수주
  • 인하우스 — 기업 내부 웹 서비스 개발 담당

포트폴리오 구성

  • React 앱 1~2개 — Vercel 배포, 실제 접속 가능한 URL 필수
  • 반응형 랜딩 페이지 1~2개 — 디자인 완성도 높게
  • GitHub 저장소 — 커밋 내역이 쌓여있는 것 (꾸준히 만든 증거)
  • README — 기술 스택, 구현 기능, AI 활용 방식 명시

취업에 성공하는 사람의 3가지 공통점

01

포트폴리오 퀄리티로 승부한다

출신 학교·자격증·수료 기관보다 "실제로 뭘 만들었는지"가 결정적입니다. 완성도 높은 결과물 1개가 엉성한 결과물 10개보다 낫습니다. 수업에서 만드는 모든 것을 포트폴리오 퀄리티로 끌어올리세요.

수업에서 만든 것을 그냥 두지 말고, 계속 개선해서 "내 작품"으로 만드세요.
02

수료 후에도 혼자 계속 만든다

4개월 수업은 시작점입니다. 수업이 끝난 후에도 매주 무언가를 만드는 습관이 있는 사람이 취업에 성공합니다. GitHub 커밋 내역이 꾸준히 쌓이는 것 자체가 실력의 증거입니다.

수료 후 2개월 안에 취업에 성공한 사람들의 공통점 — 수업 끝나도 매일 코딩했습니다.
03

AI 활용 방식을 설명할 수 있다

면접에서 "이 프로젝트를 Claude와 어떻게 만들었는지" 구체적으로 설명할 수 있다면 강력한 차별점이 됩니다. AI를 쓴 것이 부끄러운 게 아닙니다. AI를 잘 쓰는 것이 이제는 핵심 역량입니다.

"Claude에게 이렇게 요청해서 이 기능을 구현했습니다" — 이 한 마디가 면접관을 설득합니다.
