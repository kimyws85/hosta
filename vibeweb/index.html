기존 코딩 학원과 무엇이 다른가요?

단순히 코드를 외우는 수업이 아닙니다. AI 시대에 맞는 방식으로 배웁니다.

AI 기반 웹 제작 환경 구축

Claude + 웹디자인 기초 + 프롬프트 웹 제작

코딩을 전혀 몰라도 괜찮습니다. 이 달의 목표는 Claude라는 AI 도구를 완벽히 세팅하고, 프롬프트(말로 하는 명령)만으로 실제 웹사이트를 만들어보는 경험을 하는 것입니다. 디자인 감각의 기초도 함께 잡아갑니다.

웹디자인 기초

  • 웹의 구조 이해 — 모든 웹사이트는 헤더(상단), 본문(내용), 푸터(하단)로 이루어집니다. 좋은 웹사이트의 레이아웃이 어떻게 구성되는지 실제 사례로 분석합니다.
  • 색상 이론 기초 — 주색(brand color), 보조색(accent), 배경색, 텍스트 색의 역할을 이해합니다. 색상 대비(contrast)가 가독성에 미치는 영향, 어두운 배경 vs 밝은 배경의 특성을 배웁니다.
  • 폰트와 타이포그래피 — 제목/소제목/본문/캡션의 크기 위계(hierarchy)를 잡는 법을 익힙니다. 가독성 좋은 폰트 선택 기준, 줄 간격(line-height)이 읽힘에 미치는 영향을 이해합니다.
  • 여백(Spacing)의 중요성 — 요소들 사이의 간격이 왜 디자인 품질을 결정하는지 이해합니다. 여백이 없는 사이트 vs 여백이 적절한 사이트를 직접 비교 분석합니다.
  • 좋은 웹 vs 나쁜 웹 분석 — 실제 국내외 사이트들을 보며 "왜 이게 좋은가", "왜 이게 별로인가"를 색상·폰트·여백·구조 관점에서 설명할 수 있는 눈을 기릅니다.

Claude + VSCode 세팅

  • Claude Pro 가입 — Claude.ai 유료 플랜(Pro) 가입 방법, 플랜별 차이점, 무료 vs 유료를 왜 유료로 써야 하는지 이유를 설명합니다.
  • VSCode 설치 및 세팅 — 개발자들이 가장 많이 쓰는 에디터 VSCode를 설치하고, 기본 테마, 폰트 크기, 자동 저장 설정 등 편한 환경을 만듭니다.
  • Claude Code (CLI) 설치 — 터미널/명령줄에서 Claude를 사용할 수 있는 Claude Code를 설치합니다. 왜 웹 인터페이스보다 CLI(Command Line Interface)가 개발에 더 강력한지 이해합니다.
  • VSCode + Claude Code 연동 — VSCode 안에서 바로 Claude와 대화하면서 코드를 만들고 수정하는 실제 개발 환경을 완성합니다. 파일을 열고 바로 작업하는 워크플로우를 익힙니다.
  • 기본 사용법 실습 — 파일 열기, 폴더 구조 이해, Claude에게 질문하는 법, Claude가 만든 파일을 브라우저에서 확인하는 법까지 실습합니다.

프롬프트로 웹 만들기

  • UI 컴포넌트 단위로 만들기 — 버튼(크기·색상·hover 효과), 카드(이미지+텍스트 조합), 슬라이더/캐러셀, 아이콘 배치, 네비게이션 바 등 웹에서 자주 쓰이는 요소들을 하나씩 Claude에게 만들어달라고 요청하며 경험을 쌓습니다.
  • 프롬프트 수정 요청하기 — "버튼 색상을 파란색으로 바꿔줘", "글자 크기 키워줘", "카드 간격 더 넓게 해줘" 등 원하는 결과를 말로 요청하고 수정하는 반복 과정을 익힙니다.
  • 자기소개 페이지 완성 — 이름, 소개글, 프로필 이미지, SNS 링크, 기술 스택 등이 들어간 실제 포트폴리오용 자기소개 페이지를 Claude와 함께 완성합니다.
  • 랜딩 페이지 완성 — 히어로 섹션(메인 문구+이미지), 특징 소개(카드 3개), 후기/수치 섹션, CTA 버튼(행동 유도)이 포함된 실제 서비스 랜딩 페이지를 만듭니다.
1개월 후 결과물 자기소개 페이지 1개 + 랜딩 페이지 1개 완성. 코드를 한 줄도 직접 안 써도 됩니다.

웹 표준 기술 이해

HTML / CSS / JS + Claude

1개월차에서 프롬프트로만 만들었다면, 이번 달은 Claude가 만든 코드를 직접 읽고 이해합니다. HTML, CSS, JavaScript의 기본 구조를 파악하면 "왜 이렇게 보이는지"를 알게 되고, 원하는 부분을 직접 수정할 수 있는 능력이 생깁니다.

HTML — 웹의 뼈대

  • 태그 구조 이해 — <div>, <section>, <p>, <h1~h6>, <img>, <a>, <ul><li>, <span> 등 자주 쓰이는 태그들의 역할과 사용법을 이해합니다.
  • 시맨틱 태그 — <header>, <nav>, <main>, <section>, <article>, <aside>, <footer> 등 의미있는 태그를 쓰는 이유(검색엔진 최적화, 접근성)를 이해하고 적용합니다.
  • HTML 파일 구조 — <!DOCTYPE html>, <head>, <body>의 역할, CSS/JS 파일을 연결하는 방법, meta 태그의 역할을 이해합니다.

CSS — 웹의 스타일

  • 선택자·클래스·id 개념 — CSS가 어떻게 특정 요소를 찾아서 스타일을 적용하는지 이해합니다. 클래스(.class)와 id(#id)의 차이와 언제 무엇을 쓰는지 배웁니다.
  • CSS 변수(Custom Properties) — :root에 색상, 폰트, 간격을 변수로 정의하고 전체 스타일을 한 곳에서 관리하는 실무 방식을 익힙니다.
  • Flexbox 레이아웃 — 요소를 가로/세로로 정렬하는 Flexbox의 핵심 속성들 (display:flex, justify-content, align-items, gap 등)을 실습합니다.
  • Grid 레이아웃 — 카드 목록, 2단 레이아웃 등을 Grid로 구현하는 법을 배웁니다. Flexbox와 Grid를 언제 각각 쓰는지 판단하는 기준도 익힙니다.
  • 반응형 디자인 (미디어 쿼리) — 화면 크기에 따라 레이아웃이 바뀌는 반응형 웹을 만드는 법을 배웁니다. 모바일(480px), 태블릿(768px), 데스크톱 기준으로 스타일을 분기하는 법을 익힙니다.
  • transition · hover 효과 — 버튼을 올렸을 때 색이 바뀌고, 카드를 올렸을 때 살짝 올라오는 등의 부드러운 인터랙션 효과를 CSS만으로 구현하는 법을 익힙니다.

JavaScript — 웹의 동작

  • 변수·함수 기초 — let, const, 함수 선언, 화살표 함수 등 JS의 기본 문법을 이해합니다. 코드를 외우는 것이 목표가 아니라, Claude가 만든 코드를 읽고 이해하는 것이 목표입니다.
  • DOM 조작 기초 — document.querySelector로 요소를 찾고, classList.add/remove로 클래스를 추가/제거하는 법을 이해합니다. 이것만 알면 대부분의 인터랙션을 구현할 수 있습니다.
  • 이벤트 리스너 — 클릭, 스크롤, 마우스 이동 등의 이벤트에 반응하는 addEventListener의 기본 패턴을 익힙니다.
  • 실습: 모달 팝업 — 버튼 클릭 시 팝업창이 뜨고, X 클릭 시 닫히는 모달을 직접 구현합니다.
  • 실습: 모바일 햄버거 메뉴 — 모바일에서 햄버거 메뉴 클릭 시 메뉴가 펼쳐지는 토글 네비게이션을 구현합니다.
  • 실습: 스크롤 애니메이션 — 스크롤 시 요소가 부드럽게 나타나는 IntersectionObserver 패턴을 익힙니다.

Claude와 함께하는 방식

  • Claude 코드 읽고 이해하기 — Claude가 만든 코드를 보면서 "이 부분이 왜 이렇게 된 건지"를 Claude에게 물어보고 설명을 들으며 코드 이해력을 키웁니다.
  • 직접 수정 + Claude 요청 병행 — 간단한 수정(색상 변경, 텍스트 수정)은 직접 하고, 복잡한 기능은 Claude에게 요청하는 방식으로 실력을 쌓습니다.
  • 1개월차 결과물 코드로 재구현 — 1개월차에 프롬프트만으로 만든 자기소개 페이지를 이번에는 HTML/CSS 구조를 직접 이해하며 다시 만들어봅니다.
2개월 후 결과물 반응형 웹사이트 1개 완성. HTML/CSS/JS 구조를 읽고 이해할 수 있는 수준. 간단한 부분은 직접 수정 가능.

컴포넌트 기반 개발

React + Claude

React는 현재 프론트엔드 취업 시장의 표준입니다. 복잡한 웹 앱을 "컴포넌트"라는 단위로 나눠서 만드는 방식을 배웁니다. Claude와 함께라면 React의 진입장벽을 낮추고 실무 수준의 앱을 빠르게 완성할 수 있습니다.

React 핵심 개념

  • 왜 React인가 — HTML로만 만들면 같은 코드를 여러 곳에 반복 작성해야 합니다. React는 버튼, 카드, 네비게이션 같은 요소를 "컴포넌트"로 한 번만 만들고 어디서든 재사용할 수 있게 해줍니다. 현재 국내외 대부분의 웹 서비스가 React로 만들어집니다.
  • 컴포넌트 개념 이해 — Button.jsx, Card.jsx, Navbar.jsx처럼 UI를 독립적인 조각으로 나누는 법, 컴포넌트를 조합해서 페이지를 만드는 방식을 이해합니다.
  • JSX 문법 — HTML처럼 생겼지만 JavaScript인 JSX 문법을 이해합니다. className, onClick, {} 안에 JS 표현식 넣기 등 HTML과 다른 점을 익힙니다.
  • Props — 컴포넌트에 데이터 전달 — 같은 Card 컴포넌트에 제목, 내용, 이미지를 다르게 넣어서 여러 카드를 만드는 props 개념을 이해하고 실습합니다.
  • State — 변하는 데이터 관리 — 버튼 클릭 횟수, 모달 열림/닫힘, 탭 선택 상태 등 화면에서 변하는 값을 useState로 관리하는 법을 익힙니다.
  • 파일 구조 이해 — src/, components/, pages/, public/ 폴더의 역할을 이해하고 실무에서 파일을 어떻게 구조화하는지 패턴을 익힙니다.

Claude와 React 개발

  • 컴포넌트 생성 요청 — "ProductCard 컴포넌트를 만들어줘. props로 image, title, price, description을 받아야 해" 처럼 구체적으로 요청해서 바로 쓸 수 있는 컴포넌트를 만드는 법을 익힙니다.
  • 기능 구현 요청 — "탭을 클릭하면 해당 탭의 내용만 보여주는 기능 만들어줘", "검색어를 입력하면 목록이 필터링되는 기능 만들어줘" 등의 실무 기능을 Claude와 함께 구현합니다.
  • 에러 해결 — React 개발 중 만나는 에러 메시지를 Claude에게 붙여넣고 원인과 해결법을 이해하는 방식으로 디버깅 능력을 키웁니다.

최종 프로젝트: 실무형 앱 완성

  • 프로젝트 선택 — 쇼핑몰 상품 목록 페이지 / 포트폴리오 사이트 / 레스토랑 메뉴 앱 중 하나를 선택합니다.
  • 컴포넌트 설계 — 만들 앱을 어떤 컴포넌트로 나눌지 설계하고, Claude와 함께 하나씩 만들어갑니다.
  • 기능 구현 — 필터링, 검색, 탭 전환, 모달 등 실제 서비스에서 볼 수 있는 기능들을 구현합니다.
  • 배포 — Vercel을 통해 실제 인터넷에서 접속 가능한 URL로 배포하는 방법을 배웁니다. 이제 포트폴리오에 링크를 넣을 수 있습니다.

서비스 설계 역량 강화

UI/UX + 기획 + Figma

개발만 할 줄 아는 사람과 기획부터 설계까지 할 수 있는 사람의 차이는 큽니다. 4개월차에는 Figma로 와이어프레임·프로토타입을 만들고, 서비스 기획서 작성법을 익히며 "AI 활용 기획자" 방향으로 커리어를 확장합니다.

UX 리서치 & 분석

  • 사용자 인터뷰 방법 — 사용자가 서비스를 어떻게 쓰는지 파악하기 위한 인터뷰 질문을 설계하고 진행하는 방법을 익힙니다. Claude로 인터뷰 스크립트를 작성하는 법도 배웁니다.
  • 페르소나 작성 — 서비스의 주요 사용자를 구체적으로 묘사한 페르소나 카드를 작성합니다. "28세 직장인 김민지, 퇴근 후 자기계발에 관심 있음" 같은 구체적 인물을 설정하고 그 사람의 니즈에 맞게 서비스를 설계하는 방법을 배웁니다.

웹페이지는 하나의 화면이고, 웹사이트는 여러 웹페이지가 모여 있는 전체 구조입니다. 예를 들어 메인 페이지, 소개 페이지, 문의 페이지가 있으면 각각은 웹페이지이고, 이 전체를 묶어서 웹사이트라고 부릅니다.
웹페이지 = 하나의 화면 웹사이트 = 여러 페이지의 집합 예시: 홈 소개 서비스 안내 문의하기
웹앱
서비스 동작 인터랙션
단순히 정보를 보여주는 수준을 넘어 로그인, 검색, 작성, 저장처럼 사용자의 행동이 들어가면 웹앱 성격이 강해집니다. 오늘날 많은 서비스는 웹사이트이면서 동시에 웹앱입니다.
정보 중심: 회사 소개 사이트, 포트폴리오 기능 중심: 쇼핑몰, SNS, 예약 서비스, 관리자 페이지

사용자 요청 -> 서버 응답 -> 브라우저 표시 예시: 브라우저가 "메인 페이지 주세요" 서버가 "HTML, CSS, 이미지 드릴게요"

DOM 요소 조작
브라우저는 HTML을 읽은 뒤 요소들을 JavaScript가 다룰 수 있는 형태로 바꿉니다. 이 구조를 DOM이라고 합니다.
제목, 버튼, 이미지, 박스 같은 HTML 요소를 JavaScript가 선택해서 바꿀 수 있게 된 상태
기본 조작
선택 클래스 토글
JavaScript는 특정 요소를 선택하고, 클래스를 추가하거나 제거하면서 화면 상태를 바꿉니다. 드롭다운, 탭, 모달, 아코디언 같은 UI의 기본 원리입니다.
const btn = document.querySelector('.menu-btn'); const nav = document.querySelector('.nav'); btn.addEventListener('click', () => { nav.classList.toggle('open'); });
이벤트
click hover input
사용자의 행동이 발생했을 때 실행되는 것을 이벤트라고 합니다. 클릭, 입력, 스크롤, 마우스 움직임 같은 것이 모두 이벤트입니다.
click = 클릭했을 때 input = 글자를 입력할 때 scroll = 화면을 내릴 때
01
구조 / Structure

HTML

HTML은 웹페이지의 구조를 만드는 언어입니다. 브라우저에게 이 부분은 제목인지, 문단인지, 링크인지, 이미지인지, 입력창인지 알려주며 화면이 어떤 순서와 어떤 의미로 구성되어야 하는지를 정리하는 가장 기본적인 출발점입니다. 웹페이지를 만든다고 할 때 가장 먼저 잡히는 것은 디자인이 아니라 구조이고, 그 구조를 적는 언어가 HTML입니다.

HTML은 웹에서 어떤 역할을 하는가
웹페이지의 뼈대를 만드는 언어
HTML = 구조 / CSS = 모양 / JavaScript = 동작
HTML은 화면을 예쁘게 꾸미는 언어도 아니고, 복잡한 기능을 직접 처리하는 언어도 아닙니다. HTML이 하는 가장 중요한 일은 웹페이지에 무엇이 들어가는지를 정하는 것입니다. 예를 들어 페이지 상단에 로고가 있고, 그 아래 큰 제목이 있고, 설명 문장이 있고, 버튼이 있고, 그 밑에 이미지와 카드 목록이 있다면 이런 배치의 기본 틀을 먼저 만드는 것이 HTML입니다. 즉 HTML은 화면의 재료를 놓는 단계이고, CSS는 그 재료를 보기 좋게 다듬고, JavaScript는 그 재료가 움직이거나 바뀌게 만듭니다. 그래서 HTML이 없으면 CSS도 꾸밀 대상이 없고, JavaScript도 조작할 대상이 없습니다.
브라우저에게 요소의 의미를 알려주는 언어
<h1>제목</h1> <p>문단</p> <a>링크</a>
HTML은 단순히 글자를 화면에 찍는 것이 아니라, 이 글자가 어떤 역할을 하는지 브라우저에게 알려줍니다. 같은 문장이라도 제목이면 제목 태그를 써야 하고, 설명 문단이면 문단 태그를 써야 하며, 이동 기능이 있으면 링크 태그를 써야 합니다. 이것이 중요한 이유는 브라우저가 구조를 이해해야 화면을 올바르게 보여줄 수 있고, 검색 엔진이나 접근성 도구도 이 의미를 기준으로 페이지를 해석하기 때문입니다. 즉 HTML은 눈에 보이는 배치만 만드는 것이 아니라, 화면 안 요소들의 역할을 정의하는 언어입니다.
웹 제작의 가장 첫 단계
상단 - 본문 - 하단 / 제목 - 설명 - 버튼 / 목록 - 카드 - 입력창
실제로 웹페이지를 만들 때 가장 먼저 해야 하는 일은 구조를 나누는 것입니다. 상단에는 무엇이 들어가고, 본문에는 어떤 정보가 있고, 하단에는 어떤 마무리 정보가 들어가는지 먼저 정리해야 합니다. 이때 HTML은 설계도를 실제 코드로 옮기는 역할을 합니다. 그래서 HTML을 배운다는 것은 태그 이름만 외우는 것이 아니라, 화면을 어떤 덩어리로 나누고 어떤 순서로 배치할지를 이해하는 일에 가깝습니다.
HTML의 기본 문법과 구조
태그
<태그>내용</태그>
HTML의 가장 기본 단위는 태그입니다. 대부분의 태그는 여는 태그와 닫는 태그가 한 쌍으로 이루어지고, 그 사이에 내용이 들어갑니다. 제목, 문단, 버튼, 링크, 목록 같은 모든 구조가 태그로 만들어집니다. 결국 HTML 문서는 태그들이 모여 만들어진 문서이고, 브라우저는 이 태그들을 위에서 아래로 읽으면서 화면 구조를 구성합니다.
속성
<a href="https://example.com">링크</a>
태그만으로는 부족할 때 속성을 사용합니다. 속성은 태그에 추가 정보를 붙이는 방법입니다. 링크는 어디로 이동해야 하는지 주소가 필요하고, 이미지는 어떤 파일을 보여줄지 경로가 필요하며, CSS를 적용하거나 JavaScript로 선택하기 위해 class나 id 같은 이름표가 필요합니다. 즉 속성은 태그의 역할을 더 구체적으로 만드는 부가 정보입니다.
중첩 구조
<section><div><p>내용</p></div></section>
HTML은 태그 안에 다른 태그가 들어가는 중첩 구조를 가집니다. 그래서 부모와 자식이라는 관계가 생깁니다. 예를 들어 하나의 section 안에 여러 div가 들어가고, 그 div 안에 제목과 문단이 들어갈 수 있습니다. 이 구조가 중요한 이유는 실제 웹페이지가 한 줄짜리 정보가 아니라 여러 덩어리와 하위 요소로 이루어지기 때문입니다. CSS에서 특정 부분만 꾸미거나 JavaScript에서 특정 요소만 조작하는 것도 결국 이 구조를 기준으로 이루어집니다.
HTML 문서의 기본 틀
<!DOCTYPE html> <html> <head></head> <body></body> </html>
모든 HTML 파일은 기본 문서 구조를 가집니다. `html` 태그는 문서 전체를 감싸고, `head`에는 문서 제목, 문자 인코딩, CSS 연결 같은 설정 정보가 들어갑니다. `body`에는 실제로 브라우저 화면에 보이는 내용이 들어갑니다. 즉 사용자가 눈으로 보는 것은 대부분 body 안쪽이고, head는 화면 뒤에서 문서를 준비하는 영역이라고 보면 됩니다.
자주 쓰는 HTML 태그와 그 의미
제목과 문단 태그
<h1>~<h6> <p> <span>
제목 태그는 내용의 위계를 보여주는 태그입니다. h1이 가장 중요한 큰 제목이고, 숫자가 내려갈수록 하위 제목이 됩니다. p 태그는 문단을 나타내며, 여러 줄의 설명이나 본문을 담을 때 사용합니다. span은 문장 안의 짧은 일부를 묶을 때 자주 사용합니다. 즉 제목, 본문, 문장 일부를 어떤 구조로 구분할지 정하는 데 쓰이는 가장 기본적인 텍스트 태그들입니다.
링크와 이미지 태그
<a> <img>
a 태그는 페이지 이동을 담당합니다. 메뉴를 눌러 다른 페이지로 가거나, 자세히 보기 버튼을 눌러 상세 페이지로 이동하는 구조가 모두 a 태그에서 시작됩니다. img 태그는 이미지를 넣는 태그입니다. 웹페이지는 텍스트만으로 이루어지지 않기 때문에 이미지, 배너, 썸네일, 프로필 사진 같은 시각 요소가 자주 들어갑니다. 이 둘은 웹페이지를 정보와 이동 구조로 연결하는 매우 기본적인 태그입니다.
목록 태그
<ul> <ol> <li>
웹에서는 같은 성격의 항목이 여러 개 반복되는 경우가 많습니다. 메뉴 목록, 공지사항 목록, 카드형 리스트, 순서 안내 등은 모두 목록 구조와 관련이 있습니다. ul은 순서 없는 목록, ol은 순서 있는 목록, li는 각 항목 하나를 의미합니다. 겉으로 보기에는 단순 나열처럼 보여도, 구조적으로는 매우 자주 등장하는 핵심 태그입니다.
영역을 나누는 태그
<div> <section> <article>
웹페이지는 하나의 덩어리가 아니라 여러 구역으로 나뉩니다. 소개 영역, 서비스 소개 영역, 후기 영역, 문의 영역처럼 주제별로 쪼개야 읽기 쉬운 화면이 됩니다. div는 가장 범용적인 묶음 태그이고, section은 하나의 주제를 가진 구역을 나눌 때 자주 사용합니다. article은 독립적인 하나의 콘텐츠 단위에 어울립니다. 즉 이런 태그들은 레이아웃의 큰 구조를 만드는 데 중요한 역할을 합니다.
폼 태그
<form> <input> <textarea> <button> <select>
사용자가 무언가를 입력하거나 선택해야 하는 화면에는 폼 관련 태그가 필요합니다. 로그인, 회원가입, 검색창, 문의하기, 신청서, 예약 입력창 같은 구조는 모두 이 태그들로 만들어집니다. 즉 HTML은 단순히 읽는 화면만 만드는 것이 아니라, 사용자가 입력하고 제출하는 인터페이스의 기본 구조도 함께 담당합니다.
class와 id
<div class="card" id="intro">
class와 id는 태그에 붙이는 이름표입니다. CSS는 이 이름을 보고 어떤 요소를 꾸밀지 찾고, JavaScript도 이 이름을 보고 어떤 요소를 조작할지 찾습니다. class는 여러 요소에 반복해서 붙일 수 있고, id는 한 페이지에서 하나의 고유한 요소를 가리킬 때 많이 씁니다. 즉 이 둘은 HTML 구조를 CSS와 JavaScript에 연결해주는 중요한 기준입니다.
시맨틱 태그 — 의미를 가진 구조
<header>, <nav>, <main>, <footer>
HTML에는 단순히 묶기 위한 태그만 있는 것이 아니라, 이 구역이 페이지의 어디쯤에 어떤 역할을 하는지 드러내는 태그들도 있습니다. header는 상단 영역, nav는 메뉴 이동 영역, main은 핵심 본문, footer는 하단 정보 영역을 의미합니다. 이런 태그를 쓰면 구조가 더 명확해지고, 브라우저나 검색 엔진, 접근성 도구도 페이지를 더 잘 해석할 수 있습니다.
<section>와 <article>의 차이
section은 하나의 주제를 가진 구역을 나눌 때 사용하고, article은 그 자체로 독립성이 있는 콘텐츠에 어울립니다. 예를 들어 “서비스 소개”, “후기”, “문의하기”는 section이 잘 어울리고, 블로그 글 하나, 뉴스 기사 하나, 카드형 게시글 하나는 article이 잘 어울립니다. 둘 다 묶는 태그이지만, 쓰는 맥락이 조금 다릅니다.
왜 의미 있는 태그를 쓰는가
처음 배울 때는 div만으로도 화면을 만들 수 있습니다. 하지만 의미 있는 태그를 쓰면 구조를 읽기 쉬워지고, 협업할 때도 훨씬 명확합니다. 또한 접근성 측면에서도 도움이 됩니다. 즉 시맨틱 태그는 단지 보기 좋은 문법이 아니라, 구조를 더 분명하게 표현하고 해석 가능하게 만드는 방법입니다.
HTML을 배울 때 핵심적으로 익혀야 하는 관점
태그를 외우는 것보다 구조를 이해하는 것이 더 중요하다
상단 / 본문 / 하단 → 제목 / 설명 / 버튼 → 카드 / 목록 / 입력창
HTML을 공부할 때 많은 사람이 태그 이름을 외우는 것에 집중합니다. 하지만 더 중요한 것은 이 화면이 어떤 덩어리로 이루어져 있는지 파악하는 능력입니다. 즉 “이 부분은 제목”, “이 부분은 설명”, “이 부분은 이동 버튼”, “이 부분은 목록”처럼 화면을 구조로 해석할 수 있어야 HTML이 보이기 시작합니다.
예쁘게 만드는 것보다 먼저 틀을 세운다
HTML 먼저 → CSS 나중 → JavaScript 마지막
웹 제작은 보통 구조를 먼저 만들고, 그다음 꾸미고, 마지막에 동작을 넣는 흐름으로 진행됩니다. 그래서 HTML은 가장 첫 번째 작업입니다. 아무리 디자인 감각이 좋아도 구조가 엉키면 CSS도 복잡해지고 JavaScript도 다루기 어려워집니다. 결국 HTML을 잘 짠다는 것은 이후 작업 전체를 편하게 만드는 기반을 잘 깔아둔다는 뜻입니다.
HTML은 화면을 코드로 해석하는 시작점이다
보이는 화면을 요소 단위로 나누어 생각하기
완성된 웹사이트를 볼 때 그냥 예쁜 화면으로만 보면 HTML이 잘 보이지 않습니다. 그러나 그 화면을 로고, 메뉴, 제목, 설명, 버튼, 카드, 이미지, 입력창처럼 요소 단위로 나누어 보기 시작하면 그때부터 HTML 구조가 머릿속에 잡히기 시작합니다. 그래서 HTML을 배운다는 것은 단순 코딩이 아니라, 화면을 구조로 읽는 눈을 만드는 과정이라고 볼 수 있습니다.
02
스타일 / Style

CSS

HTML 요소에 색상, 크기, 간격, 배치, 애니메이션을 입히는 언어. 디자인 감각과 기술이 만나는 곳입니다.

선택자 — 어떤 요소에 스타일을 줄 것인가
태그 선택자
p { color: gray; }
해당 태그 전체에 적용. p 태그, h1 태그 등 종류별로 스타일 지정.
클래스 선택자
.card { padding: 20px; }
가장 많이 쓰는 방식. 점(.)으로 시작하며 여러 요소에 동시 적용 가능.
가상 클래스
.btn:hover { background: blue; }
마우스를 올렸을 때(hover), 클릭할 때(active), 포커스 받을 때(focus) 스타일 변화.
자식 / 후손 선택자
.card > h3 { font-size: 18px; }
>는 직접 자식만, 공백은 모든 후손에게 적용. 구조 기반으로 정밀하게 선택.
박스 모델 — 모든 요소는 박스다
content
실제 내용이 들어가는 영역. width와 height로 크기를 지정합니다.
padding
padding: 20px 16px;
내용과 테두리 사이의 안쪽 여백. 버튼, 카드의 내부 공간을 만듭니다.
border
border: 1px solid #e5e7eb;
요소의 테두리. 두께, 스타일(solid/dashed), 색상을 지정합니다.
margin
margin: 0 auto;
요소 바깥의 여백. 다른 요소와의 거리를 만들고, auto로 가운데 정렬도 가능.
레이아웃 — 요소를 어떻게 배치할 것인가
Flexbox
display: flex; justify-content: space-between; align-items: center;
1차원(가로 or 세로) 배치의 표준. 내비게이션, 카드 행 배치, 버튼 정렬에 필수.
Grid
display: grid; grid-template-columns: repeat(3, 1fr); gap: 20px;
2차원(가로+세로) 배치의 표준. 카드 그리드, 갤러리, 대시보드 레이아웃에 최적.
Position
position: absolute; top: 0; right: 0;
relative(기준점), absolute(부모 기준 절대 위치), fixed(화면 고정), sticky(스크롤 따라감).
반응형 (Media Query)
@media (max-width: 768px) { .grid { grid-template-columns: 1fr; } }
화면 크기에 따라 스타일을 다르게 적용. 모바일/태블릿/데스크톱 각각 최적화.
시각 표현 — 색상, 폰트, 효과
색상
color: #3b82f6; background: rgba(59,130,246,0.1);
HEX, rgb, rgba(투명도 포함) 형식. 텍스트 색(color)과 배경색(background) 구분.
폰트 & 텍스트
font-size: 16px; font-weight: 700; line-height: 1.6; text-align: center;
크기, 굵기, 줄간격, 정렬. 가독성 좋은 텍스트 디자인의 핵심 속성들.
CSS 변수
:root { --color-primary: #3b82f6; } .btn { color: var(--color-primary); }
색상, 간격, 폰트를 변수로 관리. 한 곳만 바꾸면 사이트 전체 스타일이 바뀝니다.
트랜지션 & 애니메이션
transition: all 0.3s ease; @keyframes fadeIn { from { opacity: 0; } }
트랜지션은 상태 변화 시 부드럽게 전환. keyframes로 복잡한 움직임을 연출합니다.
border-radius & shadow
border-radius: 12px; box-shadow: 0 4px 16px rgba(0,0,0,0.08);
모서리 둥글게, 그림자 효과. 카드, 버튼, 모달의 입체감을 만드는 데 자주 씁니다.
z-index & overflow
z-index: 100; overflow: hidden;
z-index는 요소의 쌓임 순서(모달이 위에 보이게). overflow는 넘친 내용 처리 방식.

동작 / Behavior

JavaScript

JavaScript는 웹페이지에 동작과 로직을 넣는 언어입니다. HTML이 구조를 만들고 CSS가 화면을 꾸민다면, JavaScript는 클릭, 입력, 화면 변화, 데이터 처리처럼 사용자의 행동에 반응하는 실제 기능을 담당합니다.

데이터 다루기 — 값을 저장하고 처리한다
변수
let count = 0; const name = '홍길동';
변수는 값을 저장해두는 공간입니다. 숫자, 문자, 상태값처럼 이후에 다시 써야 하는 데이터를 담아두고 필요할 때 꺼내 사용합니다. let은 값이 바뀔 수 있는 경우에, const는 바뀌지 않는 값을 둘 때 주로 사용합니다.
자료형
string: '텍스트' number: 42 boolean: true / false array: [1, 2, 3] object: { key: value }
자료형은 데이터의 종류를 뜻합니다. 문자열, 숫자, 참과 거짓, 목록 형태의 배열, 여러 값을 묶는 객체가 가장 기본이며, JavaScript는 이 자료형을 바탕으로 계산, 비교, 저장, 출력 작업을 처리합니다.
배열 (Array)
const items = ['사과', '바나나']; items.push('포도'); items.filter(i => i.length > 2); items.map(i => `<li>${i}</li>`);
배열은 여러 데이터를 순서대로 담는 구조입니다. 목록을 보여주거나 반복해서 처리할 때 자주 쓰이며, 추가, 필터링, 변환 같은 작업과 함께 많이 사용됩니다.
객체 (Object)
const user = { name: '홍길동', age: 25 }; user.name; // '홍길동'
객체는 서로 관련 있는 값을 하나로 묶는 구조입니다. 예를 들어 사용자 정보, 상품 정보, 게시글 정보처럼 하나의 대상에 여러 속성이 있을 때 사용합니다. 실제 서비스 데이터는 객체 형태로 다루는 경우가 많습니다.
로직 구성 — 조건과 반복
함수 (Function)
function greet(name) { return '안녕, ' + name; } const add = (a, b) => a + b;
함수는 여러 번 사용할 코드를 하나로 묶은 단위입니다. 같은 작업을 반복해서 작성하지 않게 해주며, 입력값을 받아 다른 결과를 만들어내는 구조의 기본이 됩니다.
조건문
if (isLoggedIn) { showProfile(); } else { showLogin(); }
조건문은 상황에 따라 다른 코드를 실행할 때 사용합니다. 로그인 여부, 값의 유효성, 특정 상태인지 아닌지를 나누는 데 쓰이며, 화면 분기와 기능 분기의 가장 기본적인 방식입니다.
반복문
items.forEach(item => { console.log(item); }); for (let i = 0; i < 10; i++) { }
반복문은 같은 작업을 여러 번 수행할 때 사용합니다. 목록 데이터를 하나씩 처리하거나, 일정 횟수만큼 같은 로직을 돌릴 때 필요합니다. 배열과 함께 사용하는 경우가 특히 많습니다.
템플릿 리터럴
const msg = `안녕하세요, ${name}님!`; const html = `<li class="item">${text}</li>`;
템플릿 리터럴은 문자열 안에 변수를 자연스럽게 넣을 수 있는 문법입니다. 문자 조합이 쉬워지고, HTML 조각을 동적으로 만들 때도 자주 활용됩니다.
DOM & 이벤트 — 화면을 직접 조작한다
DOM 선택
const btn = document.querySelector('.btn'); const items = document.querySelectorAll('li');
JavaScript가 화면을 바꾸려면 먼저 HTML 요소를 찾아야 합니다. querySelector는 CSS 선택자 방식으로 요소를 찾기 때문에 가장 자주 쓰이는 선택 방법입니다.
DOM 조작
el.innerHTML = '<b>새 내용</b>'; el.classList.add('active'); el.style.display = 'none';
선택한 요소의 내용, 클래스, 스타일을 바꾸는 작업입니다. 글자를 바꾸거나 보이게 하고 숨기거나, 특정 상태를 추가하는 식으로 화면을 직접 변화시킵니다.
이벤트 리스너
btn.addEventListener('click', () => { modal.classList.add('show'); });
이벤트 리스너는 사용자의 행동에 반응하는 코드입니다. 클릭, 입력, 스크롤, 마우스 움직임 같은 행동을 감지해서 특정 기능이 실행되도록 연결합니다.
fetch & async/await
const res = await fetch('https://api.example.com/data'); const data = await res.json();
fetch는 서버에서 데이터를 가져올 때 사용하는 방법입니다. async/await을 함께 쓰면 비동기 코드를 순서대로 읽는 것처럼 작성할 수 있어 구조가 더 명확해집니다.
JSON
{ "name": "홍길동", "age": 25 } JSON.parse(str) / JSON.stringify(obj)
JSON은 데이터를 주고받을 때 많이 쓰는 형식입니다. 서버 응답이나 저장 데이터에서 자주 보이며, JavaScript 객체와 비슷한 형태라 다루기 쉽다는 특징이 있습니다.
localStorage
localStorage.setItem('token', value); localStorage.getItem('token');
localStorage는 브라우저 안에 데이터를 저장하는 공간입니다. 새로고침을 해도 값이 유지되기 때문에 간단한 설정값이나 상태 정보를 보관할 때 자주 사용됩니다.

React

현업 프론트엔드의 표준. UI를 컴포넌트 단위로 쪼개어 복잡한 화면을 효율적으로 만들고 관리합니다.

React의 핵심 개념
컴포넌트 (Component)
function Button({ label }) { return <button>{label}</button>; }
UI를 작은 단위로 나눈 것. 한 번 만들면 여러 곳에 재사용. 레고 블록처럼 조립합니다.
JSX
return ( <div className="card"> <h3>{title}</h3> </div> );
JS 안에서 HTML처럼 쓰는 문법. class 대신 className, for 대신 htmlFor를 씁니다.
props
<Card title="제목" color="blue" /> function Card({ title, color }) { ... }
부모 컴포넌트가 자식에게 데이터를 넘겨주는 방법. 읽기 전용이며 자식이 직접 바꿀 수 없습니다.
state (useState)
const [count, setCount] = useState(0); <button onClick={() => setCount(count + 1)}> {count} </button>
컴포넌트 내부 데이터. 값이 바뀌면 화면이 자동으로 다시 렌더링됩니다. React의 핵심 원리.
useEffect
useEffect(() => { fetch('/api/data').then(...); }, []); // [] = 처음 한 번만 실행
컴포넌트가 화면에 나타난 후 실행. API 데이터 로딩, 타이머 설정, 외부 라이브러리 초기화에 씁니다.
조건부 & 리스트 렌더링
{isLoggedIn && <Profile />} {items.map(item => ( <Card key={item.id} title={item.name} /> ))}
조건에 따라 컴포넌트를 보이거나 숨깁니다. 배열 데이터를 map으로 반복 렌더링하는 것이 기본 패턴.
프로젝트 구조 & 도구
Vite로 프로젝트 시작
npm create vite@latest my-app cd my-app npm install npm run dev
React 프로젝트를 만드는 현대적인 방법. 명령어 4줄로 바로 개발 시작.
파일 구조
src/ components/ -- 재사용 컴포넌트 pages/ -- 페이지 단위 App.jsx -- 루트 컴포넌트
규칙을 지키면 팀 협업과 코드 유지보수가 쉬워집니다. 현업 구조를 그대로 익힙니다.
React Router
<Route path="/" element={<Home />} /> <Route path="/about" element={<About />} />
페이지 새로고침 없이 URL을 바꾸는 SPA(단일 페이지 앱) 라우팅. 실무 필수 라이브러리.
이벤트 핸들링
<button onClick={handleClick}> <input onChange={e => setValue(e.target.value)}>
JSX에서 이벤트를 camelCase로 작성. onChange로 입력값을 state에 실시간 반영하는 것이 기본 패턴.

서버 / Backend

백엔드

백엔드는 화면 뒤에서 데이터를 처리하고 저장하는 영역입니다. 사용자가 회원가입을 하거나 로그인하고, 글을 작성하고, 상품 정보를 불러오고, 주문을 저장하는 기능은 대부분 백엔드에서 처리됩니다. 프론트엔드가 사용자에게 보이는 부분을 담당한다면, 백엔드는 그 화면이 실제로 작동하도록 만드는 내부 엔진에 가깝습니다.

백엔드는 무엇을 하는가
요청을 받고 처리한다
사용자가 로그인 버튼을 누르거나 게시글 목록을 열면 프론트엔드는 서버에 요청을 보냅니다. 백엔드는 그 요청을 받아 필요한 작업을 처리한 뒤 결과를 다시 응답합니다. 즉 백엔드는 화면에서 일어난 행동을 실제 데이터 처리로 연결하는 역할을 합니다.
사용자 행동 -> 프론트엔드 요청 -> 백엔드 처리 -> 결과 응답
데이터를 저장하고 꺼낸다
회원 정보, 게시글, 댓글, 주문 내역, 상품 정보처럼 서비스에서 유지해야 하는 데이터는 백엔드가 관리합니다. 화면에 보이는 내용도 대부분 서버나 데이터베이스에 저장된 값을 불러와서 출력하는 구조입니다.
회원 정보 저장 게시글 목록 조회 상품 데이터 불러오기 주문 내역 기록
보안과 권한을 관리한다
로그인 상태를 확인하고, 누가 어떤 데이터를 볼 수 있는지 판단하는 일도 백엔드의 중요한 역할입니다. 단순히 데이터를 주는 것이 아니라, 이 요청을 처리해도 되는지 먼저 검토해야 하기 때문입니다.
로그인 확인 관리자 권한 검사 비밀번호 보호 민감한 데이터 제한
백엔드의 기본 구성
서버
서버는 요청을 받아 응답을 돌려주는 프로그램 또는 컴퓨터입니다. 사용자가 페이지를 열거나 데이터를 요청했을 때 백엔드 로직이 실행되는 중심 공간이라고 볼 수 있습니다.

일반 직업도 위험하다

"내 직업은 창의적이라서 괜찮다"는 생각은 위험합니다. 단순 반복뿐 아니라 지식 기반 직업도 빠르게 변화하고 있습니다.

  • 회계 담당자의 하루치 엑셀 작업 → AI가 5분 만에 처리
  • 번역가의 하루 번역량 → AI가 1초 만에 완성
  • 디자인 초안 작업 → AI가 10초 만에 시안 10개 생성
  • 고객센터 상담원 → AI 챗봇이 24시간 무인 운영
  • 법률 문서 검토 → AI가 수백 페이지를 몇 분 만에 분석
  • 콘텐츠 작성 · 보고서 초안 → AI가 즉시 초안 생성

개발자도 예외가 아니다

"개발자는 AI로 대체 안 되겠지"라는 생각은 가장 위험한 착각입니다. 코드만 짜는 개발자는 이미 위험합니다.

  • 반복적인 CRUD 구현 → AI가 더 빠르고 정확하게 처리
  • 버그 찾기 · 수정 → AI가 코드 전체를 분석해 즉시 제안
  • 단순 API 연동 작업 → AI가 코드 한 번에 완성
  • AI 활용 개발자 1명 = 일반 개발자 3~5명의 생산성
  • 실제로 IT 기업들이 채용 공고에 "AI 활용 능력" 필수 추가 중
  • 살아남는 개발자 = AI로 더 큰 문제를 해결하는 개발자

AI를 먼저 배운 사람의 기회

두려워하기보다 먼저 배우는 것이 답입니다. AI를 이해하는 사람에게는 엄청난 레버리지가 생깁니다.

  • AI 활용자 1명이 팀 전체의 생산성을 혼자 냄
  • 코드를 전혀 몰라도 AI로 실제 서비스를 만드는 시대
  • 기획자가 직접 프로토타입을 만들어 개발팀과 대화 가능
  • 더 적은 비용으로 더 빠르게 사이드 프로젝트 완성 가능
  • AI를 모르는 경쟁자보다 항상 앞서 나갈 수 있는 구조
  • AI는 위기가 아닌, 먼저 잡으면 초강력 무기

AI(LLM)는 어떻게 작동하나요? — 멘토가 꼭 알아야 할 기초 지식

수강생이 "AI가 뭔가요?" 라고 물었을 때 자신있게 설명할 수 있어야 합니다. 기술적 깊이보다 핵심 개념을 쉽게 전달하는 게 목표입니다.

PART 01

LLM이 뭔가요? — 대형 언어 모델의 기본 개념

LLM이란?

Large Language Model, 즉 "대형 언어 모델"의 줄임말입니다. 인터넷, 책, 코드, 논문 등 수천억 개의 텍스트 데이터를 학습한 AI입니다.

  • GPT-4o (OpenAI), Claude (Anthropic), Gemini (Google)이 대표적
  • 우리가 사용하는 Claude도 LLM의 한 종류
  • 단어를 하나씩 예측하며 문장을 완성하는 방식으로 작동
어떻게 학습하나요?

LLM은 3단계 과정을 통해 만들어집니다. 이 과정을 이해하면 왜 AI가 이런 특성을 가지는지 설명할 수 있습니다.

  • 사전학습(Pre-training) — 수천억 텍스트로 언어 자체를 이해
  • 파인튜닝(Fine-tuning) — 대화·코드·지시 따르기 특화 훈련
  • RLHF — 사람이 좋은 답과 나쁜 답을 평가해 AI를 개선
토큰이란?

AI는 텍스트를 "토큰" 단위로 처리합니다. 단어 하나가 1~3개의 토큰이 됩니다. 이것이 AI 사용 비용과 직결됩니다.

  • 영어 1단어 ≈ 1.3 토큰 / 한글 1단어 ≈ 2~4 토큰
  • Claude Pro: 200K 토큰 컨텍스트 (약 15만 단어 분량)
  • 대화가 길어질수록 더 많은 토큰을 소비
컨텍스트 윈도우란?

AI가 한 번에 기억할 수 있는 텍스트의 양입니다. 이 범위를 벗어나면 AI는 앞의 내용을 잊어버립니다.

  • Claude의 강점 중 하나가 매우 긴 컨텍스트 윈도우
  • 코드 전체를 붙여넣고 수정 요청해도 정확하게 처리 가능
  • 긴 대화를 해도 초반 내용을 기억하는 이유

PART 03

왜 프롬프트를 잘 써야 하나요? — 질문의 질이 답의 질을 결정합니다

AI의 성능은 고정되어 있습니다. 같은 AI라도 프롬프트 방식에 따라 결과 품질이 완전히 달라집니다.

나쁜 프롬프트의 패턴
  • "웹사이트 만들어줘" → 어떤 웹사이트인지 AI가 추측해야 함
  • "코드 고쳐줘" → 어디가 문제인지, 어떻게 고쳐야 하는지 불명확
  • "예쁘게 만들어줘" → "예쁜"의 기준이 사람마다 다름
  • "더 좋게 해줘" → 무엇이 좋은 것인지 AI가 판단해야 함
  • 역할 부여 없음 → AI가 적절한 수준을 모름
  • 배경 설명 없음 → 지금까지의 맥락을 AI가 모름
좋은 프롬프트의 요소
  • 역할 부여 — "너는 10년 경력 시니어 프론트엔드 개발자야"
  • 배경 설명 — "지금 카페 소개 랜딩 페이지를 만들고 있어"
  • 구체적 요청 — "헤더 + 카드 3개 + CTA 버튼이 필요해"
  • 스타일 지정 — "색상은 따뜻한 브라운 계열, 폰트는 깔끔하게"
  • 출력 형식 — "HTML/CSS 파일 분리해서 작성해줘"
  • 제약 조건 — "jQuery 쓰지 말고 순수 JS로만 해줘"

초급

중급

PART 04

AI의 한계 — 맹목적으로 믿으면 안 됩니다

AI를 잘 활용하려면 AI가 잘하는 것과 못하는 것을 정확히 알아야 합니다. 수강생에게 반드시 전달해야 할 내용입니다.

주의 1
할루시네이션 (Hallucination)

AI가 틀린 정보를 자신있게 말하는 현상입니다. 존재하지 않는 라이브러리, 없는 함수, 잘못된 역사 사실 등을 마치 사실인 것처럼 얘기합니다.

예: "React에서 useAutoLayout() 훅을 쓰면 돼" → 실제로 없는 훅을 자신있게 알려줌
대처법: 중요한 정보는 반드시 공식 문서로 검증. 코드는 실행해서 확인.
주의 2
학습 데이터 기준 날짜

AI는 특정 시점까지의 데이터로만 학습했습니다. 그 이후에 나온 새로운 라이브러리, 프레임워크 업데이트, 최신 뉴스는 모릅니다.

예: 최근에 출시된 Next.js 새 기능을 물어보면 예전 방식으로 알려줄 수 있음
대처법: "이게 최신 방식이 맞아?"라고 한 번 더 확인. 공식 changelog 참조.
주의 3
수학·계산 오류

언어 모델이기 때문에 복잡한 수학 계산이나 정확한 숫자 연산에서 실수를 할 수 있습니다. 단순 사칙연산은 괜찮지만 복잡한 계산은 주의해야 합니다.

예: 긴 숫자 곱셈이나 통계 계산을 시키면 가끔 틀린 값을 내놓음
대처법: 중요한 계산 결과는 직접 검산하거나 계산기/코드로 검증.
주의 4
맥락 한계 (Context 초과)

대화가 매우 길어지면 AI가 앞의 내용을 잊어버리거나 일관성을 잃을 수 있습니다. 컨텍스트 윈도우를 초과하기 때문입니다.

예: 긴 코딩 세션에서 "아까 내가 말한 디자인 시스템 색상으로 해줘" → AI가 기억 못 할 수 있음
대처법: 중요한 정보는 새 대화 시작 시 다시 붙여넣기. 세션을 짧게 관리.
주의 5
개인정보·저작권

AI에 민감한 개인정보, 회사 기밀 데이터, 저작권 있는 코드를 그대로 붙여넣는 것은 주의해야 합니다.

예: 고객 데이터 파일을 AI에 업로드해서 분석 요청 → 개인정보 유출 위험
대처법: 민감 정보는 익명화 처리 후 사용. 회사 정책 확인 필수.
주의 6
AI 의존 과다

AI가 만들어준 코드를 이해하지 못한 채 그냥 쓰면 나중에 수정이나 디버깅이 불가능해집니다. AI는 도구이지 전부가 아닙니다.

예: AI가 만든 코드가 에러 날 때 "왜 에러가 나는지" 전혀 모르는 상황
대처법: AI 코드를 보면서 반드시 이해하려는 노력. 이해 안 되면 Claude에게 설명 요청.

PART A

좋은 프롬프트의 핵심 4원칙 — 이 4가지만 지켜도 결과가 달라집니다

역할 → 맥락 → 구체적 요청 → 반복 수정. 이 순서대로 작성하는 것이 프롬프트 기본기입니다.

01
역할 부여 (Role)

Claude에게 먼저 역할을 부여합니다. 역할에 맞는 전문 수준의 답변이 나옵니다.

예시 "너는 10년 경력 시니어 프론트엔드 개발자야"
"너는 UX 디자이너야. 사용자 친화적으로 설명해줘"
"너는 웹 개발 선생님이야. 초보자가 이해할 수 있게 설명해줘"
역할이 없으면 AI가 적절한 수준을 모릅니다. 초보자에게 고급 코드를 주거나, 전문가에게 너무 쉬운 설명을 하게 됩니다.
02
맥락 제공 (Context)

지금 무엇을 만들고 있는지, 어디까지 진행됐는지 배경을 충분히 설명합니다.

예시 "지금 카페 소개 랜딩 페이지를 만들고 있어. 헤더는 이미 만들었고 지금은 메뉴 섹션을 만들려고 해"
"이 코드는 React로 작성됐고, Tailwind CSS를 쓰고 있어. 기존 스타일을 유지해줘"
맥락이 없으면 AI가 처음부터 다시 추측해야 합니다. 많을수록 정확도가 올라갑니다.
03
구체적 요청 (Specific)

색상, 크기, 레이아웃, 포함될 요소, 기술 스택, 제약 조건을 명확하게 명시합니다.

예시 "카드 3개를 가로로 배치해줘. 각 카드에는 이미지, 제목, 설명, 버튼이 들어가야 해. 색상은 파란 계열로"
"jQuery 없이 순수 JavaScript로만 작성해줘. 외부 라이브러리 사용 금지"
막연한 요청은 막연한 결과를 낳습니다. "예쁘게 만들어줘"는 기준이 없습니다.
04
반복 수정 (Iterate)

첫 결과가 완벽하길 기대하지 않습니다. 대화하며 계속 수정해나가는 것이 올바른 방식입니다.

예시 "버튼 색을 파란색으로 바꿔줘"
"카드 간격이 너무 좁아. 더 넓게 해줘"
"모바일에서 1열로 쌓이도록 반응형 추가해줘"
대화형 AI의 진짜 강점은 반복 수정입니다. 포기하지 말고 계속 요청하세요.
PART B

나쁜 프롬프트 vs 좋은 프롬프트 — 실제 예시로 배우는 차이

같은 목적이라도 어떻게 요청하느냐에 따라 결과가 완전히 달라집니다. 수강생에게 이 예시들을 직접 보여주세요.

나쁨 "웹사이트 만들어줘" 어떤 웹사이트인지, 어떤 기술을 써야 하는지 AI가 전부 추측해야 함
좋음 "너는 시니어 프론트엔드 개발자야. 카페 소개 랜딩 페이지를 HTML/CSS로 만들어줘. 헤더에는 카페 이름과 네비, 메인에는 배경 이미지와 문구, 아래에는 메뉴 카드 3개가 있어야 해. 색상은 따뜻한 브라운 계열" 원하는 결과에 가까운 코드가 바로 나옴
나쁨 "코드 고쳐줘" 어디가 문제인지, 어떻게 고쳐야 하는지 AI가 모름
좋음 "이 코드에서 버튼 클릭 시 모달이 열려야 하는데 안 열려. 에러 메시지는 'Cannot read properties of null'. 아래 코드에서 문제를 찾아서 수정해줘: [코드 붙여넣기]" 정확한 원인 파악 후 수정 코드 제공
나쁨 "더 예쁘게 만들어줘" "예쁜"의 기준이 없어서 AI가 임의로 판단함
좋음 "카드에 hover 시 살짝 위로 올라오는 효과 추가해줘. 그림자도 더 진하게. 색상은 지금 파란색 계열 유지하면서 좀 더 밝게 해줘" 구체적인 방향이 있어서 의도에 맞는 결과 나옴
나쁨 "React로 쇼핑몰 만들어줘" 범위가 너무 넓어서 AI가 기본 구조만 만들거나 과도하게 복잡하게 만듦
좋음 "React로 상품 목록 페이지만 만들어줘. 상품 카드는 이미지, 이름, 가격, '장바구니 담기' 버튼으로 구성. 카드 클릭 시 상세 페이지로 이동. useState로 장바구니 개수 표시. Tailwind 사용" 명확한 범위와 요구사항으로 바로 사용 가능한 코드 제공

Claude

STEP 2 — 피드백 & 수정
나 (수강생)
"좋아. 근데 카드 간격이 너무 좁고, 버튼 색이 너무 어두워. 카드 간격 더 넓게 하고 버튼은 밝은 카멜 색으로 바꿔줘."
↓
Claude
카드 gap 수정 + 버튼 색상만 변경한 업데이트 코드 제공. 변경된 부분 주석으로 표시.
STEP 3 — 기능 추가
나 (수강생)
"모바일에서는 카드가 1열로 쌓이게 반응형 추가해줘. 그리고 카드에 hover 시 살짝 올라오는 효과도 넣어줘."
↓
Claude
미디어 쿼리 + transition + transform 추가. 기존 코드에서 추가된 부분만 명확히 표시.
STEP 4 — 완성
총 3번의 대화로 완성도 높은 카페 랜딩 페이지 완성. 첫 번째 결과가 완벽할 필요가 없습니다. 대화할수록 완성도가 올라갑니다.

AI 활용 프론트엔드 개발자

AI 없이 개발하는 사람과 AI로 개발하는 사람의 속도 차이는 이미 3~5배입니다. 같은 시간에 더 많이 만들고, 혼자서도 완성도 높은 서비스를 만드는 개발자 — 채용 시장에서 가장 주목받는 포지션입니다.

채용 공고에 "AI 활용 능력 우대" 명시가 빠르게 늘고 있습니다.
→ 같은 실력이라면 AI 잘 쓰는 개발자가 무조건 뽑힙니다

실제로 하는 일

  • Figma 디자인 → React 컴포넌트로 구현 (Claude로 속도 3배)
  • 반응형 웹 페이지 제작 — 모바일·태블릿·데스크톱 모두 대응
  • REST API 연동으로 실시간 데이터를 화면에 표시
  • Claude로 코드 초안 생성 → 직접 리뷰·수정·최적화
  • 에러 디버깅 — Claude에게 에러 붙여넣고 원인 파악
  • Vercel로 배포 → 실제 URL로 접속 가능한 서비스 완성

취업에 성공하는 사람의 3가지 공통점

01

포트폴리오 퀄리티로 승부한다

출신 학교·자격증·수료 기관보다 "실제로 뭘 만들었는지"가 결정적입니다. 완성도 높은 결과물 1개가 엉성한 결과물 10개보다 낫습니다. 수업에서 만드는 모든 것을 포트폴리오 퀄리티로 끌어올리세요.

수업에서 만든 것을 그냥 두지 말고, 계속 개선해서 "내 작품"으로 만드세요.
02

수료 후에도 혼자 계속 만든다

4개월 수업은 시작점입니다. 수업이 끝난 후에도 매주 무언가를 만드는 습관이 있는 사람이 취업에 성공합니다. GitHub 커밋 내역이 꾸준히 쌓이는 것 자체가 실력의 증거입니다.

수료 후 2개월 안에 취업에 성공한 사람들의 공통점 — 수업 끝나도 매일 코딩했습니다.
03

AI 활용 방식을 설명할 수 있다

면접에서 "이 프로젝트를 Claude와 어떻게 만들었는지" 구체적으로 설명할 수 있다면 강력한 차별점이 됩니다. AI를 쓴 것이 부끄러운 게 아닙니다. AI를 잘 쓰는 것이 이제는 핵심 역량입니다.

"Claude에게 이렇게 요청해서 이 기능을 구현했습니다" — 이 한 마디가 면접관을 설득합니다.
