상태/피드백/흐름 검증

문제 정의를 “해결 가능한 구조”로 번역하고, 흐름을 설계 언어로 고정

도입 본질 IA 시나리오 UX 흐름 화면 구조 상태·피드백 과제

화면이 아니라, 설계 언어가 프로젝트의 뼈대가 된다

프로젝트가 설계 단계에 들어가는 순간은 화면을 그릴 때가 아니라, 정보 구조와 UX 흐름이 언어로 고정될 때다.

문제 정의가 프로젝트의 출발점이라면, 정보 구조(IA)와 UX 흐름은 그 문제를 실제로 해결 가능한 형태로 바꾸는 첫 번째 구조적 해석이다.

이 단계가 흐릿하면 이후의 화면은 모두 임시방편이 된다. 반대로 이 단계가 고정되면 이후의 화면/구현은 방향 탐색이 아니라 구체화 작업으로 전환된다.

이 주차가 강제로 묻는 질문

01 이 프로젝트는 어떤 정보 단위를 통해 문제를 풀어야 하는가
02 사용자는 어떤 상태 변화/판단 흐름으로 문제를 통과하는가
03 구조와 흐름에서 “왜 필요한지 설명 불가한 단계”는 어디인가

21주차는 덧붙이는 주차가 아니라, 덜어내고 고정하는 주차다.

정보 구조는 정리가 아니라 ‘문제의 분해’다

핵심 통찰

정보 구조(IA)는 “이 서비스에 무엇이 있는가”를 설명하는 문서가 아니다.

IA는 “이 문제를 해결하기 위해 어떤 정보가 필요하며, 그것들이 어떤 관계를 맺는가”를 고정하는 설계 문서다.

오해

“페이지/메뉴를 정리하면 IA가 된다”

구현 단위를 구조 단위로 착각
사용자 판단 단위가 사라짐
흐름은 화면 전환으로 축소됨

구조가 흐려진다

정의

“의미 단위와 관계를 고정한다”

사용자가 던지는 질문/필요한 답으로 정보 단위를 정의
관계(포함/병렬/선행/상하)를 언어로 명시
흐름은 상태 변화/판단의 논리로 서술

구조가 선명해진다

구조 단위 전환

페이지는 구현 단위일 뿐, 구조 단위가 아니다. 구조는 의미 단위의 묶음이며 사용자 판단을 기준으로 정의된다.

같은 화면 안에 여러 구조가 공존할 수도 있고, 하나의 구조가 여러 화면에 걸쳐 펼쳐질 수도 있다.

“있으면 좋아 보인다”, “보통 다 있으니까”로 포함된 정보 묶음은 이후 흐름을 복잡하게 만들고 판단 기준을 흐린다.

IA 작성 규칙: 의미 단위 + 관계 + 문제 정의 연결

IA는 “메뉴 트리”가 아니라 “정보 체계”다

IA 문서는 정보 단위(의미 단위)를 정의하고, 그 단위들 사이의 관계를 명시해야 한다.

각 단위는 문제 정의의 어떤 부분을 설명하거나 해결하는지 연결되어야 하며, 연결되지 않는 단위는 의심 대상이다.

IA에서 반드시 언어로 고정할 것

정보 단위는 어떤 사용자 질문/판단을 위해 존재하는가?
이 단위는 다른 단위와 어떤 관계인가(포함/병렬/선행/상하)?
문제 정의의 어떤 막힘을 이 단위가 직접 다루는가?

구조 단계의 절제는 이후 설계 전체의 밀도를 결정한다.

“좋은 정보”가 아니라 “필요한 정보”만 남긴다.

대표 시나리오 1개: 구조를 시간 축으로 검증

시나리오는 기능 순서가 아니라 상태 변화다

대표 시나리오는 “사용자 상태 → 불확실성 → 판단 → 상태 해소”의 흐름으로 서술되어야 한다.

모든 가능성을 다루지 않는다. 이 프로젝트가 반드시 해결해야 하는 핵심 문제 상황 하나를 선택하고 끝까지 따라간다.

“회원가입 → 검색 → 상세 → 결제” 같은 기능 나열

진입 불확실성 → 판단 단계 → 해소되는 정보 → 목표 상태 도달

예외 시나리오/경계 케이스는 이후 단계다. 이 주차는 중심축을 고정하는 단계다.

UX 플로우: 화면 전환이 아니라 판단 연쇄의 고정

단계가 아니라 “왜 존재하는지”가 핵심이다

UX 플로우는 사용자의 상태 변화와 판단 지점을 연결한 논리적 흐름이다.

단계가 많다는 것이 문제는 아니다. 문제는 “왜 필요한지 설명할 수 없는 단계”가 존재할 때다.

제거 대상의 신호

단계 존재 이유가 말로 안 됨
관행/구현 편의로 삽입됨
문제 정의와 직접 연결되지 않음

21주차의 UX 플로우는 “더하기”가 아니라 “삭제와 정합”을 위한 검증 도구다.

화면 구조 설계: 설계 언어를 물리적 구조로 정확히 변환

화면은 결과물이 아니라 검증 장치다. 이미 내려진 판단을 공간과 관계로 정확히 번역한다.

이 섹션은 이전 단계에서 고정한 문제/구조/흐름/우선순위를 화면 구조로 변환해, 설계 사고가 흔들리지 않았는지를 검증한다.

상태·피드백·흐름 검증: 사용자는 멈추지 않는다

정적인 화면이 아무리 완성도가 높아 보여도, 사용 흐름이 깨지는 순간 설계는 실패한다.

이 주차의 목표는 화면 단위의 완성도를 평가하는 것이 아니라, 사용자가 실제로 움직이며 겪게 되는 전체 흐름이 논리적으로 작동하는지를 검증하는 데 있다. 여기서 검증 대상은 디자인의 겉모습이 아니라, 상태 변화와 그에 따른 사용자 인식의 연속성이다. 화면은 멈춰 있지만, 사용자는 멈추지 않는다. 이 주차는 그 사실을 설계 차원에서 끝까지 따라가는 단계다.

많은 설계가 화면 단위에서는 그럴듯해 보이지만, 실제 사용 과정에서는 미묘한 혼란을 만들어낸다. 버튼은 명확한데 다음 상태가 불분명하거나, 로딩이 끝났는지 알 수 없거나, 사용자가 이미 한 행동을 또 하게 만드는 구조가 흔하게 등장한다. 이러한 문제는 시각적 완성도의 문제가 아니라 상태와 흐름이 제대로 설계되지 않았다는 신호다. 이 주차는 바로 그 숨겨진 오류를 드러내는 역할을 한다.

핵심 통찰

상태 정의는 UI 상태 나열이 아니라, 사용자가 “지금 어떤 상황인가”를 언어로 고정하는 작업이다.

상태가 정의되지 않으면 사용자는 추측으로 행동하고, 그 추측은 설계 의도와 어긋난다.

상태를 정의한다는 것은 단순히 기본·호버·활성 같은 UI 상태를 나열하는 작업이 아니다. 상태 정의는 사용자가 지금 어떤 상황에 놓여 있는지를 설계 언어로 고정하는 일이다. 사용자는 항상 질문을 품고 있다. 지금 이 행동이 가능한가, 이미 처리된 것인가, 기다려야 하는가, 다시 시도해야 하는가. 상태 설계는 이 질문에 즉각적으로 답을 주는 구조다.

주요 상태의 범위

이 주차에서 다루는 주요 상태는 화면 내부의 미세한 변화보다 사용자 판단에 영향을 주는 전환 지점에 집중해야 한다. 시작 상태, 진행 중 상태, 완료 상태, 오류 상태, 차단 상태처럼 사용자의 행동 가능성을 바꾸는 상태들이 핵심이다. 이 상태들이 명확히 구분되지 않으면, 사용자는 시스템이 자신을 인식하고 있는지조차 확신하지 못한다. 상태 정의는 시스템의 내부 상태를 사용자 인식 상태로 번역하는 작업이다.

피드백 구조 설계

피드백 구조 설계는 이 번역을 실제로 작동하게 만드는 장치다. 피드백은 장식이 아니라 확인이다. 사용자의 행동이 받아들여졌다는 확인, 처리가 진행 중이라는 확인, 처리가 끝났다는 확인, 문제가 발생했다는 확인이 적절한 시점에 제공되지 않으면 사용자는 다음 행동을 결정할 수 없다. 이때 발생하는 반복 클릭, 이탈, 불신은 모두 피드백 부재에서 비롯된다.

중요한 점은 피드백이 항상 눈에 띄어야 하는 것은 아니라는 사실이다. 필요한 만큼만 명확해야 한다. 과도한 피드백은 오히려 흐름을 방해하고, 부족한 피드백은 불안을 만든다. 피드백 설계의 핵심은 사용자가 다음 행동을 결정하는 데 필요한 정보만 정확히 전달하는 것이다. 형식보다 목적이 우선이다.

흐름 오류는 기능 실패가 아니라 “사용자 인식이 끊기는 지점”이다.

흐름 상 오류 점검은 이 주차의 가장 중요한 검증 단계다. 왜 이 단계가 필요한지 이해하지 못하는 순간, 지금 무엇을 해야 하는지 모르는 순간, 이미 한 행동을 다시 요구받는 순간이 바로 오류다. 이 오류들은 대부분 화면 설계 단계에서는 드러나지 않고, 흐름을 따라가 볼 때 비로소 드러난다.

이 단계에서 설계자는 화면을 보는 사람이 아니라 사용자가 되어야 한다. 처음 진입했을 때 무엇을 이해하는지, 행동 후에 어떤 확신을 가지는지, 예상과 다른 반응이 나왔을 때 어떻게 해석하는지를 하나씩 점검해야 한다. 이 점검은 공감의 문제가 아니라 논리의 문제다. 사용자가 오해할 수 있는 구조라면, 그것은 설계 오류다. 사용자가 잘못 이해했다는 설명은 통과되지 않는다.

제출물: 상태 정의 문서

제출물로 요구되는 상태 정의 문서는 상태 목록이 아니라 상태 간 관계를 설명하는 문서여야 한다. 어떤 상태에서 어떤 상태로 이동하는지, 그 이동은 어떤 사용자 행동이나 시스템 이벤트에 의해 발생하는지를 명확히 해야 한다. 또한 각 상태가 사용자에게 어떤 메시지를 전달하는지도 함께 설명되어야 한다. 상태는 내부 로직이 아니라 사용자 경험의 단위다.

제출물: 인터랙션 설명 자료

인터랙션 설명 자료 역시 동작 설명이 아니라 의미 설명이어야 한다. 이 인터랙션이 왜 필요한지, 이 변화가 사용자에게 무엇을 알려주는지, 이 반응이 없을 경우 어떤 문제가 발생하는지를 언어로 정리해야 한다. 클릭했다는 사실보다 중요한 것은, 클릭 이후 사용자가 무엇을 확신하게 되는가다. 인터랙션은 행동에 대한 보상이 아니라, 인식에 대한 안내다.

통과 기준이 사용자 관점 설명 가능으로 설정된 이유도 여기에 있다. 설계자의 의도나 내부 로직 설명이 아니라, 사용자가 어떤 상태로 이동하며 무엇을 이해하게 되는지를 중심으로 설명할 수 있어야 한다. “이렇게 설계했다”가 아니라 “사용자는 이 시점에서 이것을 알게 된다”라는 언어가 나와야 한다.

장식적 인터랙션 배제 역시 중요한 기준이다. 의미 없는 애니메이션, 상태 변화와 무관한 효과, 사용자 판단과 연결되지 않는 반응은 이 단계에서 모두 제거 대상이다. 인터랙션은 많을수록 좋은 것이 아니라, 정확할수록 좋은 것이다. 불필요한 인터랙션은 흐름을 부드럽게 만드는 것이 아니라, 흐름을 흐리게 만든다.

이 주차는 설계를 ‘움직여보는’ 단계다. 이 단계에서 설계자는 자신의 설계를 방어하지 않는다. 대신 설계가 실제 사용 과정에서 어떤 인식의 연쇄를 만들어내는지를 관찰한다. 이 검증을 통과하면, 화면은 더 이상 정적인 결과물이 아니다. 화면은 상태와 흐름, 인터랙션이 맞물려 작동하는 하나의 경험 구조가 된다. 이 지점에서 설계는 비로소 사용자 시간 속으로 들어간다.

과제 안내

과제의 목적은 IA/시나리오/UX 플로우를 “설명 자료”가 아니라 “검증 문서”로 제출하는 것이다.

화면을 상상하지 않고도, 구조와 흐름이 문제 정의를 직접 해결하고 있음을 언어로 증명해야 한다.

후보였으나 배제한 기능/정보 묶음 3~5개와 배제 이유를 작성한다.

“좋아 보이지만 현재 문제 정의/시나리오/흐름에서 역할이 없다”라는 근거를 명확히

통과 기준

구조(IA)와 흐름(UX)이 문제 정의와 1:1로 정합한다

= 연결되지 않는 정보 묶음/단계가 없다

“왜 필요한지 설명 불가한 단계”가 제거되어 있다

= 덜어내는 설계가 작동한다

대표 시나리오가 기능 나열이 아니라 상태 변화로 서술된다

= 흐름이 화면이 아니라 판단으로 구성된다

이 주차의 의미

21주차는 화면을 만드는 주차가 아니라, 화면이 따라올 수밖에 없는 구조와 흐름을 고정하는 주차다.

구조가 문제를 설명하고, 흐름이 문제를 해결할 때, 프로젝트는 아이디어가 아니라 설계 대상으로 전환된다.

구조가 흐릿하면

이후 화면은 모두 임시방편이 된다

구조와 흐름이 고정되면

이후 설계/구현은 방향 탐색이 아니라 구체화가 된다

이 정보 단위는 어떤 사용자 판단을 위해 존재하며, 이 단계는 왜 필요한가?

설명 불가한 묶음과 단계가 사라질 때, 설계는 비로소 고정된다.
