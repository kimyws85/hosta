WEEK 6

UX는 화면이 아니라 행동의 흐름이다

페이지가 아니라 목표·판단·행동의 연쇄를 설계 대상으로 고정

도입 재정의 플로우 실패 기준 과제

UX를 화면의 연속으로 인식하는 순간, 설계는 결과물로 후퇴한다

페이지 수·구성·전환이 중심이 되는 순간, UX는 UI의 확장처럼 다뤄진다.

UX를 화면의 흐름으로 보면 설계의 단위가 페이지가 되고, 사용자는 화면 사이를 이동하는 존재로 그려진다. 이 관점에서는 “페이지가 몇 개 있는지”, “각 페이지에 무엇이 들어가는지”, “전환이 어떻게 이루어지는지”가 설계의 핵심처럼 보이기 쉽다.

그러나 이 주차에서 다루는 핵심은 UX를 화면의 흐름이 아니라 행동의 흐름으로 재정의하는 데 있다. 사용자는 페이지를 이동하는 것이 아니라, 목표를 달성하기 위해 행동을 이어간다. 화면은 그 행동이 가능하도록 잠시 머무는 지점일 뿐이다.

이 주차가 강제로 묻는 질문

01 사용자는 어떤 목표로 진입하는가
02 각 단계에서 무엇을 판단하고 왜 행동하는가
03 화면은 행동을 가능하게 하는 최소 단위로 수렴하는가

UX 설계의 핵심은 “어떤 화면을 만들었는가”가 아니라 “행동의 연쇄가 끊기지 않는가”다.

기본 단위 전환: 페이지 단위 사고 → 행동 단위 사고

핵심 통찰

페이지 단위 사고는 정적이다. 여기서는 화면이 기본 단위가 되고, 사용자는 화면 사이를 이동하는 존재로 다뤄진다.

행동 단위 사고는 동적이다. 기본 단위는 화면이 아니라 의도·판단·행동이며, UX는 그 연쇄를 끊기지 않게 설계하는 일이다.

페이지 단위 사고

“화면이 몇 개고 무엇이 들어가는가”

설계의 단위가 화면으로 고정됨
사용자는 화면 사이를 이동하는 존재로 환원됨
문제 해결이 UI 구성/전환으로 수렴됨

흐름이 숨는다

행동 단위 사고

“목표를 달성하기 위해 무엇을 판단하고 행동하는가”

설계의 단위가 의도·판단·행동으로 이동
각 단계의 필요성이 논리로 검증됨
화면은 행동을 가능하게 하는 구현으로 수렴

원인이 열린다

UX 사고의 기본 순서

이 주차의 핵심 개념은 사용자 목표 → 행동 → 화면이다.

많은 경우 실제 작업은 반대로 진행된다. 먼저 화면을 만들고, 그 안에 행동을 배치하고, 나중에 목표를 끼워 맞춘다. 이 방식에서는 화면은 풍부해지지만 행동은 억지로 삽입된다.

반대로 목표에서 출발하면 행동은 필연이 되고, 화면은 최소 단위로 수렴한다. UX 설계는 항상 목표에서 시작해 행동으로 내려오고, 마지막에 화면으로 구현되어야 한다.

UI와 UX 플로우를 분리해야 한다

UX 플로우와 UI의 차이를 명확히 구분하는 것이 이 주차의 중요한 전제다. UI는 사용자가 마주하는 인터페이스의 형태이며, UX 플로우는 사용자가 목표를 달성하기까지 거치는 판단과 행동의 구조다.

UI는 보이는 것이고 UX 플로우는 보이지 않는 것이다. 그러나 보이지 않는 구조가 보이는 형태를 지배한다. 플로우가 설계되지 않은 UI는 아무리 정교해도 사용자에게 다음 행동을 설득하지 못한다.

플로우가 다루는 대상

상태 변화 “어디로 이동”이 아니라 “어떤 상태에서 어떤 상태로 변화”
판단 구조 정보 획득·비교·결정·실행·확인으로 이어지는 판단의 연쇄
연결 강도 각 단계에서 “왜 이 행동이 필요한가”가 유지되는지

UX 플로우는 “어디에서 어디로 이동하는가”가 아니라 “불확실 → 판단 → 실행 → 확인”으로 이어지는 상태 변화의 연쇄다.

사용자는 처음에 불확실한 상태로 진입하고, 정보를 얻고, 비교하고, 결정을 내리고, 실행하고, 확인하는 상태로 이동한다. 이 상태 변화가 부드럽게 이어질 때 UX는 자연스럽게 느껴진다.

반대로 특정 단계에서 필요한 정보가 부족하거나, 선택의 이유가 약하거나, 다음 행동의 결과가 불명확하면 흐름은 끊긴다. 이 지점이 이탈이 발생하는 구조적 원인이다.

UI는 결과이고,

UX 플로우는 원인이다.

이탈은 화면 품질이 아니라 행동 동기의 붕괴에서 발생한다

UX 실패의 흔한 형태

UX에서 발생하는 대부분의 실패는 특정 화면의 완성도 때문이 아니라 흐름의 단절 때문이다.

각 화면은 나쁘지 않다. 정보도 있고, 버튼도 있고, 시각적으로도 정돈되어 있다. 그럼에도 사용자는 중간에 이탈한다.

이탈의 실제 원인

“이 화면이 별로다”
“이 시점에서 이 행동을 할 이유가 사라졌다”

이탈은 우연이 아니다

집중력이 부족해서 / 성격이 급해서
특정 단계에서 “굳이 여기까지 할 이유가 없다”는 판단이 발생

해결의 방향

버튼을 키우고 애니메이션을 추가
각 단계에서 “왜 이 행동이 필요한가”를 구조적으로 설득

화면 완성도로 문제를 진단한다

행동 동기와 단계 필요성으로 문제를 진단한다

통과 기준은 화면 설명이 아니라 행동 설명이다

버튼 위치·페이지 수를 말하는 순간 사고는 UI로 후퇴한다. 중심은 “사용자가 이 시점에 무엇을 고민하고 왜 다음 행동을 선택하는가”여야 한다.

통과 기준은 설명의 대상이 화면인지 행동인지로 명확히 갈린다. 화면 설명이 아니라 행동 설명이 중심이 되어야 한다는 조건은 사고의 단위가 전환되었는지를 검증한다.

또 하나의 기준은 각 단계의 필요성을 설명할 수 있는지 여부다. 이 단계가 없다면 어떤 문제가 발생하는지, 혹은 이 단계를 건너뛰면 어떤 판단이 불가능해지는지를 설명할 수 있어야 한다. UX 플로우가 임의의 나열이 아니라 논리적으로 연결된 행동 구조임을 증명하는 과정이다.

판단 기준(질문)

각 단계에서 사용자는 무엇을 판단해야 하며, 그 판단을 가능하게 하는 정보는 충분한가?
다음 행동의 이유가 유지되는가, 아니면 “굳이”가 사라지는 지점이 있는가?
단계 하나를 제거하거나 통합했을 때, 어떤 판단이 불가능해지는지 설명할 수 있는가?

과제 안내

과제는 행동 중심 사고를 실제로 적용하기 위한 설계 훈련이다. 특정 사용자 시나리오 하나를 정의하는 이유는 UX를 추상적인 흐름이 아니라 구체적인 목적 달성 과정으로 다루기 위함이다.

이 시나리오는 “회원 가입하기” 같은 기능 설명이 아니라, 사용자가 어떤 상황에서 어떤 문제를 해결하려고 이 서비스에 들어왔는지를 포함해야 한다. 행동의 출발점은 기능이 아니라 상황이다.

작성 순서

1
사용자 시나리오 정의

사용자가 어떤 상황에서 어떤 문제를 해결하려고 진입하는지 서술한다. 목표를 “기능”이 아니라 “상황 기반 목적”으로 고정한다.

“무엇을 누른다”가 아니라 “왜 들어왔고 무엇을 해결하려는가”를 먼저 명시

행동의 출발점이 기능이 되면 플로우는 곧 UI 나열로 붕괴한다
2
행동 흐름 단계화

화면을 나열하지 말고, 사용자가 무엇을 판단하고 어떤 결정을 내리며 그 결과로 어떤 행동을 하는지 순서대로 서술한다.

“상품 상세 페이지로 이동”이 아니라 “이 상품이 목적에 부합하는지 판단”처럼 행동 단위로 작성

판단
결정
행동
3
단계별 필요 정보·선택 명시

각 단계에서 필요한 정보가 충분한지, 무관한 정보가 과도하게 개입되는지, 선택의 의미와 결과가 예측 가능한지 점검한다.

선택지가 존재해도 “선택의 의미/결과”가 불명확하면 행동은 발생하지 않는다

필요 정보
선택 의미
결과 예측

통과 기준

화면 설명이 아니라 행동 설명이 중심이어야 한다

= 사용자가 무엇을 고민하고 왜 다음 행동을 선택하는지 서술 가능

각 단계의 필요성을 논리로 설명할 수 있어야 한다

= 단계 삭제/통합 시 어떤 판단이 불가능해지는지 제시

이 주차의 의미

이 주차는 흐름도를 그리는 법을 배우는 시간이 아니다. 사용자의 행동을 상상하는 법을 훈련하는 시간이다.

화면을 먼저 떠올리는 습관을 멈추고, 목표와 판단을 먼저 따라갈 수 있을 때 UX는 설계 대상이 된다.

화면부터 만들면

UX는 UI 구성의 확장으로 축소된다

목표에서 출발하면

행동은 필연이 되고 화면은 결과로 따라온다

이 단계는 왜 필요한가, 그리고 사용자가 다음 행동을 선택할 이유는 무엇인가?

UX를 행동으로 설계할 수 있을 때, 화면은 목적이 아니라 결과로 자연스럽게 따라온다.
